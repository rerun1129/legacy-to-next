# 프로젝트 개요

## 1. 목표

### 1.1 프로젝트 성격

본 프로젝트는 **시범적(Exploratory) 성격 + 테스트 베드** 포지셔닝을 가진다.

- 최신 기술 스택 도입, AI 개발 방법론 검증, 언어/프레임워크 비교 평가가 명시적 목적에 포함된다.
- 따라서 "검증되지 않은 최신 기술 채택", "팀의 학습 비용", "AI 도구 의존" 등은 리스크가 아니라 프로젝트가 의도적으로 감수하는 조건이다.
- 현 단계는 **레거시 고객사 서비스와 직접 연결되지 않는 독립 테스트 베드**다. 레거시 시스템은 **별도 프로젝트로 계속 운영 중**이며, 본 프로젝트는 레거시와 병행 운영하거나 레거시 데이터를 직접 전환하는 대상이 아니다.
- 단, 본 프로젝트의 결과물은 **향후 신규 고객사 프로덕션에 투입될 것을 전제**로 품질을 설계한다. 따라서 데이터 정합성·가용성·보안 수준은 프로덕션 등급을 목표로 한다.

### 1.2 목표

- 레거시 물류 프로젝트에 AI 개발 방법론 적용
- 기존 프레임워크를 여러 언어·프레임워크로 교체해보며 가장 좋은 조합 탐색
- TDD와 DDD가 적용되지 않은 자바-스프링 코드 베이스를 현대 개발 방법론에 맞춰 재구축
- Vue2 개발 환경을 React로 전환 (프론트엔드는 단계 구분 없이 전환)
- **자바-스프링으로 우선 구현을 완료하여 전체 기능을 검증한 뒤, 동일 기획을 파이썬-FastAPI로 재구현**하여 성능·생산성·운영 비용 비교 데이터 확보 (정식 채택 여부는 평가 후 결정). 병렬 동시 개발이 아니라 **Java → Python 순차 구현**이다.

## 2. 배경

### 2.1 레거시 시스템 현황

레거시 시스템은 **별도 프로젝트로 분리되어 계속 운영 중**이며, 본 프로젝트의 운영 범위에 포함되지 않는다. 아래는 레거시의 기술 구성을 참고 정보로 요약한 것이다.

- (백엔드) 구 버전 자바-스프링 환경 사용 중
- (프론트엔드) Vue2-Nuxt 환경 사용 중
- DBMS 단독 사용, NoSQL 미사용

### 2.2 레거시의 핵심 문제

본 프로젝트가 해결하려는 대상 문제들. 현 레거시에서 관찰된 구조적 한계를 정리한 것이며, 본 프로젝트는 이를 동일 도메인의 신규 구현을 통해 해소한다.

- DBMS 중심 개발론에 갇혀 역정규화 등의 설계 고려가 어려운 Transaction Script 중심 코드 베이스
- 노후화된 프론트엔드 프레임워크의 고착화로 안정적인 LTS 버전으로의 이전이 어려움
- 효율적인 NoSQL을 쉽게 도입할 수 없는 쿼리 베이스 구조
- 특정 외부 프레임워크(DevExtreme, Ag-Grid)에 심각하게 종속된 프론트엔드 환경

### 2.3 왜 지금인가

- AI 개발 방법론 도입으로 비교적 짧은 기간 안에 대규모 코드 베이스 수정이 가능
- 유지보수 비용의 기하급수적 증가로 인한 개발 속도 저하를 더 늦기 전에 해결해야 함 (기술 부채 해소)
- 급변하는 개발 및 외부 환경에 대응하기 위한 빠른 트렌드 트래킹 환경 구비가 필요한 시점

## 3. 성공 기준

### 3.1 정량 지표 (KPI)

- 조회 중심 화면의 조회 성능 대폭 개선
  - 기존 30~120초 소요되는 상위 10개 대량 집계 쿼리: 속도 90% 향상 목표
  - 그 외 일반 쿼리: 속도 50% 향상 목표
  - **측정 기준선 출처**: 레거시 쿼리의 p50/p95 기준선은 **별도 레거시 운영 프로젝트에서 제공받는 값**을 사용한다. 본 프로젝트에서는 신규 시스템의 측정치만 수집하여 제공받은 기준선과 비교한다 (`05_NON_FUNCTIONAL.md` §1.1 참조).
- 회계 업무 코드와 결합된 기존 포워딩 코드를 분리하여 가볍고 안정적인 빌드 환경 구축
  - 빌드·배포 시간: 기존 18분 → 5분 (75% 절감)
- AI 개발 방법론 도입으로 설계-구현-배포-피드백 사이클의 리드타임 단축 (구체 수치는 3개월 운영 후 베이스라인 확정)

### 3.2 정성 지표

- 구조화된 개발 방법 도입으로 의사소통 원활화
- 빠르고 안정적인 서비스를 통해 향후 고객사의 일시적 업무 단절 불편함 최소화
- 필요 기술 스택의 빠른 도입으로 팀원 효능감 제고

### 3.3 비-목표 (Non-goals)

- 회계 모듈과의 깊은 결합
- 전면적 모바일 네이티브 앱 개발
- **레거시 DB 스키마의 직접 마이그레이션**. 본 프로젝트는 신규 스키마로 시작하며, 향후 타겟은 **신규 고객사**다. 따라서 레거시 데이터를 옮겨오거나 레거시와 실시간 동기화하는 요구사항은 없다.

## 4. 범위

### 4.1 In scope (포함 기능)

- 대시보드
  - 정적 영역: 인사 메시지·빠른 액션, 사이드바
  - KPI 스트립
  - Shipment Pipeline / Weekly Volume / Cargo Trace
  - Upcoming ETD/ETA Timeline / My Tasks
  - MJ Notice (공지) / Public Holidays / Exchange Rate
  - Recent Activity (팀 활동 피드)
  - **현 단계는 정적 셸만 구성**하고, 위젯의 실제 동적 동작은 BMS / PMS 등 후속 모듈 개발과 함께 점진 추가한다 (02_SCREENS §S-01 참조).
- FMS 모듈
  - House B/L Entry [해상/항공] [수출/수입]
  - House B/L List [해상/항공] [수출/수입]
  - Master B/L Entry [해상/항공] [수출/수입]
  - Master B/L List [해상/항공] [수출/수입]
  - Truck B/L Entry
  - Truck B/L List
  - Non B/L Entry
  - Non B/L List
  - 문서 출력 팝업 (B/L·HAWB·MAWB·A/N·D/O·S/A·S/R·Non B/L 원본 B/L — S-02/03/04/05/08/09 내 공용 모달, 02_SCREENS 부록 §C)
- BMS 모듈 — ✅ **구현 완료** (운임 정산·금융 서류·세금계산서/전표 발행. PRD: `docs/bms/01_BMS_DOMAIN`·`docs/bms/02_BMS_SCREENS`)
- MDM(마스터데이터) — **ADMIN 모듈로 통합**. 별도 모듈을 두지 않고 거래처 마스터(DOMAIN §E-23) 등 마스터데이터를 ADMIN이 관리한다(아래 ADMIN 모듈 항목 참조).
- PMS 모듈(실적) — 📝 **기획 완료 · 구현 예정** (단일 화면 실적 조회. PRD: `docs/pms/01_PMS_DOMAIN`·`docs/pms/02_PMS_SCREENS`)
- ADMIN 모듈 (구 CMS·MDM 통합) — **구현 진행 중** (기존 PRD가 "CMS 모듈"·"MDM 모듈"로 나눠 지칭하던 영역의 통합 실체)
  - 유저 관리·인증·권한 모델(ABAC)·공통 코드 마스터·메뉴/버튼·권한 프리셋·**거래처(구 MDM 마스터데이터)**·공지를 담당한다. `back-end/java-spring-admin` 멀티모듈로 구현됐다.
  - 상세 PRD는 **`docs/admin/`** 참조: `01_DOMAIN`(엔티티)·`02_SCREENS`(화면)·`03_ACCESS_CONTROL`(권한·인증 SSOT).

## 6. 리스크 및 가정

### 6.1 프로덕션 품질 수준의 설계

- 현 단계는 테스트 베드이지만, 결과물이 향후 신규 고객사 프로덕션에 투입될 것을 전제로 데이터 정합성·가용성·보안은 타협 대상이 아니다.
- 대응: 프로덕션 등급 관측성(메트릭·트레이싱·감사 로그) 선제 도입, 배포 파이프라인의 롤백 경로 상시 확보, 테스트 커버리지 CI 강제 등은 테스트 베드 단계부터 적용한다.
- 레거시 시스템의 운영 영향은 본 프로젝트 범위 밖이며, 레거시 측 담당은 별도 프로젝트에서 수행한다.

### 6.2 듀얼 백엔드 평가 기준 부재

- Java 스택으로 우선 구현을 완료한 뒤 동일 기획을 Python 스택으로 재구현한다 (§1.2). 정식 채택 여부를 결정할 객관적 기준이 없으면 의사결정이 지연된다.
- 대응: 사전에 비교 평가 기준(처리 성능, 개발 생산성, 운영 비용, 장애 대응 용이성 등)과 측정 지표를 정의하고, **Python 스택 재구현 완료 시점**에 Go/No-Go를 판단한다. 평가 지표는 `05_NON_FUNCTIONAL.md` §8.1의 메트릭과 연결하여 측정 가능하도록 구체화한다.

### 6.3 Phase 0 현황 (현 시점 구현 상태)

- **백엔드**: **멀티모듈** — FMS(`java-spring`, Spring Boot 3.4.4 / Java 21) + ADMIN(`java-spring-admin`, 구 CMS 영역). Python 스택 미착수 (§1.2 Java → Python 순차 원칙 적용 중). Spring Boot 4.0 GA 대기.
- **프론트엔드**: Next.js 16.x (16.2.4) + React 19.x. 헥사고널 4계층 구조 — FMS는 모드/방향별 **9개 도메인**(house-bl·master-bl·switch-bl + sea-house·air-house·sea-master·air-master + truck-bl·non-bl)으로 확장 적용. ADMIN 화면군(접근제어·코드·유저·거래처·공지)도 동일 구조 적용. (`03_TECH_STACK §10` 참조)
- **데이터**: ADMIN은 Flyway **활성** 운용(`db/migration` V1~V54). FMS는 현재 `spring.flyway.enabled=false`로 프로젝트 루트 `schema/` 폴더 수동 관리(스키마 설계 완료 후 활성화 예정).
