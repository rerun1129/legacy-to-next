# 포워딩 도메인

> 본 문서는 **도메인 언어**를 정의한다. FMS(포워딩 업무) 범위에 한정한다. 운임·정산 관련 내용은 별도 BMS 모듈에서 다룬다.

## 1. 용어집

### 1.1 B/L 관련
| 한글 | 영문/약어 | 정의 |
| --- | --- | --- |
| 선하증권 | B/L (Bill of Lading) | 운송계약과 화물 수취를 증명하는 유가증권 |
| 마스터 비엘 | MBL (Master B/L) | 선사 또는 항공사가 포워더 앞으로 발행하는 B/L |
| 하우스 비엘 | HBL (House B/L) | 포워더가 실화주 앞으로 발행하는 B/L |
| 오리지널 B/L | O B/L | 원본 선하증권 (유가증권 형태의 종이 원본) |
| Surrender B/L | - | 원본 회수 후 전자 인도로 처리된 B/L |
| Switch B/L | - | 삼국간 거래에서 송하인/수하인 등을 교체해 재발행한 B/L |
| Seaway Bill | - | 유가증권 성격이 없는 비유통 운송증서 |
| Non B/L | NBL | B/L이 발행되지 않는 업무 전반. **상위 운송 모드(§1.8)의 한 값**. 내부적으로 Work Division 4종(Sea / Air / Warehouse / Trucking)으로 세분된다. E-24 확장 엔티티로 관리 (§4.5) |
| 운송 모드 | Transport Mode / **JOB_DIV** | E-08 기본 속성. 값: Sea / Air / Truck / Non B/L. **사용자 입력 불가능한 고유 작업 구분자** — 어느 Entry 화면에서 접근했는지에 따라 자동 결정된다 (S-02 → Sea/Air · S-06 → Truck · S-08 → Non B/L). DDL 컬럼명은 `JOB_DIV`. 본 PRD에서는 "운송 모드" 명칭을 사용하되, DDL 수준에서는 JOB_DIV로 매핑 (§1.8) |
| Work Division | - | **E-24 고유 속성**. 값: Sea / Air / Warehouse / Trucking. **사용자 입력 필수 (드롭다운)** — 미선택 시 저장 불가. Non B/L 업무 내부의 하위 분류이며, 값에 따라 E-24가 발췌하는 원천 확장(E-10 · E-11 · E-20) 또는 공통 본체의 필드 구성이 달라진다. **JOB_DIV와는 다른 계층의 개념** — JOB_DIV는 운송 모드 판정자, Work Division은 Non B/L 내부 하위 구분자 |
| 비엘발행타입 | - | Original / Surrender / Seaway / Normal / Express 등 발행 형태 구분 (§1.8 B/L Type 참조) |
| 항공 운송증서 | AWB (Air Waybill) | 항공 운송증서. 해상의 B/L에 대응. HAWB·MAWB로 구분 (§4.3) |
| 하우스 항공 운송증서 | HAWB (House Air Waybill) | 포워더가 실화주 앞으로 발행하는 항공 운송증서. **IATA 표준 양식 고정** |
| 마스터 항공 운송증서 | MAWB (Master Air Waybill) | 포워더가 Co-Load 등으로 발행하는 Master 레벨 항공 운송증서. **IATA 표준 양식 고정**. 해상 Master B/L과 달리 **포워더 발행·출력 대상** |

### 1.2 당사자 (Parties)
| 한글 | 영문/약어 | 정의 |
| --- | --- | --- |
| 송하인 | Shipper | 수출자. 운송인에게 화물을 위탁하는 측 |
| 수하인 | Consignee | 수입자. 화물을 수령하는 측 |
| 통지처 | Notify Party | 도착통지를 받는 제3자 |
| 서류상 파트너 | DOC Partner | 포워딩 업무에서 B/L 등 운송 서류상에 표시되는 파트너 |
| 정산 파트너 | Settle Partner | 실제로 운임을 정산하는 파트너. 운임 입력 시 기본 파트너로 참조됨 |
| 실제 고객 | Actual Customer | B/L상의 수하인과 별개로 실제 제반 비용을 지불하는 고객 |
| 운송인 | Carrier | 실제 운송을 수행하는 선사 또는 항공사 |
| 선사 | Liner | 해상 정기선 운송인. 운송인(Carrier) 중 해상 정기선 사업자를 지칭하는 실무 표현. 화면의 Schedule·Freight 블록에서 사용 |
| 포워더 | Forwarder | 화주를 대신해 운송을 주선하는 당사자 |
| 선사대리점 | Line Agent | 선사를 대행하는 대리점 |
| 해외 파트너 | Partner | 상대국 포워더. 현지 업무 대행 |
| Co-Load Agent | - | 공동 혼재 상대 포워더 |
| 공급자 | Supplier | 실제 제조·공급 주체 |
| 바이어 | Buyer | 최종 구매자 (실화주와 구분될 수 있음) |
| 운송사 | Trucker / Transporter | 내륙운송을 담당하는 업체 |
| 보세운송사 | Bonded Transport | 보세구간 내 운송을 담당하는 업체 |
| 통관사 | Customs Broker | 수출입 통관을 대행하는 업체 |
| B/L 오너 | B/L Owner | 비엘 소유자. 실화주인지 포워더인지의 구분 |
| Sign 담당자 | - | B/L에 서명하는 책임자 |

> 비고: DOC Partner와 Settle Partner는 일치하는 경우가 많으나, 서류상 표시되는 주체와 실제 정산 주체가 다를 수 있어 분리 관리한다. Actual Customer는 B/L 수하인(Consignee)과 실제 비용 지불 주체가 다른 경우를 위해 별도로 관리한다.

| 발행지 | Place of Issue | B/L이 발행되는 장소 |
| Payable At | - | 운임이 지급되어야 하는 장소 |

### 1.9 결제·부대 비용

> 운임 산정·정산 자체는 BMS 모듈의 영역. 여기서는 FMS 업무(B/L 발행·D/O·반출입 등)에서 직접 사용되는 항목만 정의한다.

| 한글 | 영문/약어 | 정의 |
| --- | --- | --- |
| Prepaid/Collect | PP / CC | 운임의 선불/후불 결제 구분 |
| Demurrage | Dem | 체선료. 무료기간 초과 후 화물 반출이 지연될 때 부과 |
| Detention | Det | 체화료. 무료기간 초과 후 컨테이너 반납이 지연될 때 부과 |
| Free Time | - | 체선·체화 부과 전 허용되는 무료 기간 |
| 선사 부킹번호 | Line Booking No | 선사로부터 부여받는 부킹 확인번호 |

- 조건: Incoterms, Freight Term
- 기타: 상품, HS코드

#### E-03 Master B/L 해상 확장

- POL·POD Demurrage / Detention 정보 (Free Time, 기준 일자)

#### E-08 House B/L (공통)
- 식별: HBL 번호, EDI HBL 번호
- Master 연결: 상위 Master B/L 참조
- 업무 분류: 운송 모드(`JOB_DIV`) · 방향 · Cargo Type · 적재 유형 · 서비스 조건 · B/L Type · Shipment Type · Sales Class
  - **운송 모드 `JOB_DIV`**: `Sea` / `Air` / `Truck` / `Non B/L` 중 1값. **사용자 입력 불가능한 고유 작업 구분자** — Entry 화면별로 자동 결정된다 (§1.8). 값에 따라 어느 확장 엔티티가 붙는지 결정 (§3.2)
- 당사자·운송 주체·경로·일정·물량·조건은 Master B/L과 **유사한 구조를 가지나 동일 값을 보장하지 않는다.** House B/L과 Master B/L은 각각 독립된 당사자·경로·일정 값을 가질 수 있으며, 아래와 같은 사유로 값이 달라지는 경우가 빈번하다:
    - **당사자**: Master B/L의 당사자는 선사 레벨(포워더 ↔ 선사 간)이고, House B/L의 당사자는 포워더 레벨(실화주 ↔ 포워더 간)이다. 일반적으로 Master의 Shipper는 포워더 자신 또는 Co-Load Agent이며, House의 Shipper는 실화주다.
    - **경로·일정**: 삼국간 거래·환적 등으로 Master와 House의 경로·일정이 다를 수 있다 (§1.8 Triangle 참조).
    - **물량**: Master 1건에 다수의 House가 붙는 혼재(LCL, Co-Load)의 경우, House 각각의 물량 합이 Master의 물량과 일치하지만 개별 House는 Master와 다른 값을 가진다.
- 구조(필드 구성)는 Master B/L과 정렬되어 있으나, **값은 각 레벨에서 독립 관리**된다.
- HBL 특화: 순중량, 내부 포장수량, 위험물(최대 5종), Sign 담당자, 발행일, D/O 발행일
- 영업/실적 속성:
  - Operator (담당자) — B/L의 실 운영 담당자
  - Team (담당팀)
  - Sales Man (영업 담당자)
  - Customer PIC (고객사 직원 — 거래처 쪽 담당자)
  - 위 필드는 실적 집계 및 List 조회 필터링에 사용된다.
  - 권한 모델과는 분리된다 (§6.7 참조).
- Performance 속성: Actual Customer, Settle Partner
- **집계·실적 요약 속성** (비정규화 컬럼 — E-14b 배정 기준 사전 계산값):
  - `total_allocated_teu` — Σ E-14b.allocated_teu
  - `total_container_count` — COUNT DISTINCT container_id in E-14b
  - `total_gross_weight_allocated` / `total_cbm_allocated` — 필요 시 확장
  - **용도**: (a) HBL List 화면(S-03/S-05) 인라인 집계 쿼리의 N+1·3자 조인 회피, (b) MongoDB Fact Mart 동기화의 원천값. (§6.11·NFR §3.2 참조)
  - **갱신 정책**: E-14b CRUD 또는 E-14a 타입 변경 시 도메인 이벤트 기반 갱신. **최종 일관성** 기반(수 분 지연 허용). 구현 패턴(트랜잭션 내 직접 갱신 / Outbox / CDC)은 Java 첫 엔티티 커밋 시점 확정 (TECH_STACK §5.1)

#### E-17 House B/L 면장

- 면장 식별 번호 — **수출신고필증 번호 또는 화물관리번호 중 하나**(배타). 한국 관세청(UNIPASS) 체계 기준

- HSN 번호

#### E-14a Container (물리)
> 물리 컨테이너 1건을 SSOT로 보관하는 엔티티. 이전 판의 `E-14 House B/L 컨테이너`가 갖던 "컨테이너 자체 속성" 부분을 담당한다. House B/L과의 연결(파트 혼재·파트율 등)은 **E-14b 교차 엔티티**에서 관리한다 (§3.1·§6.3 참조).

- 식별: 컨테이너 번호, 컨테이너 타입
- **길이 속성**: `length_feet` — 20 / 40 / 45 등 정수 피트 값. **TEU 계산의 SSOT**. Container Type 코드(20GP·40HQ·45ft 등)와 별도 컬럼으로 직접 보관한다. Type 코드는 규격·용도(GP/HQ/RF/OT/Tank 등) 구분 표시용이며, 길이 매핑의 원천이 아니다. 비표준 길이(53ft 등 특수 컨테이너)가 들어오더라도 산식이 그대로 적용된다.
- 물리 속성: 봉인번호(최대 6), 타르중량, VGM
- 운영 속성: 소유 구분, F/E 플래그, 내륙운송방법 (Round Type)
- **TEU 환산 산식**: `teu = length_feet / 20`. 20ft=1.0 / 40ft=2.0 / 45ft=2.25 등. 별도 코드 테이블·factor 매핑을 두지 않는다 (산식이 규칙 자체)
- 카디널리티: 파트 혼재 시에도 물리 컨테이너 1건만 존재하며, 여러 House B/L이 E-14b 교차 엔티티를 통해 공유한다

#### E-14b House-Container 배정
> **House B/L : Container 간 N:M 관계를 해소하는 교차 엔티티**. 관계 속성과 사전 계산 측정값을 함께 보관한다.

- 식별: `(house_bl_id, container_id)` 복합키
- 관계 속성:
  - `is_part` (파트 여부 불리언) / `part_ratio` (파트율, 0~1 실수)
  - `is_main` (메인 House 지정 여부) — 하나의 물리 컨테이너를 공유하는 House들 중 대표 House 지정 플래그 (※ OPEN_ITEMS 재확인 항목 — `is_main`이 "메인 House 지정"인지 "메인 컨테이너 지정"인지 업무 의미 최종 확인 필요)
- **사전 계산 측정값** (실적·집계 쿼리 성능 보장용):
  - `allocated_teu` = `Container.length_feet / 20 × part_ratio` (E-14a §TEU 환산 산식 준용)
  - `allocated_gross_wt` / `allocated_cbm` — 필요 시 확장
- 총중량·CBM의 **배정 몫**(이 House가 이 컨테이너에서 차지하는 값)은 본 엔티티의 사전 계산 측정값으로 보관한다. Container의 총중량/CBM 원시값은 E-14a에서 별도 관리(파트 합의 검증·UI 표시용).
- 카디널리티:
  - House B/L : E-14b = **1:N**
  - Container (E-14a) : E-14b = **1:N**
  - 파트 혼재가 아닌 일반 케이스: `is_part = false`, `part_ratio = 1`, `is_main = true` 단일 행
- 갱신 정책: 본 엔티티의 CRUD는 도메인 이벤트 발행 → E-08 요약 컬럼 갱신 및 MongoDB Fact Mart 동기화 트리거 (§6.11)

> **레거시 호환 메모**: 레거시 `TB_FMS_HBL_CNTR` 류의 비정규화 구조(House당 컨테이너 번호 중복 저장)는 신규 설계에서 채택하지 않는다. 레거시 DDL 수신 후 이관 규칙은 별도 검토 — 단, 본 프로젝트는 레거시 스키마 직접 마이그레이션 범위 밖이므로 (TECH_STACK §5 스키마 관리 정책) 신규 스키마 기준 설계만 확정한다.

#### E-20 House B/L 트럭 정보
> Trucking 모드 House B/L에서 1:N으로 반복되는 **Truck Information 그리드 행**이다. 하나의 Trucking B/L에 여러 트럭 차수를 기록할 수 있다. 해상·항공 모드에는 붙지 않으며, 해상·항공의 내륙 운송이 별도로 필요한 경우 독립 Trucking B/L로 등록한다 (§4.4 참조).
- 식별: 업무번호(HBL 외래키), 트럭 순번, 트럭 오더 번호(내부 관리용)
- 차량: 트럭 번호, 트럭 타입
- 기사: 기사 코드, 기사명, 기사 모바일 번호
- 적재: 포장수량, 포장단위 코드, 총중량, CBM
- 컨테이너 동반(선택): 컨테이너 번호, 컨테이너 타입, 봉인번호 최대 3개. 이 필드들은 **E-14a·E-14b와 별개로 E-20 행 내부에서 간소 관리**된다. 해상 컨테이너의 풍부한 속성(VGM·Part 혼재·Main 지정 등)은 다루지 않는다 (§6.3 참조).
- 비고: 자유 텍스트 1필드. **운송 상태 코드 체계가 따로 없으므로 배차·픽업·인도 등의 상태 메모도 본 필드에 자유 서술**한다.

> **레거시 관찰 사항**: 본 엔티티는 레거시 `TB_FMS_HBL_TRUCK` 테이블과 1:1 대응된다. `NBL_JOB_DIV`는 사문화 컬럼이므로 신규 설계에서 반영하지 않는다.

#### E-21 Switch B/L
본 엔티티는 **Switch B/L 모달의 필수 입력 필드만으로 구성**된다. 모달에 노출되지 않는 속성(POL/POD 교체 경로, 기타 부가 속성 등)은 **저장하지 않고 원본 House B/L(E-08)을 조회 시점에 참조**한다. (02_SCREENS §S-02 Switch B/L 모달 참조)
- Switch B/L 번호
- B/L Type (해상 변형에만 해당. 항공은 AWB 특성상 B/L Type 개념 부재)
- Incoterms
- 교체 적용 당사자: 송하인 / 수하인 / 통지처

#### E-22 Switch B/L 설명
- Marks and Numbers (2단 분할 텍스트)
- Nature & Quantity of Goods (2단 분할 텍스트)
- 원본 House B/L 설명(E-13)과 독립 저장

#### E-23 거래처 (Party)
> **주의**: 본 엔티티는 추후 MDM(마스터 데이터) 모듈로 분리될 예정이다. FMS MVP 단계에서는 본 도메인 내에서 관리하며, 동작 검증 후 MDM 합류 시점에 이관한다. 이관 시 API 계약은 유지하되 소유 모듈과 마스터 저장 위치가 이동한다.
- 식별: 거래처 코드, 사업자번호 등 외부 식별자
- 기본 정보: 이름(다국어), 주소, 연락처, 국가
- 역할 가능성: 송하인 / 수하인 / 통지처 / 운송인 / 선사대리점 / 해외 파트너 / Co-Load Agent / 공급자 / 바이어 / 운송사 / 보세운송사 / 통관사 / B/L 오너 / Sign 담당자 등 (하나의 거래처가 복수 역할을 가질 수 있음)
- 외부 코드 매핑: 선사·관세청 등 외부 시스템이 사용하는 코드 체계와의 연결 정보
- 원천 구분: 자가 관리 거래처 / 외부 코드 거래처

#### E-24 House B/L Non B/L 확장

> **구성 방침**: Non B/L 업무는 내부적으로 **Work Division** 4종(Sea / Air / Warehouse / Trucking)으로 세분된다. 본 확장은 Work Division 값에 따라 원천 확장(E-10 해상 / E-11 항공 / E-20 트럭) 또는 공통 본체의 속성 중 **Non B/L 업무 수행에 필요한 필드만 선택 발췌**하여 구성한다. 필드 구성의 구체 목록은 S-08 Non B/L Entry(02_SCREENS)가 SSOT가 되며, **신규 DDL 구현 시점에 기준 컬럼과 부가 컬럼의 구분을 확정**한다.

- 카디널리티: E-08 : E-24 = **0..1**
- 존재 조건: **E-08의 `JOB_DIV` = `Non B/L` 일 때만 존재**. E-10·E-11·E-20 중 어느 것과도 동시에 붙지 않는다 (§3.2 상호 배타)
- **Work Division 속성** (E-24 고유 속성): `Sea` / `Air` / `Warehouse` / `Trucking` 중 하나. E-24 레코드당 1값
  - **사용자 입력 필수** — S-08 화면에서 드롭다운으로 선택. 미선택 시 저장 불가
  - **JOB_DIV와는 다른 계층의 개념** — JOB_DIV는 E-08이 갖는 운송 모드 판정자(입력 불가), Work Division은 E-24가 갖는 Non B/L 내부 하위 구분자(입력 필수)
  - 값에 따른 필드 발췌 방향:
    - `Sea` — 해상 화물에 대한 Non B/L 부대 업무. E-10 필드 중 발췌
    - `Air` — 항공 화물에 대한 Non B/L 부대 업무. E-11 필드 중 발췌
    - `Warehouse` — 창고 보관 업무 전반. 공통 본체(E-08) + E-24 고유 필드 중심. **운송 서류를 동반하지 않는 부대 업무 처리용** 슬롯이며, 전용 화면·엔트리를 별도 할애할 정도가 아니라 Non B/L 내 Work Division으로 통합한 설계적 선택
    - `Trucking` — 트럭 관련 Non B/L 부대 업무. E-20 필드 중 발췌. **상위 운송 모드의 Trucking (E-20이 붙는 독립 Trucking B/L, `JOB_DIV` = `Truck`)과 구별**된다 — 전자는 독립 Trucking B/L(S-06)로 다루고, 후자는 Non B/L 업무(S-08) 범주
- 원천 참조 범위: 공통 본체(E-08)의 필드와는 중복되지 않는다
- 원본 B/L과의 관계: Non B/L 업무는 대개 원본 해상/항공 B/L이 존재하는 화물에 대해 수행되지만, Non B/L B/L은 **별도 B/L 레코드로 신규 생성**된다 (E-08 신규 행 + E-24 신규 행). 원본 B/L과의 업무 연결은 참고번호(E-18) 등으로 기록한다
- Master 없이 단독 존재 가능 (§4.5)

## 3. 엔티티 관계

### 3.1 관계도

```
Master B/L (공통) ─┬─ 1:1 ─ Master B/L 기타정보
                  ├─ 0..1 ─ Master B/L 해상 확장
                  ├─ 0..1 ─ Master B/L 항공 확장
                  ├─ 1:1 ─ Master B/L 설명
                  ├─ 1:N ─ Master B/L 치수
                  ├─ 1:N ─ Master B/L Schedule Leg (E-07)  (항공 모드에서만; §6.12)
                  └─ 1:N ─ House B/L
                             │
House B/L (공통) ─┬─ 1:1 ─ House B/L 기타정보
                 ├─ 0..1 ─ House B/L 해상 확장
                 ├─ 0..1 ─ House B/L 항공 확장
                 ├─ 1:N ─ House B/L 트럭 정보  (Trucking 모드에서만; 그리드 행)
                 ├─ 0..1 ─ House B/L Non B/L 확장  (Non B/L 모드에서만)
                 ├─ 1:1 ─ House B/L 설명
                 ├─ 0..1 ─ Switch B/L ─ 1:1 ─ Switch B/L 설명
                 ├─ 1:N ─ House B/L 치수
                 ├─ 1:N ─ House B/L HS코드
                 ├─ 1:N ─ House B/L 면장
                 ├─ 1:N ─ House B/L 참고번호
                 ├─ 1:N ─ House B/L Schedule Leg (E-19)  (항공 모드에서만; §6.12)
                 └─ 1:N ─ House-Container 배정 (E-14b, 교차)
                            │                       │
                            │                      N:1 ─ Container (E-14a, 물리)
                            │
                           1:N ─ 컨테이너 아이템 (E-15)

거래처 (Party) ─── N:1 ─── Master B/L / House B/L / Switch B/L 등의 각 역할 슬롯
```

### 3.2 카디널리티 주석
- **Master B/L : House B/L = 1 : N** — 한 건의 Master 아래 다수의 House(혼재화물)가 붙는다. Master와 각 House는 당사자·경로·일정 등 주요 필드를 독립적으로 가지며, 값이 일치할 수도 다를 수도 있다.
- **B/L : 모드별 확장** — 모드별 확장의 카디널리티는 모드마다 다르다.
  - **해상 확장 / 항공 확장 / Non B/L 확장**: 0..1. 해당 모드일 때 확장 1건이 본체에 붙는다.
  - **트럭 정보(E-20)**: **1:N 그리드**. Trucking 모드일 때 Truck Information 그리드 행이 여러 개 붙을 수 있다 (차수·배차 단위).
  - 해상·항공·Trucking·Non B/L은 상호 배타: B/L의 운송 모드에 따라 해상 확장(E-10) / 항공 확장(E-11) / 트럭 정보 그리드(E-20) / Non B/L 확장(E-24) **중 하나만** 존재한다.
- **컨테이너 : House B/L = N:M (E-14b 교차 엔티티로 해소)** — 물리 컨테이너는 E-14a에, House와의 연결(파트 여부·파트율·메인 여부·배정 TEU)은 **E-14b House-Container 배정**에 보관한다. 파트 혼재가 아닌 일반 케이스는 E-14b에 1행만 존재(`is_part=false`, `part_ratio=1`, `is_main=true`). 파트 혼재 시 하나의 물리 컨테이너(E-14a 1행)에 여러 House가 각자의 E-14b 행을 통해 연결된다. E-15 컨테이너 아이템은 E-14b에 종속되어 어느 House의 아이템인지 항상 식별된다. E-14a·E-14b와 Trucking B/L 내부의 E-20 컨테이너 필드는 별개 구조다 (§6.3 참조).
- **면장** — **수출 방향 House B/L에만 연결된다.** 분할 면장과 동시포장을 지원한다. 여러 House가 한 면장을 공유하거나, 한 House가 여러 면장을 가질 수 있다. 선적 분할(Partial Shipment)은 별도 속성으로 관리한다. 수입 방향 면장은 포워더 업무 범위 밖이라 미지원.
- **Trucking (내륙운송)** — 해상/항공과 동등한 운송 모드의 하나로, House B/L 공통 본체 + Truck Information 그리드(E-20, 1:N)로 구성된다. Master 없이 단독 존재 가능하다. **해상/항공 B/L의 내륙 구간 운송이 필요한 경우에도 해당 B/L에 E-20을 붙이지 않고, 별도의 독립 Trucking B/L을 등록해 관계를 맺는다** (§4.4 참조).
- **Non B/L 업무** — B/L이 발행되지 않는 업무 전반. House B/L 체계 안에 담되 **Non B/L 전용 확장(E-24) 1건이 붙는다**. E-24는 Work Division 속성을 갖고 내부적으로 Sea·Air·Warehouse·Trucking 4종으로 세분된다. 필드는 Work Division 값에 따라 해상/항공/트럭 확장의 선택 발췌로 구성한다 (§4.5). Master 없이 단독 존재 가능하다. 원본 해상/항공 B/L이 있는 경우에도 Non B/L B/L은 별도 레코드로 신규 생성되며, 원본과의 연결은 참고번호(E-18) 등으로 기록한다. **상위 운송 모드의 Trucking(E-20 붙는 독립 B/L)과 Non B/L 내부 Work Division의 Trucking은 다른 계층의 개념**임에 유의한다 (§4.4·§4.5).
- **Switch B/L** — 원본 House 1건당 0 또는 1건. **재Switch는 허용되지 않는다.**
- **Schedule Leg (E-07 · E-19) — 항공 전용** — Master/House 각각 1:N 관계로 항공 Schedule 그리드의 leg를 저장한다. 직항은 1행, N회 환승은 (N+1)행. 그리드 **첫 행의 On Board = ETD**, **마지막 행의 To = POD**, **마지막 행의 Arrival = ETA** 가 자동 파생된다 (판정 기준은 그리드 **표시 순서**, §6.12 B/L 출력 연동 원칙 참조). 해상·Trucking·Non B/L에는 붙지 않는다 — 해상은 실무상 경유지 개념이 있으나 종이 B/L에 출력하지 않으므로 본 시스템에서 관리하지 않는다 (§6.12).
- **거래처 : 역할** — 하나의 거래처가 여러 역할(송하인·수하인·바이어 등)을 동시에 가질 수 있다. 각 B/L의 역할 슬롯은 거래처 마스터를 참조한다.

## 4. 핵심 비즈니스 프로세스

### 4.1 해상 수출
1. **부킹** — 선사에 부킹 요청, 선사 부킹번호 확보
2. **S/R 접수** — 화주로부터 선적요청서 접수
3. **Master 생성** — 선사·선박·항차·경로·일정 등록
4. **House 생성** — 화주별 House B/L 등록, Master에 연결
5. **컨테이너 배정** — FCL: 컨테이너 정보 입력 / LCL: 창고 반입
6. **면장 등록** — 수출 면장 정보 연결
7. **EDI 적하목록** — 관세청으로 적하목록 전송, MRN/MSN 확보
8. **B/L 발행** — 발행타입(§1.8 B/L Type) 결정, O B/L 접수
9. **출항** — 본선적재·ATD 확정

### 4.2 해상 수입
1. 해외 파트너로부터 사전 정보 수신(Pre-Alert) → Master/House 생성
2. 입항 전 EDI 적하목록 수신·처리 (MRN/MSN)
3. **A/N 발행** — 수하인에게 도착통지
4. **D/O 발행** — 인도지시서 발행
5. **터미널 반입** — 하선장소 기준 반입 처리
6. **보세운송** — 필요 시 보세구간 내륙 이동
7. **통관** — 자가통관 또는 통관사 대행
8. **내륙운송** — 해상 B/L 자체에 트럭 차수를 붙이지 않는다. 내륙 구간이 별도 계약·정산 대상이면 **독립 Trucking B/L로 등록**하고 참고번호(E-18)로 해상 B/L과 연결한다 (§4.4, §6.3 참조).
9. **Cargo Delivery** — 최종 인도 완료

### 4.3 항공 수출/수입
- Master는 MAWB, House는 HAWB 개념에 대응된다.
- 항공 특화: Rate Class, 볼륨중량, 보안검색 상태, 여객기/화물기 구분, Charter(차터) 운영.
- 운송 모드 속성으로 해상과 분기된다.

### 4.4 Trucking (내륙운송)
- 해상·항공과 동등한 운송 모드. 내륙운송만 수행하는 업무를 House B/L로 입력·관리한다.
- **화면 구성은 공통 본체 입력 폼 + Truck Information 그리드(E-20)**. 하나의 Trucking B/L에 여러 트럭 차수를 그리드 행으로 쌓을 수 있다.
- 컨테이너를 동반하는 내륙 운송(예: 터미널→내륙창고 간 컨테이너 셔틀)의 경우 해당 그리드 행 내부에 컨테이너 번호·타입·봉인번호 3개를 함께 기록한다 (§6.3).
- **Master 없이 단독 존재할 수 있다.**
- 해상/항공 House B/L과 동일한 조회·검색·B/L 체계를 공유하되, 화면은 단독 Truck B/L Entry(S-06)에서 입력한다 (SCREENS §B의 해상/항공 4변형과 별도).
- **해상/항공 B/L의 내륙 구간 운송이 필요한 경우에도 해당 B/L에 E-20을 붙이지 않는다.** 대신 독립 Trucking B/L을 별도로 등록하고 참고번호(E-18) 등으로 원 해상/항공 B/L과 업무 연결을 기록한다. 본 MVP는 범용 포워더(실화주 대상) 범위에 맞춰 이 정책을 택한다. 해상/항공 B/L에 직접 내륙 차수를 붙이는 MTB(Multimodal Through B/L) 성격 운영은 현재 범위 밖이다.
- 레거시 `NBL_JOB_DIV`는 사문화 컬럼이므로 신규 설계에서 반영하지 않는다. Trucking / Non B/L / 해상 / 항공의 구분은 운송 모드(§1.8) 속성으로 충분하다.
- 내륙 운송 상태 코드 체계는 정의하지 않는다. 배차·픽업·인도 등의 진행 상태는 E-20 "비고" 자유 텍스트에 서술한다.

### 4.5 Non B/L 업무 (Work Division 4종)
- B/L이 발행되지 않는 업무 전반. **House B/L 공통 본체(E-08) + Non B/L 확장(E-24) 구조로 관리**한다.
- 진입 조건: E-08의 운송 모드 `JOB_DIV` = `Non B/L` (Entry 화면 S-08 접근 시 자동 설정)
- Non B/L 업무는 내부적으로 **Work Division 4종**으로 세분된다. Work Division은 E-24의 고유 속성이며 **사용자 입력 필수**(드롭다운, 미선택 시 저장 불가):
  - `Sea` — 해상 화물에 대한 Non B/L 부대 업무 (E-10 필드 발췌)
  - `Air` — 항공 화물에 대한 Non B/L 부대 업무 (E-11 필드 발췌)
  - `Warehouse` — 창고 보관 업무 전반. 운송 서류를 동반하지 않는 부대 업무 처리용 슬롯으로, 양이 많지 않고 전용 화면 할애가 과하다는 판단 하에 Non B/L 내 Work Division으로 통합 (E-08 공통 본체 + E-24 고유 필드 중심)
  - `Trucking` — 트럭 관련 Non B/L 부대 업무 (E-20 필드 발췌)
- **두 계층 개념 구분**
  - **JOB_DIV (상위 운송 모드)**: E-08 기본 속성. Sea/Air/Truck/Non B/L. 입력 불가, 화면에서 자동 결정. 붙는 확장 엔티티를 결정
  - **Work Division (Non B/L 내부 구분)**: E-24 고유 속성. Sea/Air/Warehouse/Trucking. 입력 필수, 사용자 드롭다운 선택. E-24의 필드 조립 원천을 결정
  - 특히 `JOB_DIV = Truck`(독립 Trucking B/L, S-06)과 `Work Division = Trucking`(Non B/L 내 트럭 부대 업무, S-08)은 **서로 다른 계층**의 개념이며, 업무 성격에 따라 입력 경로를 선택한다
- **필드 구성과 기준 컬럼/부가 컬럼 구분은 신규 DDL 구현 시점에 확정**한다. Work Division 값별로 필수·옵션 필드의 분포가 다르므로 DDL 설계 시 이를 반영한다.
- **Master 없이 단독 존재할 수 있다.**
- Non B/L 업무 대상 화물에 원본 해상/항공 B/L이 존재하더라도 Non B/L B/L은 **별도 B/L 레코드로 신규 생성**된다. 원본과의 업무 연결은 참고번호(E-18) 등으로 기록한다.
- 해상/항공 House B/L과 동일한 조회·검색·B/L 체계를 공유하되, 화면은 단독 Non B/L Entry(S-08)에서 입력한다. S-08 화면의 필드 목록(Work Division 분기 포함)이 E-24의 필드 구성 SSOT가 된다.
- **구현 메모** — 백엔드에서 non-bl/truck-bl은 house-bl 도메인 내부 엔티티(`HouseBlNonBlJpaEntity`, `HouseBlTruckJpaEntity`, `HouseBlTruckOrderJpaEntity`)로 흡수됨. 도메인 모델로는 E-24 독립이나 패키지는 `com.freightos.fms.domain.housebl` 하위에 위치. 프론트엔드 헥사고널(`domain/`, `application/`, `adapter/`) 적용은 house-bl / master-bl / switch-bl 3개 도메인에만 현재 완료, non-bl · truck-bl은 후속 적용 후보.

## 6. 규칙 / 제약

### 6.1 B/L 및 문서 출력
- B/L·AWB의 **발행 수량·원본 수량·접수·Surrender·회수 여부** 등 발행 상태와, A/N·D/O·S/A·S/R의 **발행·발송·컨펌 이력**은 **시스템이 관리하지 않는다.** 업무 담당자의 관리 영역이다. (본 §6.1은 "시스템이 관리하는 이력"의 범위를 정의하는 것이며, §4.1~§4.3의 업무 프로세스 기술과는 별개다.)
- 문서 출력 실행 시 **Print date 타임스탬프만** MongoDB 로그 다큐먼트에 기록한다 (SCREENS 부록 §C-4, TECH_STACK §5와 정합).
- 출력 대상:
  - **B/L**: 해상 House B/L **수출 전용**.
  - **HAWB**: 항공 House B/L **수출 전용**. IATA 표준 양식 고정.
  - **MAWB**: 항공 **Master B/L 수출 전용**. IATA 표준 양식 고정. 포워더가 발행·출력한다.
  - **A/N · D/O**: 해상 House B/L **수입 전용**.
  - **S/A · S/R**: 해상·항공 **수출**에서 House 및 Master 단위로 출력 가능.
  - **Non B/L 원본 B/L**: Non B/L 업무에서 출력.
  - **제외**: **해상 Master B/L 본체**(선사 발행), **Trucking B/L**. **항공 MAWB는 포워더가 발행하므로 제외 대상이 아니다.**
  - 수입 업무에서 화주가 받는 B/L·AWB는 **해외 파트너 발행 서류**이며 본 시스템에서 출력하지 않는다.
- 모든 출력 템플릿은 **영어 고정**이다. 템플릿은 Entry 값 단순 바인딩이며, 출력 시점의 자동 채움·자동 계산 로직은 두지 않는다 (A/N Date · D/O Date · S/R 일자 등은 Entry에서 수동 입력).

### 6.2 위험물
- House B/L 한 건당 위험물은 **최대 5종까지** 관리된다. **Master B/L 레벨에서는 위험물을 관리하지 않는다.**
- 컨테이너 아이템 단위로도 위험물이 지정될 수 있다.

### 6.3 컨테이너
본 시스템에는 **두 가지 독립된 컨테이너 관리 구조**가 존재한다. 섞어서 쓰지 않는다.

**(가) 해상 House B/L의 컨테이너 — E-14a + E-14b 2계층 구조**
- **E-14a Container (물리)**: 정식 컨테이너 엔티티. 하나의 컨테이너에 **최대 6개의 씰 번호**가 부여될 수 있다. VGM, 타르중량, 소유 구분, F/E 플래그, 내륙운송방법 등 컨테이너 자체 속성을 관리한다. 물리 컨테이너 1건당 1행.
- **E-14b House-Container 배정** (교차 엔티티): House B/L과 Container 간 N:M 관계를 해소한다. 파트 여부·파트율·메인 여부 등 **관계 속성**과 `allocated_teu` 등 **사전 계산 측정값**을 보관한다.
- 파트 혼재는 "같은 Container (E-14a 1행)에 여러 House가 각자의 E-14b 행으로 연결"되는 형태로 표현된다. 파트 혼재가 아닌 일반 케이스는 `is_part=false`, `part_ratio=1`, `is_main=true`로 E-14b에 1행만 존재.
- **E-15 컨테이너 아이템은 E-14b에 종속**되어, 파트 혼재 상황에서도 아이템이 어느 House의 것인지 항상 식별된다.
- **N:M:L 실적 쿼리 문제 해소**: 과거 설계(E-14 단일 엔티티)에서는 실적 리포트의 HBL×Container×Freight 3자 조인이 카티시안 폭발을 일으켰다. 신규 설계에서는 (i) E-14b의 `allocated_teu`를 실적 단위로 직접 합산, (ii) HBL 본체의 요약 비정규화 컬럼으로 Container 조인 없이 TEU 취득, (iii) 실적·분석 쿼리 자체는 MongoDB Fact Mart 경유(§6.11)로 분리하여 해소된다.

**(나) Trucking B/L의 동반 컨테이너 — E-20 행 내부**
- Trucking 모드의 Truck Information 그리드 행(E-20)이 컨테이너를 동반할 때, 해당 행 내부에 컨테이너 번호·타입·**봉인번호 최대 3개**만 간소 기록한다.
- VGM·Part 혼재·Main 지정 같은 해상 컨테이너 전용 속성은 다루지 않는다.
- E-14a·E-14b와 FK 관계로 연결되지 않으며, 독립 기록이다. Trucking B/L의 컨테이너는 보통 터미널↔내륙창고 구간 이동 같은 물리적 이동 기록 수준이라 간소 구조로 충분하다는 전제.

### 6.4 면장
- **수출 방향 전용**: 본 시스템은 수출 면장만 관리한다. 수입 방향 면장은 포워더 업무 범위가 아니라 본 시스템 대상이 아니다.
- 하나의 House B/L이 **다건 면장**을 가질 수 있다.
- **분할 면장(Partial License)**: 하나의 화물을 세관에 여러 차수로 나누어 신고하는 형태. 다중 면장 행 구조로 자연 수용된다.
- **동시포장**: 여러 면장이 동일한 포장 단위 안에 함께 들어가 있는 경우, 해당 관계가 명시적으로 표시된다.
- **선적 분할(Partial Shipment)**: 하나의 계약·면장 물량을 여러 차례에 걸쳐 나누어 선적하는 것. 세관 분할 면장과 별개 속성(E-17 여부·차수)으로 관리한다.

### 6.5 EDI / 세관
- 적하목록 전송 시 MRN / MSN, 선박국적, 콜사인 등이 필수다.
- 하선장소·적재장소·접안부두·반출입부두·하역사 등 물리적 장소·주체는 각각 표준 부호 체계로 관리된다.
- 항공 화물의 경우 보안검색 상태가 B/L 단위로 관리된다.

### 6.6 Switch B/L
- Switch B/L 발행 시 **원본 House B/L은 그대로 보존**하고, 교체 대상이 되는 당사자·경로 등만 별도로 저장한다. 원본 덮어쓰기는 허용되지 않는다.
- **재Switch는 허용되지 않는다.** Switch B/L은 원본 House 1건당 최대 1건까지만 발행된다.

### 6.7 운송 모드·방향
- 운송 모드(해상 / 항공 / Trucking / Non B/L)와 방향(Export / Import)은 모든 B/L의 **1급 속성**이다.
- **화면 분기**:
    - 해상 / 항공은 `{모드, 방향}` 4변형으로 **UI·라우팅 단위는 분리**(Variant ID `S-XX[MODE/DIR]`, 예: `S-02[SEA/EXP]`)되며, **내부 구현은 공통 Shell + config 조립**으로 처리한다. Entry(S-02·S-04) / List(S-03·S-05) 모두 동일 정책 (02_SCREENS 부록 §B-0, §B-1 참조).
    - Trucking은 방향을 내부 속성으로만 가지며 방향별 화면 분기를 하지 않고, **단독 화면(S-06 Truck B/L Entry)** 에서 입력한다.
    - Non B/L은 방향을 내부 속성으로만 가지며 방향별 화면 분기를 하지 않고, **단독 화면**에서 입력한다.
- 업무 플로우·권한은 운송 모드 단위로 분기하되, B/L 단위의 행(Row) 권한 모델은 도입하지 않는다.
- Operator / Team / Sales Man 필드는 실적 집계와 조회 필터링 용도이며, 특정인만 열람·수정 가능한 권한 가정은 적용하지 않는다. 대리 업무 등으로 영업자와 Operator가 다를 수 있기 때문이다.

### 6.8 업무 구분
- 업무 구분은 **두 계층의 속성**으로 표현된다.
  - **JOB_DIV (상위 운송 모드)**: E-08의 기본 속성. `Sea` / `Air` / `Truck` / `Non B/L` 중 1값. **사용자 입력 불가** — 어느 Entry 화면에서 접근했는지에 따라 자동 결정 (§1.8). 값에 따라 붙는 확장이 결정된다 (Sea → E-10, Air → E-11, Truck → E-20, Non B/L → E-24 — 4종 상호 배타)
  - **Work Division (Non B/L 내부 구분)**: E-24의 고유 속성. `Sea` / `Air` / `Warehouse` / `Trucking` 중 1값. **사용자 입력 필수** (드롭다운, §4.5). 값에 따라 E-24가 발췌하는 원천 필드 구성이 달라진다
- B/L이 발행되지 않는 Non B/L 업무도 동일 체계에 포함되며, 공통 본체(E-08) + Non B/L 확장(E-24, 0..1) 구조로 관리된다. Master 없이 단독 존재할 수 있다.
- **Trucking 개념의 두 계층 구분**
  - `JOB_DIV = Truck` — 해상·항공과 동등한 상위 운송 모드. House B/L 공통 본체 + Truck Information 그리드(E-20, 1:N)로 구성. 단독 화면(S-06)에서 입력. 내륙운송 전용 B/L을 다룬다
  - `Work Division = Trucking` — E-24의 속성값. Non B/L 성격의 트럭 부대 업무를 다룬다 (§4.5). 화면은 S-08
  - 두 계층은 다른 화면·다른 엔티티로 관리되며, 업무 성격에 따라 입력 경로를 선택한다

### 6.9 거래처 마스터
- 모든 역할(송하인·수하인·통지처·운송인·대리점·공급자·바이어·운송사·통관사·B/L 오너·Sign 담당자 등)은 **공통 거래처 마스터**를 참조한다. 역할별 별도 마스터는 두지 않는다.
- 하나의 거래처가 복수 역할을 동시에 가질 수 있다.
- **자가 관리 거래처**와 **외부 코드 체계의 거래처**(선사·관세청 등)는 애플리케이션 레벨에서 통합 관리되며, 필요 시 각각의 거래처 정보로 구분해 표현된다.
- **모듈 소유권**: FMS MVP 단계에서는 본 도메인 내에서 관리하되, 추후 MDM 모듈로 이관될 예정이다. 이관 이후에도 본 규칙(공통 마스터·복수 역할·통합 관리)은 유지된다.

### 6.11 실적·분석 쿼리 OLTP 우회 원칙

본 시스템은 OLTP(PostgreSQL)와 Analytics Mart(MongoDB)를 **명시적으로 분리**하며, 실적·분석 쿼리는 OLTP를 우회한다 (TECH_STACK §5·§5.1, NFR §3.2 참조).

- **조회 경로 분리**:
  - **OLTP(PostgreSQL)**: 원장 SSOT. 모든 업무 CRUD와 FMS 운영 화면(S-02~S-09 Entry/List) · HBL List 인라인 집계(Master 하위 House 건수·요약 등)는 OLTP에서 처리.
  - **Analytics Mart(MongoDB)**: 실적·분석·집계 전용. **PFM 실적 리포트 · BMS 고객사 특화 조회 화면 · 대시보드 집계 위젯(Weekly Volume 등) · 감사 조회 화면**은 전적으로 MongoDB에서 조회.
- **OLTP에 다차원 집계 쿼리 금지**:
  - OLTP 테이블을 직접 조인하여 `HBL × Container × Freight` 같은 3자 이상 집계 쿼리를 구성하지 않는다. 과거 레거시에서 이 경로가 실적 리포트 병목(30~120초)의 주 원인이었다.
  - BMS·PFM·대시보드 측 개발자는 "원장 데이터가 필요하면 OLTP, 실적·분석이 필요하면 Mart"를 기본 규칙으로 삼는다.
- **Freight 귀속 원칙**:
  - 운임(FREIGHT)은 **HBL 레벨에만 귀속**된다. Container 레벨 운임 귀속은 도입하지 않는다(레거시·고객사 관행 부재 확인). BMS 모델링의 기본 전제로 인수한다.
- **비정규화·동기화**:
  - OLTP 내부에서도 E-08(House B/L) 본체에 요약 비정규화 컬럼(`total_allocated_teu` 등)을 두어 HBL List 인라인 집계의 3자 조인을 회피한다 (§2.2 E-08 참조).
  - OLTP → MongoDB Fact Mart 동기화는 **이벤트 기반·최종 일관성**을 따른다. 수 분 단위 미스매치는 업무상 허용된다. 강한 일관성은 도입하지 않는다.
  - 동기화 구현 패턴(Outbox / 메시지 브로커 / Spring Batch 주기 워커 / CDC)과 Fact 다큐먼트 스키마·파티셔닝·재동기화 절차는 **PFM 기획 라운드** 또는 **Java 첫 엔티티 커밋 시점**에 확정한다 (06_OPEN_ITEMS 재개 시점 맵 참조).
- **경계 원칙 요약**:
  - "원장이 필요하면 OLTP, 분석이 필요하면 Mart"
  - "HBL 본체 요약 컬럼은 OLTP 병목 해소와 Mart 동기화 원천의 이중 역할"
  - "Freight는 HBL 귀속 단일 원칙"

### 6.12 B/L 출력 연동 데이터 모델 원칙

본 시스템의 B/L Entry 화면은 **"종이 B/L 출력물에 들어갈 내용을 입력받는 화면"** 이라는 정체성을 갖는다. 이 원칙이 여러 도메인 설계 결정의 근거가 된다.

- **핵심 규칙**: B/L에 인쇄·표기되지 않는 업무 영역의 데이터는 본 B/L 도메인(E-01~E-22)에서 관리하지 않는다. 이런 데이터는 (a) 업무 담당자의 외부 관리 영역이거나, (b) 다른 모듈(BMS·PFM·EDI 등)의 관리 영역이거나, (c) 애드온·확장 성격으로 후속 도입된다.
- **적용 예시** (본 원칙으로 설명되는 결정들):
  - **§6.12.a 경유지(Via) — 항공 전용** (§1.3·§2.1 E-07·E-19·§3.2): 해상도 실무상 경유지 개념이 존재하나 종이 B/L에 경유지를 출력하지 않으므로 관리 대상 아님. 항공은 HAWB·MAWB 양식에 Schedule/경유 정보가 인쇄되므로 E-07·E-19로 관리.
  - **§6.12.b Switch B/L 필드 축소** (§2.2 E-21·E-22, §6.6): Switch B/L 모달의 필수 입력 필드(교체 당사자·B/L Type·Incoterms)만 저장하고, 모달 미노출 속성(POL/POD 교체 경로 등)은 저장하지 않음. Switch B/L 인쇄 양식 범위에 해당하는 필드만 SSOT.
  - **§6.12.c Handling Information 항공 전용** (02_SCREENS §S-02 항공 변형): HAWB/MAWB 출력 양식의 Handling Information란에 바인딩. 해상 B/L 출력 양식에는 없는 영역이므로 해상 엔티티에 관련 필드 없음.
  - **§6.12.d B/L 업무 상태 시스템 미관리** (§6.1): 발행·회수·Surrender·발송·컨펌 이력 등은 B/L 인쇄 내용이 아니라 업무 프로세스 상태이므로 시스템이 관리하지 않음. 출력 실행 시점의 Print date만 MongoDB 로그에 기록.
  - **§6.12.e Freight = HBL 귀속 단일 원칙** (§6.11): 운임 집계는 B/L 출력이 아닌 BMS 정산용이며, 출력 양식의 Charge(Print Only) 영역과는 별개 관심사.
- **원칙의 제한**: 본 원칙은 **B/L 도메인의 경계 설정 기준**이지, 업무 전체의 기능 범위를 줄이라는 의미가 아니다. 필요 시 다른 모듈(BMS·PFM·CMS·EDI)이 해당 영역을 보완한다.
- **신규 화면·필드 설계 시 체크**: "이 필드가 인쇄 양식에 들어가는가?" 를 첫 질문으로 삼는다. 인쇄 양식 범위 밖인데 필요한 데이터라면 (i) 어느 모듈에 귀속시킬지, (ii) 애드온 성격인지, (iii) 외부 관리 영역인지를 명시적으로 결정한다.
