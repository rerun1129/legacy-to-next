# 기술 스택

## 1. 선정 원칙

- 안정적인 LTS 버전을 우선하되, 시범 프로젝트 성격에 맞춰 모던 스택을 적극 채택한다.
- 현 단계는 테스트 베드이나 **결과물이 향후 신규 고객사 프로덕션에 투입될 것을 전제**로 품질을 설계한다 (OVERVIEW §1.1). 기술 도입은 이 전제를 만족하는 범위 안에서 이루어진다.
- **Java 스택으로 우선 구현**하여 기능을 완성한 뒤, **동일 기획을 Python 스택으로 재구현**한다. 병렬 동시 개발이 아닌 순차 구현이며, Python 완료 시점에 성능·생산성·운영 비용 비교 데이터를 확보한다.
- AI 기반 개발 도구로 개발 사이클 리드타임을 단축한다.
- 고객 입장에서의 빠른 응답성과 안정적 서비스 경험을 최우선한다.

## 2. 듀얼 백엔드 원칙

### 2.1 구현 순서

- **1단계: Java 스택 우선 구현.** FMS 모듈의 전체 기능(화면·API·도메인)을 Java 스택으로 완성한다. 이 단계에서 도메인 규칙·API 계약·스키마가 확정된다.
- **2단계: Python 스택 재구현.** Java 구현이 검증된 이후, 동일 기획을 Python 스택으로 재구현한다. API 계약·DB 스키마·인증 정책은 1단계 산출물을 그대로 따른다 (§2.2).
- **Go/No-Go 판단 시점**은 Python 스택 재구현 완료 이후다 (OVERVIEW §6.2). 두 스택의 성능·생산성·운영 비용 비교 결과를 근거로 정식 채택 여부를 결정한다.
- 모듈/도메인별 분담 문서는 Python 재구현 시작 시점에 별도 문서로 작성한다. 1단계에서는 Java 단일 스택이므로 분담 이슈가 없다.

### 2.2 양 스택 공통 표준

Python 재구현 시 API 계약 호환성과 공통 운영 체계를 보장하기 위해 다음 항목은 양쪽이 동일하게 따른다. 표준 자체는 1단계(Java)에서 확정되며, 2단계(Python)는 이를 준수한다.

- **API 스키마**: OpenAPI 3.x (Swagger UI 노출). Java는 `springdoc-openapi`, Python은 FastAPI 내장 기능.
- **인증 토큰**: JWT 공통 발급/검증 키 정책. Refresh 정책·Claim 구조도 통일.
- **구조화 로깅**: JSON 포맷, 공통 필드(`trace_id`, `span_id`, `service`, `level`, `event`).
- **분산 트레이싱**: OpenTelemetry. 양 스택의 trace context가 끊김 없이 연결되도록 설정.
- **메트릭**: Prometheus exposition format 통일.

## 5. 데이터 계층

- **PostgreSQL 17.x** — FMS 모듈의 메인 DB (OLTP / SSOT). 모든 원장 데이터·업무 CRUD·FMS 운영 화면(Entry/List)을 담당. FTS + Trigram 활용. 안정적 트랜잭션 + 빠른 쿼리.
- **MongoDB 8.x** — Analytics / Query Mart. 주 용도는 (a) **감사 목적 조회 화면**, (b) **BMS 고객사 특화 조회 화면 · 대시보드 집계 위젯 · PMS 실적 리포트(Phase 2)**(Weekly Volume 등 TEU 기준 집계형)의 Query Mart. 문서형 구조로 사전 결합된 Fact 다큐먼트를 보관하여 다차원 집계 쿼리를 담당하며, PostgreSQL(OLTP)과는 이벤트 기반·최종 일관성으로 동기화된다 (§5.1 참조), (c) **문서 출력 Print date 로그 다큐먼트** (B/L·HAWB·MAWB·A/N·D/O·S/A·S/R·Non B/L 원본 B/L 출력 시점 기록. RDB와 FK 관계 없이 독립 저장. 02_SCREENS 부록 §C-4 참조).
- **PostgreSQL FTS + Trigram** — 전역 검색 엔진
- **Redis 7.x** — **FMS 모듈 화면 구현 시점에서는 필수 의존이 아니다.** 본격 사용은 후속이며, 주 용도는 (a) 유저 세션 정보(인증 정보 제외), (b) 권한 기반 메뉴 노출 캐싱, (c) 임시 저장·자동 저장 데이터의 임시 저장소. TTL / Eviction / 무효화 정책은 추후 확정한다. (인증·권한·세션은 ADMIN 모듈이 담당 — `docs/admin/03_ACCESS_CONTROL`.)
- **AWS S3** — 파일 스토리지 (EDMS 첨부 문서 저장소 포함)

> **스키마 관리 정책**: 본 프로젝트는 **신규 스키마로 시작**하며, 레거시 DB 스키마의 직접 마이그레이션은 범위 밖이다 (OVERVIEW §3.3). 스키마 버전 관리 도구는 **Flyway 채택 (양 스택 통일)**. Java 스택은 Spring Boot 내장 통합을 통해 사용하며, Python 스택은 동일한 Flyway 마이그레이션 파일을 CLI 또는 컨테이너로 실행한다. **Flyway가 스키마 소유권을 갖고 Python 스택은 애플리케이션 코드만 재구현**하는 구조로 운영하여, 같은 PostgreSQL 인스턴스에 두 도구(Flyway·Alembic)가 동시 접근하는 충돌을 원천 차단한다. 마이그레이션 파일 위치·네이밍 규약(`V{version}__{description}.sql`)과 rollback 전략은 Java 첫 엔티티 커밋 시점에 구체화한다. 백업·복구 정책은 개발 진행 과정에서 결정한다.
>
> **현재 상태**: Flyway 의존성은 `build.gradle`에 추가됨. `application.yml: spring.flyway.enabled=false`로 비활성 운용 중. DDL 파일은 `schema/V1__fms_initial_schema.sql`, `schema/V_add_house_bl_non_bl_schedule_fields.sql` 2개로 프로젝트 루트 `schema/` 폴더에 위치. `src/main/resources/db/migration/`으로 이전 및 `enabled=true` 전환은 운영 배포 전 마일스톤으로 예정.

### 5.1 OLTP ↔ Analytics 저장소 분리 원칙

PostgreSQL(OLTP)과 MongoDB(Analytics Mart)는 역할을 **명시적으로 분리**한다. 실적·분석 쿼리는 OLTP를 우회하여 MongoDB Fact Mart에서 처리함을 원칙으로 하되, **PMS 정산 실적은 Phase 1에서 OLTP 직접 집계로 시작하고 성능 한계 시 Phase 2에서 Mart로 전환**한다 (01_DOMAIN §6.11 단계적 예외 참조).

- **조회 경로 분리**
  - **OLTP(PostgreSQL)**: 모든 업무 CRUD, FMS 운영 화면(S-02~S-09 Entry/List), HBL List 인라인 집계(Master 하위 House 건수·요약 등 단건·소규모 집계).
  - **Analytics Mart(MongoDB)**: BMS 고객사 특화 조회 화면, 대시보드 집계 위젯(Weekly Volume 등), 감사 조회 화면, PMS 실적 리포트(Phase 2). **FMS 물동량 실적 등 다차원 집계 쿼리를 OLTP에 직접 수행하지 않는다**(PMS 정산 실적의 Phase 1 OLTP 집계는 §6.11 단계적 예외).
- **왜 분리하는가**: 레거시 실적 리포트 병목(30~120초)의 주 원인은 `HBL × Container × Freight` 같은 N:M:L 경로의 카티시안 폭발이었다. 신규 설계에서는 (i) E-14b 교차 엔티티의 사전 계산 측정값, (ii) HBL 요약 비정규화 컬럼, (iii) Fact Mart 사전 결합 다큐먼트의 3중 구조로 해소한다 (01_DOMAIN §6.3·§6.11, NFR §3.2).
- **동기화 정책**
  - **이벤트 기반 최종 일관성**을 채택한다. 수 분 단위 지연은 업무상 허용된다. 강한 일관성(동기 갱신·2PC 등)은 도입하지 않는다.
  - 구현 패턴(Outbox 패턴 / 메시지 브로커(Kafka·SQS·RabbitMQ) / Spring Batch 주기 워커 / CDC(Debezium))은 **PMS Phase 2(Mart 전환) 라운드**에 최종 확정한다.
  - Fact 다큐먼트 스키마(HBL 단위 단일 다큐먼트 / (HBL × 월) 스냅샷 / 이벤트 로그 + 뷰 중 택일)·파티셔닝 전략(월별 컬렉션 등)·불일치 복구 프로세스는 **PMS Phase 2(Mart 전환) 라운드**에서 확정한다 (`06_OPEN_ITEMS.md` 재개 시점 맵 참조).
- **연관 원칙**
  - **Freight = HBL 귀속 단일 원칙**: 운임은 HBL 레벨에만 붙는다. Container 레벨 Freight는 도입하지 않는다 (레거시·고객사 관행 부재 확인, BMS 모델 기본 전제).
  - **TEU 환산 방식**: **산식 기반** — `TEU = length_feet / 20` (20ft=1.0 / 40ft=2.0 / 45ft=2.25). **별도 factor 코드 테이블을 두지 않는다.** E-14a Container에 `length_feet` 속성을 직접 보관하며 Container Type 코드(GP/HQ/RF/OT/Tank 등)는 규격·용도 구분 표시용으로만 분리 (DOMAIN §1.7·§2.2 E-14a 참조).

- **TDD / DDD**: 본 프로젝트는 TDD · DDD 아키텍처를 기반으로 개발한다 (OVERVIEW §1.2 참조). 레이어 분리·도메인 모델 중심·테스트 우선 원칙을 따른다.
- **AI 기반 개발 도구**: 개발은 AI 코딩 도구(Claude Code 등)를 활용해 진행한다. 구현 단계의 디테일(구체 테스트 케이스, 디렉터리 구조, 빌드 시점 사전 테스트 코드 등)은 개발 착수 시점에 AI 도구와의 협업 과정에서 확정한다.
- **빌드 사전 검증**: CI 파이프라인에서 강제할 테스트 커버리지·린트·타입 체크 규칙은 개발 착수 시 별도 합의하여 적용한다.

**헥사고널 아키텍처 적용 현황**: `domain/` · `application/` · `adapter/in/` · `adapter/out/` 4계층 구조 기반. **멀티모듈** — FMS(`java-spring`)와 ADMIN(`java-spring-admin`)이 별도 Spring Boot 모듈이며 같은 PostgreSQL 인스턴스를 공유한다(ADMIN이 `admin` 스키마 소유). FMS 도메인은 **9개로 분리**됐다: Entry(house-bl·master-bl·switch-bl·truck-bl·non-bl) + List(sea-house·air-house·sea-master·air-master). 초기 PRD의 "3개(house/master/switch)" 및 "non-bl·truck 내부 엔티티" 구상에서 확장·승격된 상태(도메인 모델 E-20·E-24 정의는 `01_DOMAIN` 유지). 프론트엔드도 동일 분리 + `adapter/out/{api,mock}/` 적용. **CodeNameResolver**: FMS가 ADMIN 마스터(거래처·항만·운송인·담당자·HS코드·팀)를 cross-schema 읽기로 조인해 코드→이름을 표시하며, cross-schema 접근은 `adapter/out/persistence/codename/`에 격리된다. 권한·인증 SSOT는 `docs/admin/03_ACCESS_CONTROL`.
