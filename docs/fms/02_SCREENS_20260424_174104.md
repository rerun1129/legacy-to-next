## 3. 화면별 상세

### S-01: 대시보드 *(공통/플랫폼)*

- **분류**: 모듈 독립 공통 화면. FMS 전용이 아니며, BMS/PFM/CMS/MDM 등 타 모듈과 공유된다.
- **구조**: **정적 레이아웃 기반 + 일부 위젯만 동적 동작**. 페이지 자체는 경량이며, 개별 위젯이 자체 데이터 로딩·폴링을 담당한다.
- **주 사용자**: 전체 사용자. 표시되는 위젯은 `{소속 모듈, 운송 모드, 방향}` 등 권한 컨텍스트에 따라 가변.
- **핵심 기능**
  1. **정적 영역**: 헤더·사이드바·인사 메시지·빠른 액션 버튼
  2. **KPI 스트립**: Active Shipments, ETD this week, Docs Pending, Invoiced (MTD), On-time Rate 등 (실제 KPI는 운영 데이터 기반 확정)
  3. **Shipment Pipeline 위젯**: 단계별(Booked → Docs → Gate-in → Loaded → Sailed) 건수 집계. 단계 정의는 도메인 워크플로우(DOMAIN §4.1 해상 수출 9단계)와 별도로 화면 노출용 단순화 단계이며, 양자 매핑은 구현 단계에서 정의한다.
  4. **Weekly Volume 위젯**: TEU 기준 적재 유형별(FCL/LCL/BULK) 일별 집계
  5. **Cargo Trace 위젯**: B/L · Container No · Booking No 입력 → 단계별 진행 상태 조회 (검색 기능 포함)
  6. **Upcoming ETD/ETA Timeline**: 향후 N일 이내 출항/도착 일정 타임라인
  7. **My Tasks**: 우선순위(High/Mid/Low) + 마감 임박/지연 표시. 별도 Task 시스템 필요
  8. **MJ Notice (공지사항)**: 카테고리 분류(Urgent/Update/Notice/Event), 핀 고정, 검색
  9. **Public Holidays**: 국가별 공휴일 캘린더 (외부 API 우선, `04_INTEGRATION.md` 참조)
  10. **Exchange Rate**: USD 기준 다국가 환율 (외부 API, `04_INTEGRATION.md` 참조)
  11. **Recent Activity**: 팀 활동 피드 (감사 로그와 별개로 사용자에게 노출되는 활동 스트림)
  12. 위젯 단위 Redis 캐싱·TTL 적용으로 초단위 응답
  13. 위젯은 권한·담당 범위 기반 조건부 렌더링. 타 모듈 위젯은 해당 모듈 개발 시점에 점진 추가
- **레거시의 문제점**: 통합 진입점이 없어 담당자가 조회 화면을 개별로 열어 상태 확인. 간이 요약 페이지가 있어도 쿼리 즉시 집계 방식이라 체감 속도가 느림.
- **신규 시스템 개선점**: 정적 셸 + 동적 위젯 분리로 TTFB 최소화. 위젯은 독립 컴포넌트로 조립되어 모듈 확장에 무중단 추가 가능.
- **주요 데이터 / 연관 엔티티**: 사용자 권한 정보, 각 모듈의 집계 API 응답(위젯별 상이)

### S-02: House B/L Entry

> **Variant ID 체계** — 본 화면은 라우팅·사이드바 메뉴 단위로는 `{모드, 방향}` 4변형으로 **분리**된다: `S-02[SEA/EXP]`, `S-02[SEA/IMP]`, `S-02[AIR/EXP]`, `S-02[AIR/IMP]`. 권한(`{모드, 방향}`, DOMAIN §6.7) 매핑과 URL 경로가 1:1로 일치한다.
> **내부 구현 레이어**는 단일 코드베이스 공통 Shell + 변형별 config 조립으로 처리한다 (부록 §B *변형 조립 규칙*). 따라서 "UI/라우팅 단위 분리" 와 "구현 단위 조립"은 의도적으로 비대칭이다.
> **UI 규약** — 화면 프레임·탭·상태 뱃지·상단 액션·필드 패턴·데이터 그리드 등 UI 표현과 공통 컴포넌트는 **부록 §V1 준수**. 본 섹션은 필드·블록 구성과 변형 Delta만 명세한다.

- **주 사용자**: FMS 업무 담당자. `{모드, 방향}` 권한 조합으로 각 업무 담당자별 접근 제어.
- **핵심 기능**
  1. **탭 분리 입력**: Main / EDI / Other / Freight (4탭 공통). 변형에 따라 각 탭 내 블록 구성이 가변 (부록 §B 변형 조립 규칙).
  2. **Master B/L 연결**. 입력 순서는 House → Master 가 기본이며, Master → House 순서도 지원된다. 양자 간 값 복사·제안(suggest) 규칙은 추후 결정. **Master와 House의 당사자·경로·일정 값은 독립 관리되며 자동으로 동기화되지 않는다** (DOMAIN §E-08 참조).
  3. 모드별 확장 필드 자동 전환 (해상: 선박·항차·VGM / 항공: Flight·Rate Class·보안검색)
  4. **위험물 최대 5종 입력** (DOMAIN §6.2). 입력 방식은 그리드가 아닌 5칸 텍스트 필드 (IMDG Code × 5 + UN No × 5).
  5. **면장 입력** (E-17) — 수출 변형(해상/항공 수출 House)에서 **EDI 탭 내 License 섹션**으로 제공. 다중 면장(분할 면장)·동시포장·Partial Shipment 지원. 수입 방향 면장은 포워더 업무 범위가 아니므로 본 시스템 대상 아님(수입자가 직접 처리. 포워더가 대행하는 극히 드문 예외 케이스는 제외). 파트 혼재 입력은 후속 기획에서 결정.
  6. **Switch B/L 분기 발행** — 별도 모달로 제공 (하단 "Switch B/L 모달" 영역 참조). 원본 House 보존·재Switch 차단 규칙은 DOMAIN §6.6 유지.
  7. **문서 출력**: 헤더 "출력" 버튼으로 공용 출력 옵션 팝업(부록 §C). 노출되는 출력 종류는 `{모드, 방향}`에 따라 결정 (부록 §C-2). 해상 수출은 B/L·S/A·S/R, 해상 수입은 A/N·D/O, 항공 수출은 HAWB·S/A·S/R, 항공 수입은 출력 대상 없음. 수입 B/L·AWB는 해외 파트너 발행분이라 본 시스템 출력 대상이 아니다. 미저장 변경분은 저장 확인 후 출력. Print date 타임스탬프는 MongoDB 로그 다큐먼트에 기록.

#### Main 탭 구성

> **기술 방식**: 아래 "공통 구성" 블록은 4변형 모두에 적용되는 모드 중립적 기술이고, 이어지는 "변형별 고유 구성" 4개 블록은 각 Variant에서 추가·치환되는 요소를 명시한다. 변형별 블록에 기술되지 않은 항목은 공통 구성을 그대로 따른다.
>
> **섹션 배치 순서**: 신규 시스템에서는 Main 탭의 섹션 배치 순서를 4변형 전체에서 통일한다. 레거시 스크린(특히 항공 수입)에서 Dimension/Cargo가 상단에 노출되는 등의 배치 차이는 UI 윤색 특성이며, 신규 설계에서는 Party → Schedule → Trade/Document/Performance → (모드별 확장 영역) → Marks·설명·Item → (항공 전용 보조 영역: Handling Information · Charge(Print Only) · Other Charge) 순을 기본으로 한다. 세부 레이아웃은 디자인 기획 라운드에서 확정.

##### Main 탭 공통 구성 (4변형 공통)

- **Party**: 4슬롯 — Shipper / Consignee / Notify / DOC Partner. 각 슬롯은 **§V1.9 Party 카드** 규약 준수
  - Consignee 보조 버튼: **To Order**
  - Notify 보조 버튼: **Same as Consignee만 채택**. Same as Shipper 옵션은 미채택
  - 항공 수입 변형에서는 Consignee가 필수 필드로 전환(아래 `S-02[AIR/IMP]` 변형 참조)
- **Performance**: Actual Customer / Customer PIC / Settle Partner / Sales Class / Sales Man / Operator / Team. Operator · Team · Sales Man · Customer PIC는 DOMAIN §E-08 영업·실적 속성이며 권한 모델이 아니다.
- **Marks & Numbers**: 2단 분할 텍스트 영역(좌/우 병기)
- **문서 설명 영역**: 2단 분할 텍스트 영역(좌/우 병기). **영역 명칭은 모드별 차이** — 해상은 `Description`, 항공은 `Nature & Quantity of Goods`. 해상 변형에서는 본 영역 상단에 **부지약관 구문 선택 드롭다운 2개**가 배치되며, 선택 시 B/L·S/A·S/R 등 출력 문서의 해당란 상단에 삽입된다. 항공 변형에서는 드롭다운 없음. 드롭다운 옵션은 **공통 코드로 관리 예정** (공통 코드 관리 방식은 후속 결정).
- **Item 블록**: Main Item + HS Code + Remark. 모든 변형에서 Main 탭에 포함되며, 구체 배치 위치는 디자인 기획 시 확정.
- **Say Information + Apply Say**: B/L·HAWB 인쇄용 영문 수량 표기 자동 생성 (예: "TWENTY FOUR THOUSAND EIGHT HUNDRED KILOGRAMS ONLY"). 출력 옵션 팝업(부록 §C)으로 인쇄할 때 사용.
- **공통 동작**: 등록자/수정자 정보(Modified, Modified Date)는 **토글로 숨김 표시**한다. 토글 컴포넌트 자체는 사용자 눈에 띄게 배치한다.

##### `S-02[SEA/EXP]` 해상 수출 변형 고유 구성

- **Schedule**: Liner, Vessel/Voyage, ETD/ETA/Onboard, POR/POL/POD/Delivery/Final Dest.
- **Issue Information** (3필드): **Issue Date / No. of B/L / Issue Place** — 해상 수출 출력물인 B/L 발행 속성. 출력 팝업 옵션이 아닌 Entry 필드(부록 §C-3 원칙과 정합).
- **Trade**: Incoterms / Freight Term / Payable At / Triangle
- **Document**: Co-load Type / Switch (Normal/Switch) / Co-Load Agent
  - Sub Master No. 필드·기능은 제외 (06_OPEN_ITEMS §1.2 결정 반영)
- **Container 영역**
  - 컨테이너 그리드 + 좌측에 Type/Qty 집계 표시 + 주변에 "No. of Containers or Packages" 입력 필드
  - Seal No.는 기본 1~2번 컬럼만 노출, "Seal ≫" 버튼으로 3~6번 컬럼 expand, "Seal ≪"로 collapse (DOMAIN §6.3 최대 6개)
  - 행 정렬은 Drag & Drop (도메인 속성으로 Seq 컬럼을 두지 않고, 단순 편의 기능으로 제공)
  - 컨테이너 아이템 분해(E-15) UI는 후속 기획에서 제공
- **문서 설명 영역 명칭**: `Description`. 상단에 부지약관 구문 선택 드롭다운 2개 (공통 구성 항목 참조).

##### `S-02[SEA/IMP]` 해상 수입 변형 고유 구성

- **Schedule / Trade / Document / Container / 설명 영역 명칭**: `S-02[SEA/EXP]` 와 동일
- **Issue Information**: **`D/O Date` 1필드만 존재**. 수입은 출력물이 A/N·D/O이므로 B/L 발행 속성(Issue Date / No. of B/L / Issue Place)이 불필요하고, 대신 D/O 발행일만 Entry에서 관리한다(부록 §C-5 "Entry 수동 입력" 원칙과 정합).

##### `S-02[AIR/EXP]` 항공 수출 변형 고유 구성

- **Schedule 블록** (해상과 구조가 다름, 항공 고유)
  - **상단 헤더 (사용자 입력 필수)**: `*Airline`(항공사 코드+명) / `*Departure`(출발지 공항 코드+명)
  - **중단 Schedule Leg 그리드** (E-19 항공 전용 엔티티와 1:1 매핑, DOMAIN §2.1 E-19·§3.2·§6.12)
    - 컬럼: `*To` / `By` / `Flight` / `*On Board` / `Time`(On Board) / `*Arrival` / `Time`(Arrival). 별표 3개 필수
    - 행 단위 추가/삭제/순서 이동 컨트롤(+/−/↑/↓). 직항은 1행, N회 환승은 (N+1)행 구조
    - 최소 1행 필수 입력 — 그리드가 비어 있으면 하단 파생 필드(ETD·ETA·Destination)가 비어 있게 되므로 저장 시점 검증
  - **하단 요약 영역 (회색 읽기 전용, 그리드에서 자동 파생)**:
    - `*Destination` = 그리드 **마지막 행의 `*To`** → Main 탭 POD에 바인딩
    - `On board` = 그리드 **첫 행의 `*On Board`** → Main 탭 ETD에 바인딩
    - `Arrival` = 그리드 **마지막 행의 `*Arrival`** → Main 탭 ETA에 바인딩
  - **행 판정 기준**: 그리드의 **표시 순서 그대로** 첫 행/마지막 행을 판정한다(업무 애플리케이션 전제상 악의적 사용은 고려하지 않음). 사용자가 Drag&Drop으로 행 순서를 재정렬하면 파생값도 즉시 재계산된다
  - **하단 요약 영역은 입력 불가** — 읽기 전용 표시. POL은 Schedule 헤더의 `*Departure` 가 원천이므로 Main 탭의 POL 필드도 Schedule 블록 내에서 간접 관리되는 셈
- **Issue Information** (3필드): **Issue Date / Issue Place / Signature**. 해상 수출의 `No. of B/L` 이 `Signature` 로 치환된 형태. HAWB 양식의 서명란에 바인딩.
- **Trade** (해상보다 필드 수 많음): Currency(예: HKD) / Incoterms / Freight Term / **Other Term** / D.V Carriage(기본값 `N.V.D.`) / Insurance(기본값 `NIL`) / D.V Customs(드롭다운 + 자유 텍스트 2필드, 예: `AS PER INV.`) / Warehouse / Account Information(예: `FREIGHT PREPAID`)
  - `Other Term`은 해상의 `Payable At·Triangle` 과 기능적으로 다른 항공 고유 필드
  - `D.V Carriage`, `Insurance`, `D.V Customs` 는 HAWB 양식의 해당 란(Declared Value for Carriage · Amount of Insurance · Declared Value for Customs)에 바인딩
- **Document**: **Triangle(체크박스)** / **Switch(Normal 라벨)** / Co-load Type / **Cargo Type** / Co-Load Agent
  - `Cargo Type` 은 항공 고유 필드(화물기/혼합기/여객기 카고 구분 등)
  - 해상의 Switch 와 달리 항공 Switch 는 Normal 라벨로만 표시됨 (운영 의미는 동일하되 UI 차이)
- **Dimension 그리드 (항공 전용 영역)**
  - 컬럼: Length / Width / Height / Qty / CBM / Volume Wt.
  - **`CM / 6000` 제수 선택자** — IATA 표준 볼륨 중량 계산 제수. 드롭다운으로 `CM/6000`, `INCH/366` 등 선택 가능 (구체 선택지는 디자인 기획 시 확정)
  - 행 단위 추가/삭제/순서 이동 컨트롤(+/−/↑/↓)
  - 하단 합계 라인: Qty · CBM · Volume Wt.
- **Cargo 블록 (항공 전용, 해상의 Container와 위상 등가)**
  - Package/Unit (예: `CT / CARTON`)
  - Gross W/T
  - **Charge W/T (청구중량)** — `max(Gross W/T, Volume W/T)` 개념. Apply Weight 버튼과 연동.
  - **Rate Class** — IATA 운임 분류(General · Specific Commodity Rate · Class Rate 등)
  - **Volume W/T** — Dimension 그리드 합계로부터 산출
  - KG/LBS 선택자
  - CBM
  - **Apply Weight** 버튼 (Dimension → Cargo 값 계산/반영)
- **Charge (Print Only) 영역 (항공 전용, Main 탭 하단 하위 영역)**
  - 그리드 컬럼: `No / Freight / Freight Name / Currency / Per / P/C / Gross W/T / Rate Class / Charge W/T / Rate / Total Amount / AS`
  - **용도 구분**: 본 영역은 **HAWB 인쇄 양식에 박히는 Charge 섹션**을 위한 입력 영역이며, **BMS 모듈의 Freight 탭(Raw 운임 데이터)과 별개**이다. Freight 탭은 BMS 정산용, Charge(Print Only)는 HAWB 출력 문서의 운임 표기용.
  - 행 단위 추가/삭제 컨트롤. 하단 합계 라인: Gross W/T · Charge W/T · Rate · Total Amount
- **Other Charge 영역 (항공 전용)**
  - 자유 텍스트 + **`Other Apply`** 버튼 — HAWB 인쇄 양식의 Other Charges란에 바인딩
- **Handling Information (항공 전용)**
  - 드롭다운 + 자유 텍스트 영역. HAWB 출력 양식의 Handling Information란에 바인딩.
- **문서 설명 영역 명칭**: `Nature & Quantity of Goods` (해상의 `Description` 과 명칭 차이)
- **Container 영역 없음** (항공은 컨테이너 개념 부재)

##### `S-02[AIR/IMP]` 항공 수입 변형 고유 구성

- **Schedule / Document / Dimension / Cargo / Charge(Print Only) / Other Charge / Handling Information / 설명 영역 명칭**: `S-02[AIR/EXP]` 와 동일
- **Issue Information 블록 전체 없음** — 해상 수입의 `D/O Date 1필드`와도 달리, 항공 수입은 Issue 관련 필드를 Main 탭에서 관리하지 않는다. 수입 측 D/O 관련 관리는 공용 출력 팝업 대상이 아니므로 Entry 필드가 불필요(수입 항공은 출력 대상 자체가 없음, 부록 §C-2).
- **Trade** (`S-02[AIR/EXP]` 공통 필드 + 항공 수입 고유 필드):
  - 항공 수출과 동일한 Currency / Incoterms / Freight Term / Other Term / D.V Carriage / Insurance / D.V Customs / Warehouse / Account Information
  - **추가: `F.H.D` 라디오** (`Not / F.H.D / To Door`) — 항공 수입 전용 배송 구분 필드. Free House Delivery / Door delivery 여부 관리.

- **House B/L No · House AWB No 채번 정책**: 수출 변형(`SEA/EXP`·`AIR/EXP`)은 미입력 시 저장 시점에 시스템이 자동 채번(§V1.5 `Auto on save` placeholder 규약). 수입 변형(`SEA/IMP`·`AIR/IMP`)은 해외 파트너 발행 번호이므로 **사용자 필수 입력**.
- **ETD / ETA / POL / POD ※ 표시 (항공 변형)**: 항공 변형의 ETD·ETA·POL·POD 4개 필드는 **Main 탭에서 직접 입력하지 않는다**. Schedule 블록 내에서 간접 관리되며 바인딩은 아래와 같다:
  - **POL** ← Schedule 헤더의 `*Departure` (사용자 직접 입력)
  - **ETD** ← Schedule Leg 그리드 **첫 행의 `*On Board`** (파생, 읽기 전용)
  - **POD** ← Schedule Leg 그리드 **마지막 행의 `*To`** (파생, 읽기 전용)
  - **ETA** ← Schedule Leg 그리드 **마지막 행의 `*Arrival`** (파생, 읽기 전용)
  - 따라서 항공 변형에서 실질적 필수는 "**Schedule 헤더 2필드(`*Airline`·`*Departure`) + Schedule Leg 그리드 최소 1행 + 각 행의 별표 3개(`*To`·`*On Board`·`*Arrival`)**" 로 치환된다. DOMAIN §3.2 Schedule Leg 엔티티(E-07·E-19) 및 §6.12 B/L 출력 연동 원칙과 정합.
- **Schedule Leg 그리드 = E-19 (항공 전용)**: 본 그리드의 각 행이 E-19 레코드와 1:1 매핑. 해상 변형에는 E-19가 붙지 않으므로 해상의 Schedule 블록은 Vessel/Voyage 단일 축으로 관리한다.
- **검증 규칙**:
  - UI 필수 표시는 §11 `Required style` 설정(`bar` / `dot` / `fill` / `subtle`)으로 렌더링
  - 프론트 Zod 스키마 + 백엔드 이중 검증 (부록 §A 일반 원칙)
  - 조건부 필수(예: `Cargo Type = Danger`면 UN No 필수 등)는 별도 기획 턴에서 명세
- **경유지(Via)**: 항공은 Schedule Leg 그리드 자체가 경유지 관리(E-07·E-19)이므로 별도 Via 섹션·필드 없음. **해상 / Trucking / Non B/L은 경유지 관리 안 함** — 해상도 실무상 경유지 개념은 존재하나 종이 B/L에 인쇄하지 않으므로 본 시스템 관리 대상 아님 (DOMAIN §6.12).

#### EDI 탭 영역 구성 (해상/항공 공통)

- EDI Identifiers: EDI B/L No, EDI Item, Cargo Class, T/S
- **위험물 관리**: IMDG Code 5칸 + UN No 5칸 (디자인 초안의 명칭 "AFR"은 채택하지 않음, "AFR Send" 필드 제거)
- CCAM 매트릭스(Shipper/Consignee/Notify별 PIC·Tel·Country·Biz No·Email)는 채택하지 않음 (디자인에서도 제거 예정)
- **License 섹션 (Korea Only)** — E-17 House B/L 면장 입력 영역. **해상/항공 수출 House 변형에서만 노출**. 한국 세관(관세청 UNIPASS) 체계 기반이므로 한국 외 사용자 환경에서는 섹션 전체 숨김 (사용자 국적/소속 지점 기준 노출 조건의 구체 룰은 추후 `{모드, 방향}` 권한 변형과 함께 결정). 기존 부록 §B의 "E-17 방향 전용 탭" 방침은 본 섹션으로 대체.
  - **다중 행 그리드**. 행 단위 추가/삭제/순서 이동(Row Count 컨트롤). 다중 면장 = 분할 면장 케이스를 그리드 구조 자체로 수용.
  - **컬럼 구성**:
    - `No` / 선택 체크박스
    - **`수출신고필증 번호 / 화물관리번호`** — 단일 컬럼. 행마다 둘 중 하나만 입력(배타). 값 형식으로 구분(필드 자체는 하나).
    - `수량` / `단위` / `중량` — 면장 기재 물량
    - **`동시 포장`** 그룹: 기호 / 수량 / 단위 — 동일 포장 단위에 여러 면장이 묶이는 케이스 표기
    - **`분할 선적`** 그룹: 여부 / 차수 — **Partial Shipment** 개념(전체 계약 물량을 여러 차례 나눠 선적). 세관 분할 면장과 구분.
    - `진행 내역` — 행별 **UNIPASS 외부 링크** 컬럼(편의 기능). 행 상태 조회용으로 세관 시스템으로 딥링크. 링크 생성·인증 방식 등 세부 동작은 추후 결정.
  - **합계 행(하단)**: 수량/중량 집계는 **단위별 그룹 합계**로 표시 (예: `KG 24,800.000 / MT 1.200` 병기)
- **수출입 신고 적하목록 EDI 영역**(MRN/MSN 트리거·전송 이력 등)은 본 License 섹션과 별개이며 후속 기획에서 제공

#### Other 탭 (해상/항공 공통)

- Reference Numbers / Co-load B/L (현 디자인 구성 유지)

#### Freight 탭 (BMS 영역, 해상/항공 공통)

- 본 화면은 BMS 모듈에 필요한 운임 Raw 데이터를 한 화면에서 입력하기 위한 UX 통합용
- **현 단계는 화면 셸(컴포넌트 구성)만 제공하며, 데이터 바인딩·실제 동작은 BMS 모듈 기획 시 확정**
- 포함 영역(셸 단계): Actual Customer / Liner / Settle Partner Rate Header, Selling/Debit, Buying/Credit, Account Documents, Remark
- Freight 그리드 컬럼은 현재 노출된 부분만 구성. 세부 컬럼·입력 동작은 BMS 기획 단계에서 결정
- **항공 Main 탭의 `Charge (Print Only)` 영역과의 구분**: Freight 탭은 BMS 정산용 Raw 운임 데이터이며, Charge(Print Only)는 HAWB 출력 문서의 운임 표기용이다. 두 영역은 바인딩 대상과 소비자(BMS 모듈 vs 출력 템플릿)가 다르므로 혼동하지 않는다.

#### Switch B/L 모달 (S-02에서 트리거하는 별도 모달)

- **목적·제약**: 원본 House B/L 1건당 **Switch B/L 최대 1건** (DOMAIN §6.6). 원본 House는 보존하고 교체분만 E-21·E-22에 저장. **재Switch 불가.**
- **타이틀**: `SEA Switch B/L Management` / `AIR Switch B/L Management` (모드별)
- **상단 액션 툴바**: `Search` / `Reset` / `Save` / `Delete`
  - `Search` — 원본 House B/L을 조회하여 Switch 입력 영역에 재로드하는 편의 기능. 수정 중 실수했을 때 원본 House 기준으로 되돌리는 용도. 1:1 제약은 그대로 유지(다른 원본 House의 Switch로 전환하는 용도가 아님).
  - `Reset` — 현재 Switch 입력 내용 초기화. (디자인 초안의 `New` 라벨은 오용이며, **Reset으로 확정**.)
  - `Save` / `Delete` — 저장 / 삭제. `Delete`는 Switch B/L만 삭제하고 원본 House에는 영향을 주지 않는다 (§6.6).
  - **`Print` 버튼 없음.** Switch B/L은 본 시스템의 문서 출력 대상이 아니다 (출력 매트릭스는 부록 §C-1 기준).
- **헤더 영역** (E-21 Switch B/L 고유 속성)
  - `Switch B/L No` — 편집 가능
  - `House B/L No.` — 원본 House 참조 표시, 읽기 전용
  - `Incoterms`
  - `*B/L Type` — 필수, **해상 변형에만 노출. 항공 변형에서는 필드 자체 제거** (AWB는 B/L Type 구분 없음)
- **Party 영역** (E-21 교체 적용 당사자 3슬롯)
  - `*Shipper` (필수) / `Consignee` / `Notify`. 각 슬롯은 **§V1.9 Party 카드** 규약 준수
  - Notify에 `Same Cne.`(Same as Consignee) 버튼 제공. **`Same as Shipper` 버튼은 채택하지 않음** (S-02 본 화면과 동일 정책, §1.2).
- **Marks and Numbers / Nature & Quantity of Goods** (E-22 Switch B/L 설명)
  - 각각 2단 분할 텍스트 영역(좌/우). 원본 House의 Mark & Description과 독립 저장.
- **미노출 속성의 처리** — E-21 속성 중 본 모달에 노출되지 않는 것(예: 교체 적용 경로 POL/POD, 기타 부가 속성)은 **원본 House B/L 값을 상속해 사용**한다. 엔티티 스키마(E-21 속성 목록)는 현행 유지하되, 해당 속성은 본 모달을 통해서는 입력·변경할 수 없다. 필요 시 UI 확장은 추후 결정.
- **모드별 변형 요약**
  - 해상: 위 구성 전체
  - 항공: `B/L Type` 필드 1개만 제거, 그 외 완전 동일
- **엔티티 바인딩**: E-21 Switch B/L + E-22 Switch B/L 설명 (현행 스키마 유지)
- **트리거 위치**: S-02 House B/L Entry에서 별도 액션으로 모달 호출. 구체 버튼 위치·라벨은 S-02 본 화면 UI 확정 시 함께 결정.

### S-03: House B/L List

> **Variant ID 체계** — 본 화면은 라우팅·사이드바 메뉴 단위로 `{모드, 방향}` 4변형으로 **분리**된다: `S-03[SEA/EXP]`, `S-03[SEA/IMP]`, `S-03[AIR/EXP]`, `S-03[AIR/IMP]`. 권한 매핑·URL·기본 데이터셋이 변형별로 자연 분리되어, React Query 캐시 키·북마크·권한 미들웨어 체크가 모두 경로 단위로 깔끔해진다.
> **내부 구현 레이어**는 공통 Shell(가상 스크롤·페이지네이션·일괄 선택·Excel export·전역 검색·다중 컬럼 정렬·공용 출력 팝업 트리거) + 변형별 config(컬럼 정의·필터 스키마·기본 정렬·행 인라인 액션·기본 고정 WHERE) 조립 (부록 §B).

- **주 사용자**: FMS 업무 담당자. 권한 조합별 기본 필터.
- **핵심 기능**
  1. 변형별 고정 컨텍스트(예: `{SEA, EXP}`) 는 URL·Shell에서 고정되고, **고급 검색**(선박·항차·선사·송/수하인·ETD/ETA·HBL 번호·참고번호·컨테이너 번호)만 화면 내 필터로 제공
  2. 전역 검색 (PostgreSQL FTS + Trigram 기반 부분 일치·오타 허용)
  3. 대량 선택 후 일괄 액션 (EDI 재전송 등의 외부 연동 액션 + 공용 출력 팝업 경유의 문서 일괄 출력)
  4. Excel/CSV 내보내기, 개인화 컬럼 저장, **다중 컬럼 정렬 지원** (정렬 우선순위를 컬럼 헤더에 숫자 + ↑↓로 표시)
  5. 행 인라인 상태 표시 — 외부 연동 추적 상태(EDI 전송 상태 등)에 한정. 발행·회수·Surrender·발송·컨펌 등 업무 상태는 시스템 관리 범위 밖이므로 표시하지 않는다 (DOMAIN §6.1·§6.10).
  6. 조회 조건의 구체 컬럼·옵션 구성은 별도 후속 정리 예정. 디자인 초안의 "Authority" 필터는 채택하지 않음 (행 권한 모델 미도입에 따름, DOMAIN §6.7 참조).
  7. **문서 출력 (단건·일괄)**: 행 인라인 액션 단건 / 다중 선택 + 상단 출력 버튼 일괄. 공용 출력 옵션 팝업(부록 §C) 사용. 출력 종류는 행의 `{모드, 방향}` 변형 기준 (부록 §C-2). 일괄은 선택 행이 같은 변형일 때만 허용. 기본 결과 형태는 개별 PDF (ZIP). 대량은 비동기 처리 (NFR §1.3).
- **레거시의 문제점**
  - 상위 10개 대량 집계 쿼리가 30~120초 소요 (OVERVIEW §3.1)
  - Ag-Grid 종속으로 커스터마이징·업그레이드 어려움
  - 부분 일치 검색이 `LIKE %...%` 기반이라 성능 저하
- **신규 시스템 개선점**
  - 대량 집계 쿼리 **속도 90% 향상 목표** (OVERVIEW §3.1)
  - TanStack Table + TanStack Virtual로 수만 건 스크롤 렌더
  - FTS + Trigram 인덱스로 검색 속도·품질 동시 개선
- **주요 데이터 / 연관 엔티티**: E-08 House B/L 집계 뷰(요약 비정규화 컬럼 포함 — `total_allocated_teu` 등), E-01 Master B/L, E-14a Container · E-14b House-Container 배정

### S-04: Master B/L Entry

> **Variant ID 체계** — 본 화면은 라우팅·사이드바 메뉴 단위로 `{모드, 방향}` 4변형으로 **분리**된다: `S-04[SEA/EXP]`, `S-04[SEA/IMP]`, `S-04[AIR/EXP]`, `S-04[AIR/IMP]`. **내부 구현**은 S-02와 동일하게 공통 Shell + config 조립 (부록 §B).
> **UI 규약** — UI 표현·공통 컴포넌트는 **부록 §V1 준수**. 본 섹션은 필드·블록·변형 Delta와 Master 고유 구조(하단 액션 스트립 등)만 명세한다.

- **주 사용자**: FMS 업무 담당자. `{모드, 방향}` 권한 조합으로 접근 제어.
- **MVP 개발 순서**: **해상 수출(`S-04[SEA/EXP]`) 화면을 먼저 개발**하고, 이를 기반으로 나머지 3변형을 후속 개발한다. 본 명세는 4변형 전체의 Main 탭 구성 확정본이다.
- **핵심 기능**
  1. 선사·항공사 부킹번호 기반 Master 생성·조회. 입력 순서는 Master → House 가 기본이며, House가 먼저 생성된 경우 기존 House B/L을 Master에 연결하는 플로우도 지원한다 (S-02 참조, 하단 액션 버튼 **House B/L Load** 와 연동).
  2. 기본정보 / 경로·일정 / 경유지 / 치수 / 설명 / EDI 탭 입력
  3. 해상/항공 모드별 확장 필드 (선박·항차·VGM / Flight·Charter·보안검색)
  4. Master 하위 House B/L 목록·물량 집계 인라인 표시
  5. 적하목록 EDI 전송 트리거 및 MRN/MSN 반영
  6. **문서 출력 (수출 변형 한정)**: Master B/L Entry는 **헤더에 인쇄 버튼을 두지 않는다**. 출력은 **하단 전용 액션 버튼**(Shipping Request · Shipping Advice, 항공 수출 변형은 MAWB 추가)으로 트리거하며, 공용 출력 옵션 팝업(부록 §C)을 호출한다. 해상 수출 변형은 S/A·S/R, **항공 수출 변형은 MAWB·S/A·S/R**을 출력한다. MAWB는 IATA 표준 양식 고정. 수입 변형에서는 출력 없음. Print date 타임스탬프는 MongoDB 로그 다큐먼트에 기록.
- **외부 연동 추적**: EDI 전송 이력은 외부 연동 추적(DOMAIN §6.10)으로 화면 내에서 조회 가능. 이는 B/L 발행 이력과는 별개 범주다.

#### Main 탭 구성

> **기술 방식**: S-02와 동일하게 "공통 구성" + "변형별 고유 구성" 4개 블록으로 기술한다. 변형별 블록에 기술되지 않은 항목은 공통 구성을 그대로 따른다.
>
> **섹션 배치 순서**: 신규 시스템에서는 4변형 전체에서 섹션 배치 순서를 통일한다. 레거시 스크린(특히 항공 수입)에서 Dimension/Cargo가 우측 상단에 배치되는 등의 차이는 UI 윤색 특성이며, 신규 설계에서는 헤더 → House B/L List 인라인 그리드 → Party → Schedule → (모드별 확장 영역) → Cargo → Document → Trade/Issue/Other → Container(해상) / Dimension(항공) → Marks·설명·Remark → (항공 전용 Charge 영역) → 하단 액션 스트립 순을 기본으로 한다. 세부 레이아웃은 디자인 기획 라운드에서 확정.

##### Main 탭 공통 구성 (4변형 공통)

- **헤더 기본 속성 블록**
  - 공통 필드: `Master Ref. No.` / `Master B/L No.`(해상) 또는 `Master AWB No.`(항공) / `Shipment Type` / **Status**
  - 해상 추가 필드: `Line Bkg. No.` / `Load Type` / `Service Term` / `B/L Type` (해상 전용 4필드. 항공은 AWB 특성상 B/L Type 개념이 없고, Load Type·Service Term도 적용되지 않음 — 항공 변형에서는 본 4필드 제거)
  - **Status**: B/L 마감 관련 플래그. **MVP 범위 외 — 상세 정의는 후속 기획**
  - **헤더 액션 버튼**: §V1.7 공통 셋 중 **Print 제외** (`Search / New / Copy / Delete / Save`) + 화면별 추가 액션 **Change B/L No**(해상) / **Change AWB No**(항공). **Print 제외 사유** — 출력은 하단 전용 버튼으로 트리거(핵심 기능 6 참조)
- **Party**: **3슬롯** — Shipper / Consignee / Notify
  - **DOC Partner 슬롯 없음** (Master는 서류 파트너와 실제 파트너를 구분할 필요가 없음)
  - 보조 버튼: Shipper **O/B OF**, Consignee **To Order**, Notify **Same as Consignee**
  - Notify에 Same as Shipper 옵션은 미채택 (S-02 정책과 동일)
- **Cargo 블록 (우측 상단 위치, House와 배치 차이)**
  - Main Item + HS 코드 / Main Item 설명 / Package(수량 + 단위) / Gross W/T + KG/LBS / CBM
  - **Apply Weight** 버튼 (존재만 명시, 동작 정의는 후속)
  - 모드별 추가 필드는 변형별 고유 구성에서 정의
- **Say Information + Apply Say**: S-02 공통 구성과 동일
- **Marks and Numbers / 문서 설명 / Remark** (3분할)
  - 문서 설명 영역 명칭은 모드별 차이 — 해상은 `Description`, 항공은 `Nature & Quantity of Goods`
  - 해상 변형의 Description 영역 상단에 **부지약관 구문 선택 드롭다운 2개** (S/R·S/A 출력 문서 Description란 상단에 삽입). 항공 변형에서는 드롭다운 없음.
  - **Mark / Description 재설정** 버튼 (존재만 명시, 동작 정의는 후속)
  - **Remark** 영역
- **공통 동작**: 등록자/수정자 정보(Modified, Modified Date)는 S-02와 동일하게 **토글로 숨김 표시**한다. (레거시 스크린샷에서는 기본 노출 상태이나, 신규 시스템은 토글 방식으로 통일)

##### `S-04[SEA/EXP]` 해상 수출 변형 고유 구성

- **헤더**: 공통 구성의 해상 추가 4필드(Line Bkg. No. / Load Type / Service Term / B/L Type) 포함, 총 8필드
- **House B/L List 인라인 그리드**
  - 컬럼: `No` / `House B/L No.` / `Shipper` / `Consignee` / `DOC Partner` / `Package` / `Unit` / `Gross W/T` / `CBM` / `ETD` / `ETA` / `Vessel` / `Voyage` / `POL`
  - 하단 합계 라인: Package · Gross W/T · CBM
  - 그리드 버튼: **New HBL** / **House Consol** (Master 단위 Consolidation 별도 모달) / 엑셀 내보내기 / 접기·펼치기
- **Schedule**: Liner / Liner Pic / Vessel Code + VSL Country / Vessel/Voyage / ETD / ETA / Onboard / Final ETA / POR / POL / POD / Delivery / Final Dest. + **Schedule 리셋 버튼**
- **Cargo** (공통 Cargo 블록 + 해상 추가 필드): 공통 필드 + **R/Ton**
- **Document**: Settle Partner / Co-Load Agent / 업무담당자 / Team
  - **Switch 필드 없음** (Master에는 해당 없음)
  - **Performance 블록 없음** (House의 Actual Customer / Sales Man 등은 Master에 없음)
- **Trade & Issue Information**: `Freight Term` / `Issue Date` **2필드만**
  - House의 Incoterms / Payable At / Triangle 없음
- **Container 그리드 (읽기 전용 집계 뷰)**: Master 자체에는 컨테이너 엔티티가 없다. 본 그리드는 **Master에 연결된 House B/L들의 컨테이너 레코드(E-14a · E-14b)를 조회·집계 표시**만 하며, 입력·편집은 불가하다. 컬럼 구조는 S-02와 동일 (No, Container No., Type, Seal No. 1~2 + Seal ≫ expand, Qty, Unit, Gross W/T, CBM, SOC/COC, Part, TARE 중량, VGM W/T).
- **문서 설명 영역 명칭**: `Description`. 상단에 부지약관 구문 선택 드롭다운 2개.
- **하단 액션 버튼 스트립** (Master 고유, 7개 버튼)
  - **Profit/Loss**: 손익 연동 (버튼 위치만 확정, 기능은 후속)
  - **House B/L Load**: 기존 House B/L을 Master에 연결 (버튼 위치만 확정, 기능은 후속)
  - **Shipping Request**: S/R 출력 트리거 → **부록 §C 공용 출력 팝업** 호출
  - **Shipping Advice**: S/A 출력 트리거 → **부록 §C 공용 출력 팝업** 호출
  - **M/F Send**: 적하목록(Manifest) EDI 전송 (DOMAIN §6.10 외부 연동)
  - **AFR Send**: AFR(Advance Filing Rules; 미국·일본 등 해외 세관 사전 신고) EDI 전송 (DOMAIN §6.10 외부 연동). S-02 EDI 탭에서 폐기된 "AFR" 명칭(위험물 관리로 개칭, 06_OPEN_ITEMS §2.2)과는 별개 개념.
  - **EDMS**: **AWS S3 연동** 전자문서관리. **MVP 단계에서는 버튼 컴포넌트 외형·위치만 확정**, 상세 기능은 후속 기획에서 결정

##### `S-04[SEA/IMP]` 해상 수입 변형 고유 구성

위 `S-04[SEA/EXP]` 를 공통 베이스로 하되, 아래 2개 항목만 다르다. 그 외 헤더·Party·Schedule·Cargo·Document·Container·Marks/Description/Remark 구성은 수출과 동일.

- **Trade & Issue Information 블록 전체 없음** — 수출의 Freight Term / Issue Date 2필드 자체가 수입에서는 관리 대상 아님
- **하단 액션 버튼 스트립**: **Shipping Request / Shipping Advice 2개 버튼 제거** (수입 변형은 출력 대상 없음, 부록 §C-2). 나머지 5개 버튼(Profit/Loss · House B/L Load · M/F Send · AFR Send · EDMS)은 유지.

##### `S-04[AIR/EXP]` 항공 수출 변형 고유 구성

- **헤더**: 공통 필드 4개(Master AWB No. / Master Ref No. / Shipment Type / Status)만. 해상 추가 4필드(Line Bkg. No. / Load Type / Service Term / B/L Type)는 **모두 없음** (AWB 특성).
- **House AWB List 인라인 그리드**
  - 컬럼: `No` / `House AWB` / `Shipper` / `Consignee` / `DOC Partner` / `Package` / `Gross Wt.` / **`Charge Wt.`** / `CBM` / `Order No`
  - 해상 대비 차이: **Vessel / Voyage / POL / ETD / ETA 컬럼 제거**(항공은 flight 기반), **Charge Wt. 컬럼 추가**(청구중량, 항공 고유)
  - 하단 합계 라인: Package · Gross Wt. · Charge Wt. · CBM
  - 그리드 버튼: **New HAWB** / **House Consol** / 엑셀 내보내기 / 접기·펼치기
- **Schedule 블록** (항공 고유, S-02 항공 변형과 동일한 leg 기반 구조 — §S-02[AIR/EXP] Schedule 블록 명세 준용)
  - 상단 헤더: `*Airline` (수출 관점 라벨, 수입은 `Carrier`) + `*Departure`
  - 중단 Schedule Leg 그리드: E-07 Master Schedule Leg 엔티티와 1:1 매핑. 컬럼·행 정책·순서 판정은 S-02와 동일
  - 하단 요약 영역 (읽기 전용 파생): Destination ← 마지막 행 `*To` / On board ← 첫 행 `*On Board` / Arrival ← 마지막 행 `*Arrival`
  - **Master 고유: `Update HAWB Schedule` 버튼** — 연결된 HAWB에 Master Schedule을 일괄 전파 (항공 Master 고유 기능). 덮어쓰기 vs 선택 확인 세부는 후속 결정 (06_OPEN_ITEMS §3.1)
  - 바인딩 규칙: DOMAIN §3.2 Schedule Leg 엔티티 및 §6.12 B/L 출력 연동 원칙 준수
- **Cargo** (공통 Cargo 블록 + 항공 전용 필드): 공통 Main Item / HS Code / Package/Unit / Gross W/T / KG/LBS / CBM에 더하여 **Volume W/T** / **Charge W/T** / **Rate Class** 추가. **R/Ton 없음**(해상 고유).
- **Document**: Co-load Type / Co-Load Agent
- **Other 섹션 (항공 Master 전용 신규 블록)**: Flight Type / **Security Status** / Settle Partner / Operator / Team
  - **Flight Type**: 여객기(Passenger) / 화물기(Freighter) / 혼합기 등 구분 (DOMAIN §4.3)
  - **Security Status**: 보안검색 상태 (예: `SPX` = Secured for Passenger and all-cargo aircraft). DOMAIN §6.5·§4.3 항공 보안검색 관리 필드.
- **Issue Information** (3필드): **Issue Date / Signature / Issue Place** — 해상 Master 수출의 `Issue Date 1필드` 와 달리 3필드로 확장. Signature는 MAWB 양식의 서명란에 바인딩.
- **Trade** (해상 Master보다 필드 수 많음): `Currency` / `Freight Term` / **`Other Term`** / `D.V Carriage`(기본값 `N.V.D.`) / `Insurance`(기본값 `NIL`) / `D.V Customs`(드롭다운) / `Account Information`(예: `FREIGHT PREPAID`)
  - 해상 Master의 `Freight Term / Issue Date 2필드만` 구성과 근본적으로 다름. 항공 MAWB 출력 양식에 필요한 Raw 값을 Entry에서 관리해야 하므로 필드 확장.
- **Handling Information 선반입 상수**: `ATTACHED : COMM INV & P/LIST` (편집 가능한 초기값)
- **Dimension 그리드 (항공 전용, 해상 Container 그리드 위상 등가)**
  - 컬럼: Length / Width / Height / Quantity / CBM / Volume Wt.
  - **`CM / 6000` 제수 선택자**
  - **`Load Dimension` 버튼** — 연결된 HAWB들의 Dimension 레코드를 Master로 일괄 가져오기. Master Dimension은 하위 HAWB 집계 성격이므로 수동 편집·`Load Dimension` 버튼 기반 동기화 두 가지 방식 모두 지원 예정(세부 동작 후속 결정).
  - 행 단위 추가/삭제 컨트롤
  - 하단 합계 라인: Quantity · CBM · Volume Wt.
- **문서 설명 영역 명칭**: `Nature & Quantity of Goods`. **선반입 상수**: `CONSOLIDATION SHIPMENT / AS PER ATTACHED MANIFEST / FREIGHT PREPAID` (편집 가능한 초기값)
  - 부지약관 드롭다운 없음 (해상 전용 기능)
- **Charge (Print Only) 영역 (항공 전용, Main 탭 하단 하위 영역)**
  - 그리드 컬럼: `No / Freight / Freight Name / Currency / Per / P/C / Gross W/T / Rate Class / Charge W/T / Rate / Total Amount / AS`
  - 용도 구분은 S-02 Main 탭의 Charge(Print Only) 정의와 동일 — MAWB 인쇄 양식의 Charge 섹션용이며 BMS Freight 탭과 별개
  - 하단 합계 라인: Gross W/T · Charge W/T · Rate · Total Amount
- **Other Charge 영역 (항공 전용)**: 자유 텍스트 + **`Other Apply`** 버튼. MAWB 인쇄 양식의 Other Charges란에 바인딩
- **Container 그리드 없음** (항공은 컨테이너 개념 부재)
- **하단 액션 버튼 스트립** (항공 Master 수출 고유)
  - **출력 트리거 버튼** (3개): **MAWB** / **Shipping Request** / **Shipping Advice** → 공용 출력 팝업(부록 §C) 호출
  - **운영 버튼** (5개): **Profit/Loss** / **House B/L Load** / **M/F Send** / **AFR Send** / **EDMS** — 해상 수출과 동일 구성
  - 버튼 그룹 내 배치 순서(출력 3개 vs 운영 5개의 시각 구분 포함)는 디자인 기획 시 확정

##### `S-04[AIR/IMP]` 항공 수입 변형 고유 구성

위 `S-04[AIR/EXP]` 를 공통 베이스로 하되, 아래 항목이 다르다.

- **헤더**: 수출과 동일 4필드 (Master AWB No. / Master Ref No. / Shipment Type / Status)
- **Schedule 라벨**: **`Carrier`** (수출의 `Airline` 대신. 수입 관점에서는 도착지로의 운송인 개념)
- **Issue Information 블록 전체 없음** — 해상 수입과 마찬가지로, 항공 수입도 출력 대상이 없으므로(부록 §C-2) Issue 관련 필드가 Entry에 불필요. `S-02[AIR/IMP]` 와도 일관.
- **Other 섹션**: **Security Status 제거**. Flight Type / Settle Partner / Operator / Team만 유지 (보안검색은 출발지 소관이므로 수입 측에서 관리하지 않음)
- **Consignee pre-fill 관행**: 레거시에서 포워더 자체(MJLHKG 등)가 Consignee로 pre-fill되는 패턴이 관찰됨. 이는 MAWB 수입 측에서 해외 발송인(Co-Load Agent)이 국내 포워더를 수하인으로 지정하는 업무 특성 때문. 신규 시스템에서 pre-fill 자동화 여부는 후속 결정 (기본값: 수동 입력).
- **하단 액션 버튼 스트립**: **출력 트리거 버튼 3개(MAWB / S/R / S/A) 모두 제거** (수입 변형은 출력 대상 없음, 부록 §C-2). **운영 버튼 5개(Profit/Loss · House B/L Load · M/F Send · AFR Send · EDMS)만 유지**.

### S-06: Truck B/L Entry

> **UI 규약** — UI 표현·공통 컴포넌트는 **부록 §V1 준수**. 본 섹션은 Trucking 고유 필드·블록 구성만 명세한다.

- **주 사용자**: 내륙운송 담당자. 운송 모드 = Trucking 인 House B/L 입력.
- **분류**: **단독 화면**. 해상/항공 House B/L Entry(S-02)와 별개이며 변형 조립 대상이 아니다 (부록 §B 적용 대상은 해상/항공 4변형뿐). House B/L 공통 본체 + **트럭 확장(E-20)** 으로 구성되는 B/L을 입력한다.
- **핵심 기능**
    1. Trucking House B/L 생성 (Master 없이 단독 존재 가능, DOMAIN §4.4)
    2. 출발지·도착지 담당자 연락처 입력
    3. 트럭·기사·배차 정보 등록 (E-20 트럭 확장)
    4. 운송 상태 업데이트 (배차·픽업·인도 완료)
- **출력 기능 없음**: Trucking B/L은 본 시스템 출력 대상이 아니다(DOMAIN §6.1, 부록 §C-1). 따라서 **헤더에 출력 버튼을 두지 않는다**. 레거시 디자인에 보이는 "Print" 버튼은 특정 포워더 전용 커스터마이징이며 신규 시스템에서는 채택하지 않는다.
- **Main 탭 영역 구성**
    - **탭 구성**: Main / Freight **2탭**. Document / EDI / Other 탭 없음. Freight 탭은 S-02와 동일하게 BMS 영역 셸만 구성하며, 세부 컬럼·바인딩은 BMS 기획 시 확정.
  - **Party**: 4슬롯 — Shipper / Consignee / Notify / DOC Partner (S-02 해상 House와 동일 슬롯)
    - Notify 보조 버튼은 **`Same as Consignee` 1개만 채택** — S-02 해상 House 정책과 동일. 레거시 디자인의 `Same As`(=Same as Shipper) 옵션은 미채택.
  - **Schedule**: Vessel/Voyage(고정값 `"TRUCK"`) / ETD / ETA / POL / POD — 해상 House 대비 **의도적으로 단순화**. Liner · Onboard · POR · Delivery · Final Dest. · Final ETA 없음.
  - **Cargo**: Package / Unit / Gross W/T / CBM / **Charge W/T(청구중량)** — Charge W/T는 Trucking 전용 필드.
  - **Trade**: **Incoterms / Freight Term 2필드만**. 해상 House의 Payable At · Triangle 없음.
  - **Document**: Pick-up Date / Trucker / Trucker PIC — Trucking 전용 구성.
  - **Performance**: Actual Customer / Customer PIC / Settle Partner / Sales Man / Operator / Team. **Sales Class 없음** (Trucking은 영업 분류 관리 대상 아님).
  - 그 외 Dimension 그리드 · Truck Information 그리드 · Reference Numbers · Marks & Description 등 구성은 기존 B/L 패턴 준용.

#### 필수 입력 필드 (Main 탭 기준)

- **필수**: Actual Customer / Operator / Team / Sales Man / POL / POD / ETD / ETA
- **필수 아님**: Vessel Name·Voyage No(고정값 `"TRUCK"`으로 저장되므로 사용자 입력 대상 아님) / Shipper · Consignee · Notify · DOC Partner / Incoterms · Freight Term / Pick-up Date · Trucker · Trucker PIC / Cargo 세부·Dimension 그리드·Truck Information 그리드
- **Trucking B/L No 채번 정책**: 미입력 시 자동 채번(§V1.5 `Auto on save` placeholder 규약). 별도 결정 사항 없음
- **검증 규칙**: S-02와 동일 (§11 `Required style`, 프론트 Zod + 백엔드 이중 검증)
- **경유지(Via)**: 관리 안 함. Trucking은 B/L 출력 대상도 아니고 경유지 개념이 화면 대상 아님 (DOMAIN §6.12).

- **레거시의 문제점**: Trucking B/L이 House B/L 도메인에 별도 확장 엔티티 분리 없이 그대로 통합 관리됨. Trucking에는 쓰이지 않는 해상/항공 전용 필드까지 공통 테이블에 전부 함께 정의되어 NULL 필드가 범람, 테이블 비대화와 데이터 일관성 저하 발생.
- **신규 시스템 개선점**: DOMAIN §4.4 원칙대로 Trucking을 해상·항공과 동등한 운송 모드로 취급. 해상/항공 House B/L과 동일한 조회·검색·B/L 체계 공유. **트럭 전용 필드는 E-20 확장 엔티티로 분리**하여 공통 본체 테이블의 NULL 필드 범람 문제 해소. 단, 화면은 단독으로 유지해 해상/항공 입력 UX와 혼재하지 않는다.
- **주요 데이터 / 연관 엔티티**: E-08 House B/L (운송 모드 Trucking), E-20 House B/L 트럭 확장

### S-08: Non B/L Entry

> **UI 규약** — UI 표현·공통 컴포넌트는 **부록 §V1 준수**. Party 카드의 주소 블록 생략 등 S-08 고유 Delta는 아래 본문에서 명세한다.

- **분류**: **단독 화면**. 해상/항공 House B/L Entry(S-02), Truck B/L Entry(S-06/S-07)와 별개이며 변형 조립 대상이 아니다 (부록 §B 적용 대상 아님). 본 화면 접근 시 E-08의 운송 모드 `JOB_DIV`가 자동으로 `Non B/L`로 설정되어 E-24 확장이 붙는다 (사용자 편집 불가). E-24 내부의 **Work Division 속성(Sea/Air/Warehouse/Trucking 4종)** 은 사용자가 드롭다운으로 선택하며, 값에 따라 Main 탭의 블록 구성이 동적으로 달라진다.
- **핵심 기능**
    1. Non B/L House B/L 생성 (Master 없이 단독 존재 가능, DOMAIN §4.5). 공통 본체(E-08, `JOB_DIV` = `Non B/L`) + Non B/L 확장(E-24) 구조
    2. **Work Division 선택 (필수)** — Sea / Air / Warehouse / Trucking 중 1값. E-24의 필수 입력 속성이며, **미선택 시 저장 차단**. 값에 따라 Main 탭의 확장 컴포넌트가 재조립되어 E-10 / E-11 / (공통+E-24 고유) / E-20 중 어느 원천에서 필드를 발췌할지가 결정됨
    3. 기본 정보 · 당사자 · 참고 번호 입력. 원본 해상/항공 B/L이 있는 경우 참고번호 필드로 원본 업무 연결
    4. 업무 상태 관리 (접수·처리·완료)
    5. **Non B/L 원본 B/L 출력**: 헤더 "출력" 버튼으로 공용 출력 옵션 팝업(부록 §C). Print date 타임스탬프는 MongoDB 로그 다큐먼트에 기록.
- **Main 탭 영역 구성**
  - **탭 구성**: Main / Freight **2탭**. 레거시 스크린샷의 Document 탭은 채택하지 않음. Freight 탭은 S-02와 동일하게 BMS 영역 셸만 구성하며, 세부 컬럼·바인딩은 BMS 기획 시 확정.

  - **Party**: **5슬롯** — Actual Customer / Shipper / Consignee / Notify / Settle Partner. 각 슬롯은 **§V1.9 Party 카드** 준수하되 **주소 블록 생략** (거래처 코드 + Name만 표시. 주소는 Party 마스터 조회로 확인). Notify 슬롯 보조 버튼은 S-02 해상 House와 동일 정책 — **Same as Consignee만 채택**, Same as Shipper는 미채택.

    - `Air` → E-11 발췌 블록 (Dimension 그리드·Cargo 블록 등)
    - `Warehouse` → 공통 본체 + E-24 고유 필드 중심. 별도 확장 원천 없음

## 부록: 공통 설계 가이드라인

모든 화면에 공통으로 적용되는 원칙. 개별 화면 상세에 반복 기재하지 않는다.

### A. 일반 원칙

- **권한 모델**: 해상/항공 화면은 `{운송 모드, 방향}` 권한 단위, Trucking / Non B/L 화면은 `{운송 모드}` 권한 단위로 접근 제어 (DOMAIN §6.7).
- **탭 분리**: 입력 화면은 탭 분리 전략으로 복잡도 관리 (TECH_STACK §4.1).
- **검증**: 프론트엔드는 React Hook Form + Zod, 백엔드는 양 스택(Spring Validation / Pydantic)에서 동일 스키마로 이중 검증.
- **조회 성능**: 대량 집계 쿼리 90% 개선, 일반 쿼리 50% 개선 목표 (OVERVIEW §3.1).
- **감사**: 모든 등록·수정에 등록자/시각, 수정자/시각 기록 (DOMAIN §6.10).
- **국제화**: 기본 다국어 지원. 라이브러리는 `next-intl` 유력.

### A-1. 사이드바 구조

- **모듈 라벨 그룹화**: 사이드바 메뉴는 모듈명(FMS / BMS / MDM / PFM / CMS 등) 단위로 그룹화한다. 각 모듈 라벨은 클릭 시 Expand / Collapse 동작.
- **사이드바 상단**: 전체 Expand / Collapse 버튼 + 즐겨찾기(즐겨 찾는 화면) 버튼을 유지한다.
- 디자인 초안의 Workspace / Shipments / Back office / System 같은 기능 분류 그룹은 채택하지 않는다.
- **사이드바 FMS 그룹 내 항목 수 (21개)**: 대시보드 `S-01` (공통/플랫폼 화면이나 FMS 그룹에도 중복 노출) · House B/L Entry/List 각 4변형(`S-02[*]`, `S-03[*]`) · Master B/L Entry/List 각 4변형(`S-04[*]`, `S-05[*]`) · Truck B/L Entry/List(`S-06`, `S-07`) · Non B/L Entry/List(`S-08`, `S-09`). **FMS 고유 화면은 20개**(Variant 확장 후)이며 대시보드는 공통/플랫폼 배번. 모든 Variant는 독립 메뉴 항목이다 (부록 §B-1 UI 분리 정책).
- FMS 내부 메뉴 수가 많아지므로 **모듈 하위 2단 그룹화(예: `Ocean` / `Air` / `Truck` / `Non B/L`)** 도입 여부는 CMS 모듈 기획 시 결정한다.
- My Menu Option(개인 즐겨찾기), System Develop 등 추가 메뉴는 CMS 모듈 기획 시 결정한다.

### V1. Entry 화면 공통 UI 규약

운영 레이아웃 V1 단일 기준(NFR §11 참조)으로 **모든 Entry 화면에 공통 적용되는 UI 표현 규약과 공통 컴포넌트 기능 계약**을 정의한다. 각 Entry 섹션 본문(S-02 / S-04 / S-06 / S-08)은 필드·블록 구성과 변형 Delta 기술에 집중하며, UI 표현·컴포넌트 패턴은 본 섹션을 단일 근거로 참조한다.

- 본 섹션은 **어떻게 표시하는가(How)** 를 담당한다. **무엇을 표시하는가(What)** — 필드·블록 실제 목록 — 는 각 Entry 섹션 본문에서 정의한다
- 본 섹션은 **필드·블록 단위 공통 컴포넌트 계약**을 정의한다. **탭 단위 조립 레지스트리**는 §B-2에서 정의하며, §B-2의 탭 컴포넌트는 §V1의 공통 컴포넌트를 조합해 구성한다
- V2~V4는 고객사 시연용 정적 목업으로 본 규약 대상이 아니다(NFR §11)
- 구체 색상값·픽셀 간격·폰트 크기 등 시각 토큰은 디자인 토큰 확정 시 명세

#### V1.1 화면 프레임

화면은 상단 고정 2-row 헤더 + 탭 내비게이션 + Main 콘텐츠 영역으로 구성된다.

- **Row 1 (액션바)**: 좌측에 화면 타이틀·모드·상태 뱃지·저장 메타·오퍼레이터, 우측에 상단 공통 액션 버튼 그룹 (§V1.7)
- **Row 2 (Document Key 필드 row)**: 문서 식별 핵심 필드를 탭 밖 헤더 위치에 가로 배치. 어느 탭에 있든 항상 가시. 필드 목록은 화면별 config
- 탭 영역 · Main 콘텐츠는 Row 1·2 아래에 위치

| 상단 3-column | 컨텍스트 블록 3개 병렬 (예: Party / Schedule / Trade·Performance) |
| 중간 full-width | 데이터 그리드 영역 (Container / Dimension 등) |

#### V1.4 섹션 헤더 스타일

- 섹션 헤더: 아이콘 + UPPERCASE 라벨 (예: PARTY, SCHEDULE, CONTAINER)
- 섹션 헤더 우측에 섹션별 추가 액션 버튼 그룹 배치 가능 (예: CONTAINER 헤더 우측의 `Batch split`, `Container → Cargo`, `Seal ≫`)
- 컬럼 내부 서브섹션은 small-caps 라벨 (예: TRADE·PERFORMANCE 컬럼 내부의 TRADE, DOCUMENT)

#### V1.5 필드 패턴

**필수 필드 표시**
- 필드 좌측에 얇은 붉은 수직 bar (indicator)
- 필드 라벨도 붉은 색상으로 강조

**엔티티 참조 필드 (Code + Name 페어)**
- 거래처·선사·항구 등 엔티티 조회 필드는 `code` + `name` 2-영역으로 구성
- code 영역에 검색 아이콘(돋보기). 클릭 시 엔티티 조회 모달 호출
- 선택 후 name 영역은 읽기 전용으로 자동 채움
- 거래처(Shipper/Consignee/Notify 등)는 추가로 하단에 **읽기 전용 주소 블록** 표시 (Name / Address line 1 / Address line 2 / City, District / Country)

**자동 생성 필드**
- 저장 시점에 시스템이 번호·식별자를 자동 채우는 필드는 placeholder `Auto on save` + 필드 우측 **상태 점(●) 아이콘** 규약 준수
- 적용 예: HOUSE B/L NO. / MASTER B/L NO.(연결 없음 케이스) / 기타 자동 채번 필드
- 상태 점의 색상·상태 구분(예: 대기·할당 완료) 세부는 디자인 토큰 확정 시 정의

**플레이스홀더 컨벤션**

| 필드 유형 | placeholder |
| --- | --- |
| 엔티티 참조 code 영역 | `code` |
| 엔티티 참조 name 영역 | `name` 또는 타입별(예: `Liner name`, `port name`) |
| 자동 생성 | `Auto on save` |
| 미입력 드롭다운 | `Select` |
| 날짜 필드 | `YYYY-MM-DD` |
| 값 없음 | `—` |

#### V1.6 상태·메타 표시 (Row 1 좌측 영역)

- **상태 뱃지**: 화면 타이틀·모드 우측에 작은 뱃지로 표시 (예: `DRAFT`). 뱃지 값 체계(상태 전이)는 후속 기획
- **저장 메타**: `Last saved` + 시각. 미저장 상태는 `-` 표시
- **오퍼레이터**: `Operator {username}` 형태로 현재 사용자 표시

#### V1.7 상단 공통 액션 버튼 (Row 1 우측 영역)

Entry 화면 Row 1 우측에 배치. **기본 공통 셋 6개**:

**Search / New / Copy / Print / Delete / Save**

- `Print`는 공통 셋이되 화면별 포함·제외 가변. 예: S-04 Master B/L Entry는 헤더 Print 없음(출력은 하단 전용 버튼으로 트리거, 부록 §C-2). Trucking B/L(S-06)은 출력 대상 아니므로 Print 없음
- `Search`는 현 단계 버튼 컴포넌트. 향후 전역 검색바(전문 검색)로 대체될 가능성 있음 — 현재는 버튼이 디폴트
- **화면별 추가 액션** — 위 6개 셋 외에 각 Entry 화면이 필요로 하는 액션 (예: `Change B/L No`, `Profit/Loss`, `House B/L Load`, `List`, `EDMS` 등)은 **각 Entry 섹션 본문에서 개별 명세**한다. Row 1 우측에 추가될 수도 있고, 하단 전용 스트립으로 배치될 수도 있다
- **현재 구현 상태** — 별도 Button 컴포넌트 파일 없음. (dev) 카탈로그(`/preview`)에 specimen으로 등록 (page-bundle-group · page-head-actions-group 2종으로 분할).

#### V1.8 데이터 그리드 규약

- **하단 집계 footer**: 수치 컬럼만 합계 표시 (예: `No. of Containers N | Gross W/T ... | CBM ... | Tare ... | VGM ...`)
- **다중 선택**: 좌측 끝 체크박스 컬럼
- 행 정렬(Drag & Drop), 행 단위 추가/삭제/순서 이동(+/−/↑/↓) 등 화면별 고유 동작은 각 Entry 섹션에서 명세

### B. 변형 조립 규칙 (Variant Composition)

#### B-0. Variant ID 체계

- `{운송 모드, 방향}` 4변형을 갖는 화면(`S-02` House B/L Entry, `S-03` House B/L List, `S-04` Master B/L Entry, `S-05` Master B/L List)은 **Variant ID 접미어** `[MODE/DIR]` 를 붙여 식별한다. 예: `S-02[SEA/EXP]`, `S-05[AIR/IMP]`.
- 접미어는 라우팅 경로·사이드바 메뉴·권한 매핑·문서 참조에서 일관되게 사용한다.
- **Variant 없는 화면**(`S-01` 대시보드, `S-06` Truck B/L Entry, `S-07` Truck B/L List, `S-08` Non B/L Entry, `S-09` Non B/L List)은 접미어 없이 그대로 참조한다. Trucking·Non B/L은 방향을 내부 속성으로만 가지며 화면 분기가 없다 (DOMAIN §6.7).

#### B-1. UI 분리 ↔ 구현 조립 (비대칭 정책)

- **UI·라우팅 단위는 변형별로 분리**한다. 즉 `S-02[SEA/EXP]` 와 `S-02[SEA/IMP]` 는 별개 URL·별개 사이드바 항목·별개 권한 체크 지점이다. 사용자는 "해상 수출 House Entry" 를 직접 북마크하거나 딥링크로 받을 수 있다.
- **구현 단위는 단일 코드베이스 + config 조립**이다. 각 Variant는 동일한 Shell 컴포넌트를 공유하고, 탭·컬럼·필터·행 액션 등 변형별로 달라지는 부분만 레지스트리(config)에 선언한다.
- 이 분리/조립 비대칭이 주는 이득:
    - *권한 단위와 라우팅이 1:1* — `{모드, 방향}` 권한이 URL 경로에 바로 매핑되어 미들웨어 체크가 단순해진다
    - *북마크·딥링크 실용성* — 쿼리파라미터 인코딩 없이 변형을 직접 URL로 지정
    - *캐시·무효화 분리* — React Query 키가 경로별로 자연 분리
    - *컬럼/탭 셋 차이 수용* — 변형마다 실제로 보여줘야 할 컬럼·탭이 다르므로 토글 UX 부담이 없다
- 본 규칙은 **Entry와 List 모두에 적용**된다. 배번 고정 원칙(S-01~S-09)은 유지되고, 변형은 접미어로만 표현된다.

#### B-2. Entry 화면 3층 구조 (S-02, S-04)

```
Shell (라우팅 · 컨텍스트 · 저장/제출 플로우)
  └─ <Context value={{ mode, direction }}>
      └─ TabRegistry[mode][direction]          ← 설정(config)
          └─ Tab 컴포넌트 (공용 / 모드 전용 / 방향 전용)
```

- **분기는 코드가 아닌 설정(config)으로.** `if (mode === 'air')` 같은 분기문을 컴포넌트 내부에 흩뿌리지 않는다. 변형별로 어떤 탭·필드 블록을 조립할지 **레지스트리에 선언적으로** 기술한다. 새 변형 추가 = 레지스트리 한 줄.
- **탭 컴포넌트 3분류.**
  - *공용*: `<BasicInfoTab />`, `<PartiesTab />`, `<DescriptionTab />` — 4변형 공통
  - *모드 전용*: `<ContainerTab />`(해상), `<DimensionTab />` / `<AirCargoTab />` / `<ChargePrintOnlyTab />`(항공), `<SecurityScreeningTab />`(항공)
  - *방향 전용*: `<ANDOTab />`(해상 수입), `<FHDTab />`(항공 수입) — **수출 면장(E-17)은 EDI 탭 내 License 섹션으로 편입되어 독립 탭에서 제외**. 수입 방향 면장은 포워더 업무 범위가 아니므로 본 시스템 대상 아님.
- **§V1 공통 컴포넌트와의 관계**: 본 §B-2의 탭 컴포넌트(위 3분류)는 **§V1.9의 필드·블록 단위 공통 컴포넌트**(Party 카드, Code+Name 페어, 데이터 그리드 등)를 조합해 구성한다. §B-2는 탭 단위 조립 레지스트리의 SSOT, §V1은 필드·블록 단위 UI 규약·컴포넌트 계약의 SSOT다.
- **폼 스키마는 Zod discriminated union.** `z.discriminatedUnion('mode', [oceanSchema, airSchema])` 로 변형을 타입 레벨에서 강제한다. "항공인데 VGM 접근" 같은 실수가 컴파일·런타임 양쪽에서 차단된다. 방향 축도 동일하게 discriminated union 으로 관리하여 `{모드, 방향}` 4조합이 타입 레벨에서 배타적으로 체크된다.
- **크로스탭 검증은 Shell의 submit 파이프라인에서.** 각 탭이 독립적일수록 "해상 수출인데 컨테이너 탭 비어 있음" 같은 검증 책임 위치가 모호해진다. Shell이 최종 submit 직전 통합 Zod 스키마로 한 번 더 검증한다.
- **탭 순서·라벨의 변형별 차이는 레지스트리의 `order` · `label` 필드로 조정.** 컴포넌트 자체는 건드리지 않는다.

#### B-3. List 화면 구조 (S-03, S-05)

```
Shell (라우팅 · 변형 컨텍스트 · 가상 스크롤 · 페이지네이션 · 일괄 선택
       · Excel export · 전역 검색 · 다중 컬럼 정렬 · 공용 출력 팝업 트리거)
  └─ <Context value={{ mode, direction }}>      ← URL에서 고정
      └─ ListRegistry[mode][direction]          ← 설정(config)
          ├─ columns: ColumnDef<Row>[]
          ├─ filterSchema: ZodSchema
          ├─ defaultSort: SortState
          ├─ rowActions: RowAction[]            ← 변형별 출력/EDI 재전송 등
          └─ baseWhere: QueryPredicate          ← 전역 고정 WHERE
```

- **공용 Shell이 담당**: 가상 스크롤(TanStack Virtual), 페이지네이션, 다중 행 선택, Excel/CSV export, 전역 FTS 검색, 다중 컬럼 정렬(헤더에 우선순위 숫자+↑↓), 공용 출력 옵션 팝업(부록 §C) 트리거.
- **변형별 config가 담당**:
    - *컬럼 정의* — 변형마다 표시할 컬럼이 실제로 다르다. 예: `S-03[SEA/EXP]` 는 Container No·VGM·POL/POD 중심, `S-03[AIR/IMP]` 는 Flight·MAWB·보안검색 중심
    - *필터 스키마* — 변형 고유 검색 조건 (Zod로 타입 강제)
    - *기본 정렬* — 예: 수출은 ETD 내림차순, 수입은 ETA 내림차순
    - *행 인라인 액션* — 변형별 출력 종류(부록 §C-1 매트릭스), EDI 재전송 등
    - *기본 고정 WHERE* — URL에서 고정된 `mode`·`direction`이 모든 쿼리에 자동 주입
- **쿼리는 전역 고정 WHERE + 사용자 필터 합성**. 변형이 바뀌면 완전히 다른 쿼리가 나가므로 캐시 키도 자연스럽게 분리된다.
- **일괄 출력의 동일 변형 제약**은 본질적으로 유지된다. 변형별로 화면이 분리되어 있으므로 한 화면에서는 선택 행이 자동으로 동일 변형이 된다.

### C. 출력 옵션 팝업 (공용 모달)

FMS의 모든 문서 출력은 독립 화면을 두지 않고 공용 모달 컴포넌트로 처리한다. 팝업은 **단순 출력 옵션 표시용**이며, 데이터 자체를 수정하지 않는다. **개인화·자동화는 지양**하고, 필요한 경우 추후 고객 요구 사항에 따라 애드온으로 확장한다.

#### C-1. 출력 종류와 모드 매핑

| 출력 종류 | 해상 | 항공 | Non B/L | 대상 |
| --- | :---: | :---: | :---: | --- |
| B/L | ✅ (수출) | ❌ | — | House B/L (해상 수출) |
| **HAWB** (House Air Waybill) | ❌ | ✅ (수출) | — | House B/L (항공 수출). **IATA 표준 양식 고정** |
| **MAWB** (Master Air Waybill) | ❌ | ✅ (수출) | — | **Master B/L (항공 수출)**. **IATA 표준 양식 고정** |
| A/N, D/O | ✅ (수입) | ❌ | — | House B/L (해상 수입) |
| S/A (Shipping Advice) | ✅ (수출) | ✅ (수출) | — | House + Master 수출 |
| S/R (Shipping Request) | ✅ (수출) | ✅ (수출) | — | House + Master 수출 |
| Non B/L 원본 B/L | — | — | ✅ | Non B/L |

**제외**: 해상 Master B/L 본체(선사 발행), Trucking B/L. 항공 MAWB는 포워더가 발행·출력하므로 제외 대상이 아니다.

> S/A·S/R이 Master 수출에서 가능한 이유: Master 단위 묶음으로 수출 업무상 S/A·S/R을 내보내는 경우가 있다. 이는 Master B/L 자체의 출력이 아니며, Master 화면을 트리거 지점으로 사용할 뿐이다.

> 수입 업무에서 화주가 받는 B/L·AWB는 **해외 파트너 발행 서류**이며 본 시스템에서 출력하지 않는다.

| 트리거 위치 | 출력 가능 종류 | 트리거 UI |
| --- | --- | --- |

> **Master B/L Entry(S-04)는 헤더에 인쇄 버튼을 두지 않고 하단 전용 버튼으로만 출력을 트리거**한다. 출력 팝업 모달 자체는 House와 공용으로 사용한다.

#### C-3. 옵션 항목

- **Entry 화면에서 관리되는 도메인 필드는 옵션이 아니다.** B/L Type / Issue Date / 원본·사본 부수 / Sign 담당자 / A/N Date / D/O Date / S/R 일자 등은 Entry 필드이며 출력 팝업의 선택지가 아니다.

#### C-4. 출력 이력

- 발행·회수·Surrender·발송·컨펌·Switch 발행·재발행 등 업무 상태는 **시스템이 관리하지 않는다** (사용자 업무 영역).
- 출력 실행 시 **Print date 타임스탬프만** MongoDB 로그 다큐먼트에 기록한다 (TECH_STACK §5 (c) 용도와 정합). 기존 RDB 테이블과 FK 관계를 두지 않는다.
- 로그 다큐먼트 최소 필드: 출력 종류 / 대상 식별자(HBL No · MBL No · 대상 구분) / 출력자 / Print date 타임스탬프 / `trace_id`.
- 단건 출력 시 팝업 하단에 최근 Print date 요약을 표시할 수 있다.
- 재발행·취소에 대한 애플리케이션 차원 차단·경고 로직은 두지 않는다.

#### C-5. 비기능 / 템플릿

- PDF 생성 p95 < 1초 (NFR §1.1, 단건). 대량 일괄은 NFR §1.3 배치 정책에 따라 비동기 처리.
- 템플릿·폰트 사전 로딩 전제.
- **HAWB·MAWB 템플릿은 IATA 표준 양식 고정.**
- **모든 출력 템플릿은 영어 고정.** i18n 미적용 (NFR §9.3 출력 템플릿 예외 조항과 정합).
- 템플릿은 Entry에 입력된 값을 **단순 바인딩**한다. 출력 시점의 자동 채움·자동 계산·자동 세팅 로직은 두지 않는다 (A/N Date · D/O Date · S/R 일자 등은 Entry에서 수동 입력).

#### C-6. 미지원 기능 (개인화·자동화 지양 원칙)

본 시스템은 문서 출력까지만 담당한다. 아래 기능은 **MVP 미지원**이며, 추후 고객 요구에 따라 애드온으로 검토한다.

- 이메일 발송 통합 (PDF 다운로드·인쇄까지만 담당)
- A/N Date · D/O Date · S/R 일자 자동 세팅
- 출력 시점의 템플릿 자동 채움·자동 계산
- S/R → House B/L 데이터 자동 승계
- 화주 컨펌 이력 기록
- 발송·발행·회수·재발행 이력 관리
