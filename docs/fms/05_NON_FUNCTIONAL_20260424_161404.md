## 1. 성능

### 1.1 목표 응답시간 (주요 화면/API별)

OVERVIEW §3.1의 KPI가 기준점.

| 분류 | 레거시 실측 (외부 제공) | 목표 | 근거 |
| --- | --- | --- | --- |
| 상위 10개 대량 집계 쿼리 (List 화면의 무거운 필터·집계) | 30~120초 | **3~12초 (90% 단축)** | OVERVIEW §3.1 |
| 일반 조회 쿼리 (List·Entry 기본 조회) | 사용자 제공 예정 | **50% 단축** | OVERVIEW §3.1 |
| 대시보드 (S-01) 초기 TTFB | - | 초단위 | SCREENS S-01 (Redis 캐싱은 CMS 도입 이후 적용, TECH_STACK §5) |
| Entry 화면 저장 | - | 2~3초 이내 (권장 초안) | **TBD** |
| Report PDF 단건 생성 (B/L·HAWB·MAWB·A/N·D/O·S/A·S/R·Non B/L 원본 B/L 공용 팝업, SCREENS 부록 §C) | - | **p95 < 1초** | 고정 템플릿 + 필드 치환 방식. AcroForm 또는 사전 컴파일 템플릿, 템플릿·폰트 사전 로딩 전제. 대량 출력은 §1.3 배치 정책에 따라 비동기 처리. |

- **측정 기준**: 응답시간 목표는 **p95 기준**으로 관리한다.
- **기준선 출처**: 레거시 쿼리의 p50/p95 기준선은 **별도 레거시 운영 프로젝트로부터 사용자가 제공**한다 (OVERVIEW §3.1). 본 프로젝트에서는 **신규 시스템 쪽의 p50/p95만 측정**하여 제공받은 기준선과 비교한다. 따라서 레거시 측 기준선 측정을 본 프로젝트의 초기 작업에 포함하지 않는다.

### 1.2 처리량 (TPS, 동시 사용자)

실제 운영 트래픽은 고객사 확장에 따라 점진적으로 증가하므로 **운영 환경 절대 수치 확정은 보류**한다. 대신 **합성 부하 테스트 환경에서의 목표치를 가혹하게 설정**하고, 이를 빌드/CI 파이프라인에서 회귀 테스트로 강제한다. 부하 테스트는 다음 4축으로 정의한다.

#### 부하 프로파일

| 구분 | 목표치 | 비고 |
| --- | --- | --- |
| 동시 사용자 (concurrent) | **5,000명** | 복수 중형 고객사 + 복수 소형 고객사 + 멀티테넌트 확장 여유 |
| 평상 RPS | think time 설정에 따라 도출 (가이드라인: 약 500~1,500 RPS sustained) | 사용자 플로우와 think time으로 역산 |
| 피크 RPS (Spike) | **평상의 3배 (10분 지속)** | 적하목록 마감 직전, 야간 EDI 컷오프 직전 폭증 가정 |
| Soak 지속 시간 | **24시간 연속** | 메모리·커넥션 누수 검출 |

#### 데이터 볼륨 (적재 상태)

부하 테스트는 아래 데이터량이 적재된 상태에서 수행한다. **빈 DB 상태의 부하 테스트는 의미 없다** — 실제 병목은 인덱스 효율과 쿼리 플랜이 달라지는 데이터 규모에서 발생한다.

| 엔티티 | 목표 데이터량 | 비고 |
| --- | --- | --- |
| Master B/L | **200만 건** | 30년치 운영 데이터 가정 |
| House B/L | **600만 건** | 30년치 운영 데이터 가정 |
| 컨테이너 | **3,000만 건** | HBL의 5배 (LCL 혼재 고려) |

**데이터 확보 방식**: 위 규모의 부하 테스트용 데이터는 **레거시 데이터를 샘플링한 뒤 합성 규칙으로 증식**하여 생성한다 (OVERVIEW §3.3에 따라 레거시 스키마를 직접 마이그레이션하지는 않으며, 개인정보·영업정보는 마스킹·합성으로 대체한다, §6.3 참조).

#### 시나리오 (사용자 플로우 비중)

사용자 플로우 단위로 비중을 정의한다.

| 사용자 플로우 | 비중 | 플로우 구성 |
| --- | --- | --- |
| List → Entry 조회 플로우 | **5** | List 진입 → 검색·필터 → 행 클릭 → Entry 단건 조회 |
| Entry 단독 조회 | **2** | 직접 URL·즐겨찾기로 Entry 진입 |
| Entry 저장 | **1** | 신규 작성 또는 수정 후 저장 |
| Report 출력 | **1** | B/L·HAWB·MAWB·A/N·D/O·S/A·S/R·Non B/L 원본 B/L 출력 PDF 생성 (공용 팝업, SCREENS 부록 §C) |

**도메인 특성 반영**: 컨테이너는 HBL에 종속되어 조회되며 단독 조건 조회는 거의 발생하지 않는다 (DOMAIN §3.2 카디널리티 참조). 따라서 부하 시나리오는 **HBL List의 복합 필터·집계 쿼리에 가혹 모드**로 집중한다 — 이는 OVERVIEW §3.1의 "30~120초 대량 집계 쿼리" 시나리오와 정확히 일치하는 경로다.

#### 성공 기준 (위 부하 적용 중)

- p95 응답시간이 §1.1 목표치 이내 유지
- p99 응답시간이 §1.1 목표치의 2배 이내
- 에러율 < 0.1%
- DB 커넥션 풀 사용률 < 80%
- 애플리케이션 인스턴스 CPU < 70% (스케일아웃 여지 확보)

이 기준이 **24시간 Soak 동안 어느 시점에도 깨지지 않아야** 통과로 본다. 일시적 통과는 무효.

#### 테스트 환경

- **부하 발생 도구**: **k6** 우선 검토 (Grafana 생태계 통합, §8.1과 자연 연계). Locust(Python)·Gatling(JVM)도 후보지만 Python 재구현 완료 후 비교 평가를 위해 도구 중립성이 필요한 k6를 선호.
- **인프라 사양**: **프로덕션과 동등하거나 작은 환경**에서 통과해야 함. 큰 환경에서 통과시키는 것은 의미 없음.
- **양 스택 동시 측정**: Java 스택은 1단계에서, Python 스택은 2단계(재구현 완료 후)에서 동일 부하 프로파일·성공 기준을 통과해야 한다. 이는 OVERVIEW §6.2의 듀얼 백엔드 평가에 직접 반영된다 (TECH_STACK §2.1 참조).

#### 트래픽 패턴 메모

포워딩 업무는 **주간 업무 시간대 집중 + EDI 컷오프 전후 스파이크** 패턴. 피크-평시 비율은 운영 측정 데이터 확보 후 보정.

## 3. 확장성

### 3.1 수평/수직 확장 기준

- **애플리케이션 (1단계, EC2 ASG)**: CPU·요청수 기반 Auto Scaling. 스케일-아웃 트리거 수치 **TBD**.
- **RDS PostgreSQL**: 수직 확장이 기본. 읽기 부하 증가 시 Read Replica 추가 검토.
- **Redis**: 세션·캐시 용도로 CMS 도입 시점부터 단일 노드로 시작 (TECH_STACK §5). 트래픽 증가 시 클러스터 전환 검토.
- **MongoDB**: Analytics Mart 전용 (TECH_STACK §5·§5.1). **PFM 실적 리포트 · BMS 고객사 특화 조회 화면 · 대시보드 집계 위젯** 부하 중심이므로 읽기 분산 우선. 쓰기 부하는 OLTP → Mart 이벤트 동기화 경로의 처리량에 연동된다.
- **2단계 (ECS Fargate 검토)**: 트래픽·비용 추이를 기반으로 EC2 ASG → ECS 전환 평가 (TECH_STACK §6.3).

### 3.2 병목 예상 지점

- **List 화면 대량 집계 쿼리**: 레거시의 주 병목 (30~120초). PostgreSQL 인덱스 설계 + FTS/Trigram으로 해소 (OVERVIEW §3.1, SCREENS S-03).
- **전역 검색 (FTS + Trigram)**: 데이터량 증가 시 인덱스 크기로 인한 성능 저하 가능. 파티셔닝·정기 REINDEX 정책 **TBD**.
- **EDI 전송 / 외부 연동**: 관세청·선사 장애 시 큐 적체 가능. resilience4j / tenacity + DLQ 전략 필요 (TECH_STACK §3.1, §3.2, §7).
- **Report PDF 생성**: 대량 출력 시 CPU 집중. 비동기 처리 + 파일 캐싱 검토.
- **Master-House 집계 (SCREENS S-05)**: Master 하위 House 건수·물량 인라인 집계는 단일 쿼리로 구성해야 N+1 방지. **HBL 본체 요약 비정규화 컬럼**(`total_allocated_teu`·`total_container_count` 등, DOMAIN §2.2 E-08)으로 Container 조인 없이 TEU·컨테이너 수 집계가 가능하도록 설계한다.
- **실적·분석 쿼리의 N:M:L 경로 (PFM·BMS·대시보드 집계 위젯)**: 레거시 실적 리포트가 `HBL × Container × Freight` 3자 조인으로 카티시안 폭발을 겪었다. 신규 설계는 (i) E-14b 교차 엔티티의 사전 계산 `allocated_teu`, (ii) HBL 본체 요약 비정규화 컬럼, (iii) **MongoDB Fact Mart 경유 조회**의 3중 구조로 해소한다 (DOMAIN §6.3·§6.11, TECH_STACK §5.1). OLTP 테이블에 다차원 집계 쿼리를 직접 돌리는 경로는 설계 단계에서 차단한다.

## 4. 보안

### 4.1 인증

- **기본**: JWT (TECH_STACK §3.4).
- **추가 예정**: OAuth2 — 외부 파트너 연동 또는 내부 SSO 통합 시점에 도입.
- **양 스택 공통 정책**: Python 재구현 시 1단계(Java)에서 확정한 JWT 발급/검증 키, Claim 구조, Refresh 정책을 그대로 따른다 (TECH_STACK §2.2).
- **세부 정책 확정 시점**: **CMS 모듈 기획 시점**에 확정한다. CMS가 유저 관리·세션·권한을 포함하는 모듈이므로 인증 스펙은 CMS와 함께 다룬다 (`06_OPEN_ITEMS.md` 재개 시점 맵 참조). Access / Refresh TTL, 재발급 절차 등은 이 시점 이전에 FMS 구현에서 임시 값으로 운용한다.

### 4.2 인가 / 권한 모델

- **핵심 권한 단위**: `{운송 모드, 방향}` 조합 (DOMAIN §6.7, SCREENS 부록 §A).
  - 예: 해상 수출 담당자는 해상 수출 데이터·화면에만 접근.
- **역할 계층**: 담당자 / 팀장 / 매니저 / 시스템 관리자 — 세부 매트릭스는 **CMS 모듈 기획 시점에 확정** (유저 관리가 CMS 범위이므로 권한 매트릭스도 동일 시점).
- **데이터 범위 필터**: B/L 단위로 "내가 등록한 것만 / 우리 팀 것만 보기" 식의 **행(Row) 단위 권한 모델은 도입하지 않는다** (DOMAIN §6.7 참조). 대리 업무 등으로 영업자와 Operator가 다를 수 있기 때문이다. Operator / Team / Sales Man 필드는 권한이 아닌 실적 집계·List 필터링 용도로만 쓰인다.
- **일관성 원칙**: 프론트엔드 라우트 가드, 백엔드 API 인가, DB 쿼리 필터가 **동일한 권한 모델**을 공유한다. 어느 한쪽이라도 누락되면 보안 공백 발생.

### 4.3 통신 암호화

- **외부 ↔ 클라이언트**: HTTPS (TLS 1.2 이상). CloudFront 경유 (TECH_STACK §6.2).
- **내부 서비스 간**: VPC 내부 통신. TLS 적용 범위는 VPC 설계 확정 시 결정 (TECH_STACK §6.4 말미 주석에서 VPC 정책 보류).
- **EDI / 외부 API**: 연동처별 정책에 따름 — `04_INTEGRATION.md`에서 정의.

### 4.4 저장 데이터 암호화

- **RDS PostgreSQL**: AWS RDS 저장소 암호화 활성화 (권장, 기본값).
- **S3**: SSE-S3 또는 SSE-KMS (권장).
- **MongoDB**: 저장소 암호화 설정 필요. Atlas 사용 여부 미확정 — 자체 호스팅 시 별도 설정.
- **Redis (ElastiCache 기준)**: 세션·캐시 용도이므로 암호화 필수는 아니나, 개인정보 유출 방지 차원에서 전송 중 TLS 활성화 권장.

- **응답 포맷 표준 — RFC 7807 Problem Details 채택**: 양 스택(Java Spring / Python FastAPI) 모두 **RFC 7807 `application/problem+json` 포맷**을 공통 에러 응답 스키마로 사용한다. 표준 필드(`type`·`title`·`status`·`detail`·`instance`)를 기본으로 하고, 필요 시 확장 필드(`trace_id`·`code`·`violations` 배열 등)를 덧붙인다.
  - **Java**: Spring Framework 6+ 내장 `ProblemDetail` 클래스 사용 (별도 라이브러리 불필요, TECH_STACK §3.1)
  - **Python**: FastAPI 예외 핸들러 또는 `fastapi-problem-details` 등으로 동일 스키마 구현 (TECH_STACK §3.2)
  - **다국어 처리**: `detail` 필드에 로케일별 메시지를 바인딩하는 방식을 기본으로 한다
- **HTTP 상태 코드 바인딩**: 각 에러 유형(시스템/애플리케이션 세부 유형)과 HTTP 상태 코드의 구체 매핑은 **개발 진행 과정에서 점진적으로 확정**한다. 일반 원칙은 다음을 기준으로 한다 — 인가 실패 401/403, 자원 없음 404, 유효성 실패 400 또는 422, 도메인 규칙 위반 409, 시스템 예외 500. CMS 도입 시 사용자 관리·권한 관련 에러 유형이 확장되며, 해당 시점에 카탈로그를 본격 정비한다.
- **에러 추적 연결**: 모든 에러 응답에는 `trace_id`를 확장 필드로 포함하여 Sentry·로그·알림과 즉시 연결 가능하게 한다 (§5.3).

## 11. 사용자 환경 설정

사용자별 화면 환경 개인화를 지원한다. 본 섹션은 MVP 단계(localStorage 전용)와 CMS 도입 이후 단계(서버 프로필)의 2단계 운영을 전제로 작성되었다.

### 11.1 설정 항목

| 항목 | enum | 기본값 | 비고 |
| --- | --- | --- | --- |
| Theme | `light` / `dark` | `light` | OS 연동(`auto`)은 현 단계 미지원 |
| Tone | `cool` / `neutral` / `warm` | `cool` | 색조 디테일은 디자인 토큰 확정 시 명세 |
| Density | `compact` / `cozy` / `comfortable` | `cozy` | 간격 실측값은 디자인 토큰 확정 시 명세 |
| FontSize | 11 / 12 / 13 / 14 / 15 / 16 (px, 1px 단위) | `13` | 슬라이더 UI 전제 |
| Required style | `bar` / `dot` / `fill` / `subtle` | `bar` | 입력 필드 필수 표시 방식 |

> **레이아웃 변형**: 운영에서는 **V1 단일 레이아웃**만 사용한다. V2~V4는 고객사 시연용 정적 목업으로 운영 라우팅에서 분리되어 있으며, 동적 전환 대상이 아니다 — 따라서 §11의 개인화 항목이 아니다. V1 레이아웃 자체는 `02_SCREENS.md`의 Entry 화면 기본 레이아웃 기준으로 관리한다 (`06_OPEN_ITEMS.md` §1.3, §6.1 참조).

### 11.2 저장 방식 — MVP / CMS 이후 2단계

#### MVP (CMS 도입 전)

- **localStorage 전용.** 사용자 관리·인증·서버 세션이 아직 없음 (`06_OPEN_ITEMS.md` §3.5 CMS 기획 참조)
- **단일 사용자 가정** — 브라우저·디바이스별 독립. 디바이스 간 동기화 없음
- 저장 키 단일 JSON 오브젝트:
  ```
  localStorage key: fms_user_preferences
  value: { theme, tone, density, fontSize, requiredStyle, _schemaVersion }
  ```
- `_schemaVersion`은 CMS 이후 서버 마이그레이션 진입 판정 근거

#### CMS 도입 이후

- **서버 프로필 저장.** `users` 테이블의 `preferences` 컬럼을 **JSONB 단일 컬럼**으로 운영
- JSONB를 채택하는 이유 — 환경 설정 항목은 시간 경과에 따라 증가 경향이 크고, 핵심 검색 대상이 되는 일이 없어 JSONB 성능 우려가 없음. 항목 추가 시 DDL 변경이 불필요함
- CMS 도입 첫 로그인 시 **localStorage → 서버 1회 마이그레이션** 수행 후 localStorage 키 제거
- 이후 localStorage는 서버 동기화 전 임시 저장 또는 미로그인 상태 fallback 용도로만 사용

### 11.3 적용 메커니즘 — FOUC(Flash of Unstyled Content) 방지

Theme·Tone·Density·FontSize는 첫 페이지 로드 시점에 적용되지 않으면 깜빡임이 사용자 체감으로 크게 잡힌다. 단계별 전략:

| 단계 | 전략 |
| --- | --- |
| MVP | `<head>` 내 동기 `<script>`로 localStorage 읽어 `<html data-theme data-tone data-density>` 속성과 `:root` CSS 변수 주입. HTML 본문 렌더 전에 적용되므로 FOUC 없음 |
| CMS 이후 | 로그인 시 preferences 값을 **쿠키에 미러링**. Next.js App Router 서버 컴포넌트에서 쿠키 읽어 SSR 응답의 `<html>` 태그에 속성·CSS 변수 포함해 내려보냄. 클라이언트 재적용 불필요 |

### 11.4 구현 범위 경계

- CMS 단계 서버 측 **API 계약·validation·권한 규칙**은 CMS 기획 라운드에서 확정 (`06_OPEN_ITEMS.md` §3.5)
- V2~V4 레이아웃을 운영 개인화 대상으로 편입하려면 **별도 기획 라운드**가 필요하다. 본 NFR은 해당 확장을 금지하지 않으나 현 시점 범위가 아니다
- 고대비·접근성(WCAG 관련) 프리셋은 본 섹션과 별개로 `§9.2 접근성`에서 관리한다

## 검토 요청 사항

1차 초안에서 확정되지 않은 항목을 우선순위별로 정리. 실제 구현·운영 시점에 맞춰 순차 확정한다.

### 우선순위 A — 초기 구현 전 확정 필요

- **듀얼 백엔드 평가 지표 구체화**: 성능·생산성·운영 비용 비교 항목을 §8.1 메트릭과 연결해 측정 가능하도록 정의 (OVERVIEW §6.2). Java 1단계 완료 시점 이전까지 확정 필요.
- **SLA 목표 수치**: 향후 신규 고객사 온보딩 시점에 고객사와 합의. 현 단계는 권장 초안(§2.1)으로 설계 진행.
- **감사 로그 보존 기간**: 관세법상 장부 보존 기간 확인 (§5.2). 법무 검토 필요.

> **이관 항목**: "성능 기준선 측정"(구 우선순위 A)은 사용자가 레거시 측정치를 제공하는 방식으로 전환되어 본 우선순위 목록에서 제거 (§1.1 참조). "권한 모델 세부 매트릭스"(구 우선순위 A)는 **CMS 모듈 기획 시점**으로 이관 (§4.2, `06_OPEN_ITEMS.md` 재개 시점 맵 참조).

### 우선순위 B — 1단계 운영 중 확정

- RTO / RPO 목표 (§7.2).
- 마스킹 / 익명화 규칙 (§6.3).
- 알림 임계치 수치 (§8.3).
- 브라우저 지원 버전 범위 (§9.1, 고객사 실사용 분포 기반).
- 추가 언어 지원 여부 — 중국어, 일본어 (§9.3).
- 취약점 등급별 대응 SLA 확정 (§4.6).
- 시크릿 순환 주기 (§4.5).
- 로그 보존 기간 세부 확정 (§5.2).
- ~~에러 응답 포맷 표준(RFC 7807 등) 채택 여부 (§9.4)~~ → **5라운드 후속 완료: RFC 7807 Problem Details 채택 확정** (§9.4 본문 참조)

### 우선순위 C — 향후 확장 시

- ECS 전환 기준 (TECH_STACK §6.3과 연계).
- 서비스 내 AI 기능 편입 시 §10 신규 작성 (응답 지연·비용 상한·프롬프트 로깅·폴백 등). §11은 사용자 환경 설정이 차지하고 있으므로 AI 관련은 §10에 배정.
- DR (재해 복구) 다른 리전 복제 전략 (§7.1).
- 모바일 좁은 화면 대응 (§9.2).
