## 1. 성능

### 1.1 목표 응답시간 (주요 화면/API별)

OVERVIEW §3.1의 KPI가 기준점.

| 분류 | 레거시 실측 (외부 제공) | 목표 | 근거 |
| --- | --- | --- | --- |
| 상위 10개 대량 집계 쿼리 (List 화면의 무거운 필터·집계) | 30~120초 | **3~12초 (90% 단축)** | OVERVIEW §3.1 |
| 일반 조회 쿼리 (List·Entry 기본 조회) | 사용자 제공 예정 | **50% 단축** | OVERVIEW §3.1 |
| 대시보드 (S-01) 초기 TTFB | - | 초단위 | SCREENS S-01 (Redis 캐싱은 ADMIN 운영 도입 이후 적용, TECH_STACK §5) |
| Entry 화면 저장 | - | 2~3초 이내 (권장 초안) | **TBD** |
| Report PDF 단건 생성 (B/L·HAWB·MAWB·A/N·D/O·S/A·S/R·Non B/L 원본 B/L 공용 팝업, SCREENS 부록 §C) | - | **p95 < 1초** | 고정 템플릿 + 필드 치환 방식. AcroForm 또는 사전 컴파일 템플릿, 템플릿·폰트 사전 로딩 전제. 대량 출력은 §1.3 배치 정책에 따라 비동기 처리. |

- **측정 기준**: 응답시간 목표는 **p95 기준**으로 관리한다.
- **기준선 출처**: 레거시 쿼리의 p50/p95 기준선은 **별도 레거시 운영 프로젝트로부터 사용자가 제공**한다 (OVERVIEW §3.1). 본 프로젝트에서는 **신규 시스템 쪽의 p50/p95만 측정**하여 제공받은 기준선과 비교한다. 따라서 레거시 측 기준선 측정을 본 프로젝트의 초기 작업에 포함하지 않는다.

### 1.2 처리량 (TPS, 동시 사용자)

실제 운영 트래픽은 고객사 확장에 따라 점진적으로 증가하므로 **운영 환경 절대 수치 확정은 보류**한다. 대신 **합성 부하 테스트 환경에서의 목표치를 가혹하게 설정**하고, 이를 빌드/CI 파이프라인에서 회귀 테스트로 강제한다. 부하 테스트는 다음 4축으로 정의한다.

#### 부하 프로파일

| 구분 | 목표치 | 비고 |
| --- | --- | --- |
| 동시 사용자 (concurrent) | **5,000명** | 복수 중형 고객사 + 복수 소형 고객사 + 멀티테넌트 확장 여유 |
| 평상 RPS | think time 설정에 따라 도출 (가이드라인: 약 500~1,500 RPS sustained) | 사용자 플로우와 think time으로 역산 |
| 피크 RPS (Spike) | **평상의 3배 (10분 지속)** | 적하목록 마감 직전, 야간 EDI 컷오프 직전 폭증 가정 |
| Soak 지속 시간 | **24시간 연속** | 메모리·커넥션 누수 검출 |

#### 데이터 볼륨 (적재 상태)

부하 테스트는 아래 데이터량이 적재된 상태에서 수행한다. **빈 DB 상태의 부하 테스트는 의미 없다** — 실제 병목은 인덱스 효율과 쿼리 플랜이 달라지는 데이터 규모에서 발생한다.

| 엔티티 | 목표 데이터량 | 비고 |
| --- | --- | --- |
| Master B/L | **200만 건** | 30년치 운영 데이터 가정 |
| House B/L | **600만 건** | 30년치 운영 데이터 가정 |
| 컨테이너 | **3,000만 건** | HBL의 5배 (LCL 혼재 고려) |

**데이터 확보 방식**: 위 규모의 부하 테스트용 데이터는 **레거시 데이터를 샘플링한 뒤 합성 규칙으로 증식**하여 생성한다 (OVERVIEW §3.3에 따라 레거시 스키마를 직접 마이그레이션하지는 않으며, 개인정보·영업정보는 마스킹·합성으로 대체한다, §6.3 참조).

#### 시나리오 (사용자 플로우 비중)

사용자 플로우 단위로 비중을 정의한다.

| 사용자 플로우 | 비중 | 플로우 구성 |
| --- | --- | --- |
| List → Entry 조회 플로우 | **5** | List 진입 → 검색·필터 → 행 클릭 → Entry 단건 조회 |
| Entry 단독 조회 | **2** | 직접 URL·즐겨찾기로 Entry 진입 |
| Entry 저장 | **1** | 신규 작성 또는 수정 후 저장 |
| Report 출력 | **1** | B/L·HAWB·MAWB·A/N·D/O·S/A·S/R·Non B/L 원본 B/L 출력 PDF 생성 (공용 팝업, SCREENS 부록 §C) |

**도메인 특성 반영**: 컨테이너는 HBL에 종속되어 조회되며 단독 조건 조회는 거의 발생하지 않는다 (DOMAIN §3.2 카디널리티 참조). 따라서 부하 시나리오는 **HBL List의 복합 필터·집계 쿼리에 가혹 모드**로 집중한다 — 이는 OVERVIEW §3.1의 "30~120초 대량 집계 쿼리" 시나리오와 정확히 일치하는 경로다.

#### 성공 기준 (위 부하 적용 중)

- p95 응답시간이 §1.1 목표치 이내 유지
- p99 응답시간이 §1.1 목표치의 2배 이내
- 에러율 < 0.1%
- DB 커넥션 풀 사용률 < 80%
- 애플리케이션 인스턴스 CPU < 70% (스케일아웃 여지 확보)

이 기준이 **24시간 Soak 동안 어느 시점에도 깨지지 않아야** 통과로 본다. 일시적 통과는 무효.

#### 테스트 환경

- **부하 발생 도구**: **k6** 우선 검토 (§8.1과 자연 연계). Locust·Gatling(JVM) 등도 후보지만 시나리오 정의가 간결한 k6를 선호.
- **인프라 사양**: **프로덕션과 동등하거나 작은 환경**에서 통과해야 함. 큰 환경에서 통과시키는 것은 의미 없음.

#### 트래픽 패턴 메모

포워딩 업무는 **주간 업무 시간대 집중 + EDI 컷오프 전후 스파이크** 패턴. 피크-평시 비율은 운영 측정 데이터 확보 후 보정.

## 3. 확장성

### 3.1 수평/수직 확장 기준

- **애플리케이션 (1단계, EC2 ASG)**: CPU·요청수 기반 Auto Scaling. 스케일-아웃 트리거 수치 **TBD**.
- **RDS PostgreSQL**: 수직 확장이 기본. 읽기 부하 증가 시 Read Replica 추가 검토.
- **Redis**: 세션·캐시 용도로 ADMIN 운영 도입 시점부터 단일 노드로 시작 (TECH_STACK §5). 트래픽 증가 시 클러스터 전환 검토.
- **MongoDB**: Analytics Mart 전용 (TECH_STACK §5·§5.1). **PMS 실적 리포트(Phase 2) · BMS 고객사 특화 조회 화면 · 대시보드 집계 위젯** 부하 중심이므로 읽기 분산 우선. 쓰기 부하는 OLTP → Mart 증분 동기화(주기 폴링) 경로의 처리량에 연동된다. (PMS Phase 1은 OLTP 직접 집계 — TECH_STACK §5.1.)
- **2단계 (ECS Fargate 검토)**: 트래픽·비용 추이를 기반으로 EC2 ASG → ECS 전환 평가 (TECH_STACK §6.3).

### 3.2 병목 예상 지점

- **List 화면 대량 집계 쿼리**: 레거시의 주 병목 (30~120초). PostgreSQL 인덱스 설계로 해소 (OVERVIEW §3.1, SCREENS S-03).
- **EDI 전송 / 외부 연동**: 관세청·선사 장애 시 큐 적체 가능. resilience4j + DLQ 전략 필요 (TECH_STACK §3.1, §7).
- **Report PDF 생성**: 대량 출력 시 CPU 집중. 비동기 처리 + 파일 캐싱 검토.
- **Master-House 집계 (SCREENS S-05)**: Master 하위 House 건수·물량 인라인 집계는 단일 쿼리로 구성해야 N+1 방지. **HBL 본체 요약 비정규화 컬럼**(`total_allocated_teu`·`total_container_count` 등, DOMAIN §2.2 E-08)으로 Container 조인 없이 TEU·컨테이너 수 집계가 가능하도록 설계한다.
- **실적·분석 쿼리의 N:M:L 경로 (BMS·대시보드 집계 위젯 등)**: 레거시 실적 리포트가 `HBL × Container × Freight` 3자 조인으로 카티시안 폭발을 겪었다. 신규 설계는 (i) E-14b 교차 엔티티의 사전 계산 `allocated_teu`, (ii) HBL 본체 요약 비정규화 컬럼, (iii) **MongoDB Fact Mart 경유 조회**의 3중 구조로 해소한다 (DOMAIN §6.3·§6.11, TECH_STACK §5.1). 이 N:M:L(물동량) 경로의 다차원 집계는 OLTP에 직접 돌리지 않는다. (**PMS 정산 실적의 평면 집계는 Phase 1 OLTP 직접 집계 — DOMAIN §6.11 단계적 예외**.)

## 4. 보안

### 4.1 인증

- **기본**: JWT (TECH_STACK §3.3).
- **구현 위치**: 인증은 **ADMIN 모듈(`java-spring-admin`)**이 담당한다. JWT 발급/검증·Refresh 회전·TTL·Claim 구조의 SSOT는 `docs/admin/03_ACCESS_CONTROL §5`이며, 공통 토큰 표준은 `03_TECH_STACK §2`를 따른다.

### 4.2 인가 / 권한 모델

- **권한 메커니즘 SSOT**: ABAC(속성·정책·프리셋)·역할 계층·인증의 상세는 **`docs/admin/03_ACCESS_CONTROL`**. 본 절은 FMS 고유 정책만 명시한다.
- **FMS 접근 단위**: `{운송 모드, 방향}` 조합 — ADMIN의 `module=FMS` + `fms_scope`(SEA/AIR/TRUCK/NON_BL) 속성과 메뉴 정책으로 실현 (DOMAIN §6.7, SCREENS 부록 §A). 예: 해상 수출 담당자는 해상 화면에만 접근.
- **행(Row) 단위 권한 미도입**: "내가 등록한 것만 / 우리 팀 것만" 식의 행 권한은 도입하지 않는다 (DOMAIN §6.7). Operator / Team / Sales Man은 권한이 아닌 실적 집계·List 필터링 용도.
- **일관성 원칙**: 프론트 라우트 가드(`MENU_*`)·백엔드 API 인가·DB 쿼리 필터가 동일 권한 모델을 공유한다 (`docs/admin/03_ACCESS_CONTROL §7`).

### 4.3 통신 암호화

- **외부 ↔ 클라이언트**: HTTPS (TLS 1.2 이상). CloudFront 경유 (TECH_STACK §6.2).
- **내부 서비스 간**: VPC 내부 통신. TLS 적용 범위는 VPC 설계 확정 시 결정 (TECH_STACK §6.4 말미 주석에서 VPC 정책 보류).
- **EDI / 외부 API**: 연동처별 정책에 따름 — `04_INTEGRATION.md`에서 정의.

### 4.4 저장 데이터 암호화

- **RDS PostgreSQL**: AWS RDS 저장소 암호화 활성화 (권장, 기본값).
- **S3**: SSE-S3 또는 SSE-KMS (권장).
- **MongoDB**: 저장소 암호화 설정 필요. Atlas 사용 여부 미확정 — 자체 호스팅 시 별도 설정.
- **Redis (ElastiCache 기준)**: 세션·캐시 용도이므로 암호화 필수는 아니나, 개인정보 유출 방지 차원에서 전송 중 TLS 활성화 권장.

### 4.5 시크릿 관리

- 환경변수 파일·리포지터리에 시크릿 직접 포함 금지.

- **시스템 관리 범위 밖 이벤트**: 발행·회수·Surrender 처리, Switch B/L 발행, A/N·D/O 발송, Shipping Advice 발송, S/R 컨펌 등은 사용자 업무 영역으로 간주하여 감사 로그 대상에서 제외한다 (DOMAIN §6.1).

- **외부 연동 이벤트**: EDI 전송/수신, 외부 API 호출 — 추적 ID 기반 (DOMAIN §6.10).

### 9.4 에러 처리 체계

- **에러 분류 2계층**
  - **시스템 에러**: 인프라·프레임워크 레이어에서 발생하는 예외(연결 실패·타임아웃·5xx 등). 내부 상세는 로그로 기록하고, 사용자에게는 일반 메시지와 추적 ID만 노출한다.
  - **애플리케이션 에러**: 도메인·유효성 검증 레이어에서 발생하는 에러(커스텀 Validation, 도메인 규칙 위반 등). 사용자에게 의미 있는 메시지를 직접 전달한다.
- **다국어 처리**: §9.3 기본 지원 언어에 맞춰 **한국어·영어**를 기본 제공한다. 에러 메시지 카탈로그는 i18n 키 기반으로 관리한다.
- **응답 포맷 표준 — RFC 7807 Problem Details 채택**: **RFC 7807 `application/problem+json` 포맷**을 공통 에러 응답 스키마로 사용한다. 표준 필드(`type`·`title`·`status`·`detail`·`instance`)를 기본으로 하고, 필요 시 확장 필드(`trace_id`·`code`·`violations` 배열 등)를 덧붙인다.
  - **구현**: Spring Framework 6+ 내장 `ProblemDetail` 클래스 사용 (별도 라이브러리 불필요, TECH_STACK §3.1)
  - **다국어 처리**: `detail` 필드에 로케일별 메시지를 바인딩하는 방식을 기본으로 한다
- **HTTP 상태 코드 바인딩**: 각 에러 유형(시스템/애플리케이션 세부 유형)과 HTTP 상태 코드의 구체 매핑은 **개발 진행 과정에서 점진적으로 확정**한다. 일반 원칙은 다음을 기준으로 한다 — 인가 실패 401/403, 자원 없음 404, 유효성 실패 400 또는 422, 도메인 규칙 위반 409, 시스템 예외 500. ADMIN 모듈의 사용자 관리·인증·권한 관련 에러 유형은 `docs/admin/03_ACCESS_CONTROL` 기준으로 카탈로그를 정비한다.
- **에러 추적 연결**: 모든 에러 응답에는 `trace_id`를 확장 필드로 포함하여 로그·알림과 즉시 연결 가능하게 한다 (§5.3).
