# 관리 도메인 (ADMIN)

> 본 문서는 **ADMIN 모듈의 도메인 언어와 엔티티**를 정의한다. ADMIN은 FMS(포워딩 업무)의 운영을 뒷받침하는 **사용자·조직·권한·공통 코드·콘텐츠 관리**를 담당한다.
> 프로젝트 공통 사항(목표·범위·기술·비기능)은 `docs/fms/00_OVERVIEW`·`03_TECH_STACK`·`05_NON_FUNCTIONAL`을 공유한다. **권한·인증의 상세 메커니즘과 사용자 환경설정은 `03_ACCESS_CONTROL`을 SSOT로 참조**하며, 본 문서는 그 도메인 모델(엔티티)만 정의한다.

## 0. 위치와 경계

- **ADMIN 모듈 = 기존 PRD의 "CMS 모듈"·"MDM 모듈"이 통합 실체화된 것.** `00_OVERVIEW §4.1`이 후순위(CMS·MDM)로 미뤄둔 유저 관리·인증·권한 모델·공통 코드·**마스터데이터(거래처 등)**의 DB 관리가 본 모듈로 구현됐다.
- **멀티모듈 구성**: ADMIN 백엔드는 `back-end/java-spring-admin` (FMS는 `back-end/java-spring`). 두 모듈은 **같은 PostgreSQL 인스턴스**를 쓰되 ADMIN이 `admin` 스키마를 소유한다 (`03_TECH_STACK §5` 스키마 정책).
- **Flyway 운용**: ADMIN은 Flyway를 **활성 운용**하며 `db/migration/V1~V54`로 스키마를 버전 관리한다. (FMS는 현재 `flyway.enabled=false`로 `schema/` 폴더 수동 관리 — 두 모듈의 운용 차이.)
- **모듈 간 의존 방향**: FMS는 ADMIN의 코드·유저 마스터를 **cross-schema 읽기**로만 참조한다(코드→이름 표시 등, `docs/fms/01_DOMAIN` CodeNameResolver). 마스터의 쓰기·소유권은 ADMIN에만 있다.

## 1. 용어집

### 1.1 접근 제어 (ABAC)
| 한글 | 영문/약어 | 정의 |
| --- | --- | --- |
| 속성 기반 접근 제어 | ABAC (Attribute-Based Access Control) | 사용자에게 부여된 **속성(attribute)** 집합과 자원(메뉴·버튼)에 걸린 **정책(policy)**을 대조해 접근을 결정하는 모델. 역할(Role) 고정 매핑 대신 속성 조합으로 유연하게 제어한다. |
| 속성 정의 | Attribute Definition | 속성의 키와 메타(값 타입·다중 허용 여부)를 정의하는 카탈로그. 예: `role`, `module`, `fms_scope` |
| 속성 값 | Attribute Value | ENUM 타입 속성이 가질 수 있는 허용 값 목록. 예: `module = {ADMIN, FMS}` |
| 모듈 속성 | module | 사용자가 접근 가능한 상위 모듈. `ADMIN` / `FMS` (다중 허용) |
| ADMIN 범위 속성 | admin_scope | ADMIN 모듈 내부에서 접근 가능한 관리 영역. `CODE` / `USER` / `CUSTOMER` / `CMS_NOTICE` / `ACCESS` (다중 허용) |
| FMS 범위 속성 | fms_scope | FMS 내부에서 접근 가능한 운송 라인. `SEA` / `AIR` / `TRUCK` / `NON_BL` (다중 허용). FMS 화면의 `{모드, 방향}` 접근 제어와 연결 (`docs/fms/01_DOMAIN §6.7`) |
| 정책 | Policy | 메뉴·버튼에 걸리는 접근 조건. `(자원, 속성키, 요구값)` 형태. 다중 조건은 **모두 만족(AND)** |
| 유효 속성 | Effective Attributes | 사용자에게 **직접 부여된 속성 ∪ 보유 프리셋의 속성**을 합집합(Union)한 결과. 정책 평가의 입력값 |
| 권한 프리셋 | Permission Preset | 재활용 가능한 속성 묶음 템플릿. 프리셋을 보유한 사용자에게 프리셋의 속성이 **즉시 반영(live binding)**된다 |

### 1.2 사용자·조직
| 한글 | 영문/약어 | 정의 |
| --- | --- | --- |
| 관리자/운영자 | Admin User | 시스템 사용자. 로그인 주체이며 속성·팀·프리셋을 가진다. 본 PRD에서는 "사용자"로 표기 |
| 역할 | role | 기본 역할 속성. `ADMIN` / `MANAGER` / `USER` (단일) |
| 팀 | Team | 조직/부서 단위. 사용자의 소속을 나타내며 FMS 실적·조회 필터의 담당팀(`teamCode`)으로 참조된다 |

### 1.3 공통 코드·콘텐츠
| 한글 | 영문/약어 | 정의 |
| --- | --- | --- |
| 코드 마스터 | Code Master | 공통 코드 그룹(카탈로그)의 머리 레코드 |
| 코드 디테일 | Code Detail | 코드 마스터에 속한 개별 코드 값 |
| 특화 코드 | Specialized Code | 별도 테이블로 관리되는 도메인 전용 마스터(거래처·항만·운송사·통화·환율·운임·HS코드·포장단위·국가) |
| 메뉴 | Menu | 화면 네비게이션 항목. 부모-자식 트리(self-ref) |
| 버튼 | Button | 메뉴에 속한 액션(생성·수정·삭제·내보내기 등) |
| 거래처 | Customer | FMS가 당사자(Party)로 참조하는 거래처 마스터. **ADMIN이 관리**(구 MDM 마스터데이터를 ADMIN으로 통합, `docs/fms/01_DOMAIN §6.9`) |
| 공지 | Notice | 대시보드·운영 공지(MJ Notice). 게시·만료 기간 관리 |

## 2. 핵심 엔티티

### 2.1 엔티티 목록
> ADMIN 엔티티는 `A-` 접두로 식별한다. 모든 마스터 엔티티는 단일 BIGINT IDENTITY PK를 가지며, 비즈니스 키는 UNIQUE 보조로 둔다(복합 PK 회피 원칙, §6).

| # | 엔티티 | 설명 |
| --- | --- | --- |
| A-01 | Admin User | 시스템 사용자. username(UNIQUE)·email·password_hash·active·attributes(JSONB)·team_id |
| A-02 | Team | 조직/부서. team_code(UNIQUE)·name·sort_order·active |
| A-03 | Attribute Definition | 속성 키 카탈로그. attribute_key(UNIQUE)·value_type·allow_multi |
| A-04 | Attribute Value | ENUM 속성의 허용 값. (attribute_key, value) UNIQUE |
| A-05 | Permission Preset | 권한 묶음 템플릿. code(UNIQUE, `PRESET_*`)·attributeValueIds(live binding) |
| A-06 | Preset–AttributeValue | 프리셋 ↔ 속성값 N:M 교차 |
| A-07 | User–Preset | 사용자 ↔ 프리셋 N:M 교차 |
| A-08 | Menu | 메뉴 트리. menu_code(UNIQUE)·parent_id(self-ref)·module_code |
| A-09 | Menu Policy | 메뉴 접근 정책. (menu_id, attribute_key, required_value) |
| A-10 | Button | 메뉴별 액션 버튼. button_code(UNIQUE)·menu_id·action_type |
| A-11 | Button Policy | 버튼 노출/실행 정책. (button_id, attribute_key, required_value) |
| A-12 | Code Master | 공통 코드 그룹. master_code(UNIQUE) |
| A-13 | Code Detail | 코드 값. (master_id, code_value) UNIQUE |
| A-14 | Carrier | 운송사(선사·항공사) 마스터 |
| A-15 | Country | 국가 코드 |
| A-16 | Currency | 통화 |
| A-17 | Exchange Rate | 환율 |
| A-18 | Freight | 운임 항목 마스터 |
| A-19 | HS Code | 관세 품목분류 |
| A-20 | Package Unit | 포장 단위 |
| A-21 | Port | 항만/공항 |
| A-22 | Customer | 거래처 (FMS 당사자 원천) |
| A-23 | Notice | 공지 |
| A-24 | Refresh Token | JWT 리프레시 토큰(해시 저장) |

### 2.2 엔티티별 핵심 속성
> 비즈니스 속성 기준. 모든 마스터는 공통 감사 필드(created_at/by·updated_at/by)와 `active` 플래그를 가진다(§6.1).

#### A-01 Admin User
- 식별: `username` (UNIQUE, **변경 불가**)
- 기본: `email`, `password_hash`(BCrypt, `$2a$` 접두), `active`
- 소프트 삭제: `deleted_at` (삭제 시각 기록 + 비활성화)
- **`attributes`** (JSONB): `Map<String, List<String>>` 구조. 사용자에게 **직접 부여된 속성**. 예: `{"role":["USER"], "module":["FMS"], "fms_scope":["SEA","AIR"]}`
- `team_id` (FK → A-02 Team, NULL 허용 = 미배정)
- 권한 계산의 입력값은 `attributes`(직접) ∪ 보유 프리셋 속성 → **유효 속성** (§4.1, `03_ACCESS_CONTROL`)

#### A-02 Team
- 식별: `team_code` (UNIQUE, 길이 20)
- `name`(길이 100), `description`, `sort_order`, `active`
- 기본 시드 6팀: SEA(해상)·AIR(항공)·SALES(영업)·FINANCE(재무)·HR(인사)·STRATEGY(전략)
- A-01 Admin User : A-02 Team = N : 1

#### A-03 Attribute Definition
- 식별: `attribute_key` (UNIQUE — String 자연키)
- `name`, `description`, `value_type` (`STRING` / `NUMBER` / `BOOLEAN` / `ENUM`), `active`, `allow_multi`(다중 값 허용 여부)
- 기본 키: `role`(ENUM/단일), `module`(ENUM/다중), `admin_scope`(ENUM/다중), `fms_scope`(ENUM/다중), `department`·`region`·`level`(조직 커스텀)

#### A-04 Attribute Value
- 식별: 단일 `id` (BIGINT) — 과거 복합 PK에서 단일 ID로 전환(마이그레이션 V34)
- 비즈니스 키: `(attribute_key, value)` UNIQUE
- `label`(표시명), `sort_order`, `active`
- 예: `module=ADMIN`/`module=FMS`, `admin_scope=CODE`/`USER`/`CUSTOMER`/`CMS_NOTICE`/`ACCESS`, `fms_scope=SEA`/`AIR`/`TRUCK`/`NON_BL`, `role=ADMIN`/`MANAGER`/`USER`

#### A-05 Permission Preset
- 식별: `code` (UNIQUE, **`PRESET_*` 접두 강제** — `PermissionPresetCodeValidator`)
- `name`, `description`, `active`
- `attributeValueIds` (List<Long>): 프리셋이 포함하는 A-04 속성값 id 목록. **live binding** — 이 목록 변경 시 보유 사용자 전원에 즉시 반영
- 예: `PRESET_SEA_MANAGER`, `PRESET_AIR_MANAGER`

#### A-08 Menu
- 식별: `menu_code` (UNIQUE, **변경 불가**) — FMS/ADMIN 양쪽의 화면 노출 단위. 로그인 응답에서 `MENU_` 접두가 부착돼 권한 식별자가 된다(§6.4)
- `parent_id` (self-ref, 트리), `path`, `label`/`label_en`, `icon`, `sort_order`, `active`, `module_code`(ADMIN/FMS)
- ADMIN 메뉴 예: `ADMIN_CODE`→`ADMIN_CODE_LIST`, `ADMIN_USER`→`ADMIN_USER_LIST` 등
- FMS 메뉴 예: `FMS_HOUSE_BL`→leaf `{SEA_EXP/SEA_IMP/AIR_EXP/AIR_IMP}` (FMS `{모드,방향}` 변형과 1:1)

#### A-09 Menu Policy / A-11 Button Policy
- `(menu_id|button_id, attribute_key, required_value)` 3필드. 한 자원에 여러 정책 행 → **AND 평가**
- 예: 메뉴 `ADMIN_CODE_LIST`에 `(module, ADMIN)`; FMS leaf에 `(module, FMS)` + `(fms_scope, SEA)`

#### A-10 Button
- 식별: `button_code` (UNIQUE) — 로그인 응답에서 `BTN_` 접두 부착(§6.4)
- `menu_id` (FK → A-08, CASCADE), `label`, `action_type`(`CREATE`/`UPDATE`/`DELETE`/`EXPORT`/`CUSTOM`), `sort_order`, `active`
- 인라인 그리드 화면 표준 버튼: Reset / Search / Save (§ `02_SCREENS`)

#### A-12 Code Master / A-13 Code Detail
- Code Master: `master_code`(UNIQUE), `master_name`, `description`, `sort_order`, `active`
- Code Detail: `master_id`(FK → A-12), `code_value`, `code_label`, `sort_order`, `active`, `remark`. `(master_id, code_value)` UNIQUE
- Master : Detail = 1 : N

#### A-14~A-21 특화 코드 (공통 패턴 + 고유 필드)
> 8종 모두 `{식별코드(UNIQUE), 이름, sort_order, active}` 공통. 아래는 고유 필드.

| 엔티티 | 식별코드 | 고유 필드 |
| --- | --- | --- |
| A-14 Carrier | `carrier_code` | `name_en`, `carrier_type`(ENUM), `address`, `edi_code`, soft-delete |
| A-15 Country | `country_code` | `name_en`, `currency_code` |
| A-16 Currency | `currency_code` | `symbol` |
| A-17 Exchange Rate | `(from_currency, to_currency, effective_date)` UNIQUE | `rate`, FK(from/to → A-16) |
| A-18 Freight | `freight_code` | `freight_group`(ENUM) |
| A-19 HS Code | `hs_code` | `description`, `chapter` |
| A-20 Package Unit | `unit_code` | `symbol` |
| A-21 Port | `port_code` | `port_type`(ENUM: SEA/AIR), `country_code`, `iata_code` |

#### A-22 Customer
- 식별: `customer_code` (UNIQUE)
- `customer_type`(ENUM), `name`, `name_en`, `business_no`, `representative`, `phone`, `email`, 주소(local/en), `memo`, `active`, soft-delete
- **FMS 당사자(Party)의 원천**(`docs/fms/01_DOMAIN E-23`). **ADMIN이 관리**(구 MDM 마스터데이터를 ADMIN으로 통합)

#### A-23 Notice
- `title`, `content`(TEXT), `pinned`, `published_at`, `expires_at`, `active`, soft-delete
- 게시 상태는 `published_at`/`expires_at` 기간으로 제어. 대시보드 MJ Notice 위젯의 원천

#### A-24 Refresh Token
- `token_hash`(BCrypt), `user_id`(FK → A-01, CASCADE), `issued_at`, `expires_at`, `revoked_at`(soft revoke)
- 인증 토큰 회전 정책은 `03_ACCESS_CONTROL`

## 3. 엔티티 관계

### 3.1 관계도
```
Admin User (A-01) ─┬─ N:1 ─ Team (A-02)
                   ├─ JSONB ─ attributes(직접 속성)
                   ├─ N:M ─ User–Preset (A-07) ─ N:1 ─ Permission Preset (A-05)
                   └─ 1:N ─ Refresh Token (A-24)

Permission Preset (A-05) ─ N:M ─ Preset–AttributeValue (A-06) ─ N:1 ─ Attribute Value (A-04)

Attribute Definition (A-03) ─ 1:N ─ Attribute Value (A-04)   [attribute_key 기준]

Menu (A-08) ─┬─ self-ref ─ parent_id (트리)
             ├─ 1:N ─ Menu Policy (A-09) ─ (attribute_key, required_value)
             └─ 1:N ─ Button (A-10) ─ 1:N ─ Button Policy (A-11)

Code Master (A-12) ─ 1:N ─ Code Detail (A-13)
Currency (A-16) ─ 1:N ─ Exchange Rate (A-17) [from/to]
```

### 3.2 카디널리티 주석
- **User : Team = N : 1** — 사용자는 0 또는 1개 팀에 소속(`team_id` NULL = 미배정).
- **User : Preset = N : M** (A-07 교차) — 한 사용자가 여러 프리셋 보유. 프리셋의 속성 변경은 보유자 전원에 live 반영.
- **Preset : AttributeValue = N : M** (A-06 교차) — 프리셋은 여러 속성값을 묶음.
- **유효 속성 산식**: `effective = user.attributes ∪ (보유 active preset 각각의 attribute_value)`. attribute_key별 합집합(중복 제거) → 정책 평가 입력 (§4.1).
- **Menu : Policy = 1 : N**, **Button : Policy = 1 : N** — 정책 다중 시 AND. 정책 0개 메뉴는 노출 후처리 규칙 적용(§6.5).
- **Menu : Button = 1 : N** (CASCADE).
- **Code Master : Code Detail = 1 : N**.

## 4. 핵심 업무 프로세스

### 4.1 로그인 → 권한 계산
1. username/password 검증(BCrypt)
2. **유효 속성 계산**: 직접 `attributes` ∪ 보유 active 프리셋의 속성값 (attribute_key별 Union)
3. **접근 가능 메뉴/버튼 평가**: 각 Menu/Button Policy를 순회하며 `유효속성[key] ⊇ {required_value}` 충족 여부 검사(다중 정책 AND)
4. JWT 발급: accessToken에 `authorities`(`MENU_*`/`BTN_*`) + `attributes` 포함, refreshToken 발급·해시 저장
> 상세 흐름·토큰 정책은 `03_ACCESS_CONTROL`.

### 4.2 사용자 생성·권한 배정
- 사용자 생성 시 직접 속성을 부여하거나, 프리셋(A-05)을 배정(A-07)한다. 프리셋 배정이 권장 경로(운영 일관성·live 반영).

### 4.3 공통 코드 관리
- Code Master/Detail 및 특화 코드 8종은 인라인 그리드에서 Reset/Search/Save로 관리(`02_SCREENS`). 식별 코드는 하드 삭제하지 않고 비활성화(§6.6).

## 6. 규칙 / 제약

### 6.1 공통 감사·상태
- 모든 마스터는 `created_at/by`, `updated_at/by` 감사 필드와 `active` 플래그를 가진다.
- 삭제 이력이 필요한 엔티티(User·Carrier·Customer·Notice 등)는 `deleted_at` **소프트 삭제**.

### 6.2 단일 BIGINT PK 원칙
- 신규 마스터는 단일 `BIGINT GENERATED ALWAYS AS IDENTITY` PK를 갖고, 비즈니스 키는 UNIQUE 보조로 둔다. 복합 PK는 회피한다(마이그레이션 V34에서 AttributeValue를 복합 PK→단일 id로 전환한 교훈).

### 6.3 식별 키 불변
- `username`, `team_code`, `attribute_key`, `menu_code`, `button_code`, `master_code`, `code_value`, 프리셋 `code` 등 비즈니스 식별 키는 생성 후 변경 불가(표시 필드만 수정).

### 6.4 권한 식별자 접두(prefix)
- DB에는 `menu_code`/`button_code`를 **접두 없이** 저장한다. 로그인 응답 구성 시 `AuthService`가 `MENU_`/`BTN_`를 자동 부착해 권한 식별자로 만든다. 시드에 접두를 박으면 이중 접두가 되므로 금지.

### 6.5 정책 0개 자원 노출 후처리
- 정책이 0개인 부모 메뉴가 자식 없이 노출되는 over-expose를 막기 위해, 메뉴 트리 구성 시 "자식 0개 root 제거" 후처리를 Service 계층에서 수행한다.

### 6.6 식별 코드 deactivate-only
- 인라인 그리드에서 `menu_code`·`button_code` 등 식별 코드 행은 하드 삭제하지 않고 `active` 토글로만 비활성화한다. 저장 payload는 생성/수정만 다룬다.

### 6.7 프리셋 코드 규약
- `PermissionPreset.code`는 `PRESET_` 접두를 강제(`PermissionPresetCodeValidator`). 예: `PRESET_SEA_MANAGER`.

### 6.8 비밀번호 해시
- `password_hash`는 BCrypt이며 **`$2a$` 접두**를 사용한다(Spring `BCryptPasswordEncoder` 호환). 시드 해시도 동일.

### 6.9 ABAC ↔ FMS 접근 연결
- FMS 화면의 `{모드, 방향}` 접근 제어(`docs/fms/01_DOMAIN §6.7`)는 ADMIN의 `module=FMS` + `fms_scope=SEA/AIR/TRUCK/NON_BL` 속성과 메뉴 정책으로 실현된다. 행(Row) 단위 권한은 도입하지 않는다(FMS 정책 유지).
