# 접근 제어 · 인증 (ADMIN) — 권한 SSOT

> 본 문서는 **권한 모델·인증·사용자 환경설정의 단일 진실 공급원(SSOT)**이다. FMS PRD(`docs/fms/05_NON_FUNCTIONAL §4.2·§11`, `03_TECH_STACK §3.4`, `docs/fms/01_DOMAIN §6.7`)는 본 문서를 참조하며, FMS 고유 정책(`{모드,방향}` 화면 접근·행권한 미도입)만 자기 문서에 유지한다.
> 도메인 엔티티(AttributeDefinition/Value·Menu/Button·Policy·Preset·RefreshToken)는 `01_DOMAIN`을 참조한다(중복 정의 회피).

## 1. 모델 개요 — 왜 ABAC인가
- 레거시는 역할(Role)에 권한을 고정 매핑했다. 포워딩 운영은 `{모듈} × {관리영역 또는 운송라인} × {역할}` 조합이 다양해, 역할 고정 매핑은 역할 폭증을 부른다.
- 본 시스템은 **속성 기반 접근 제어(ABAC)**를 채택한다. 사용자에게 **속성(attribute)** 집합을 부여하고, 자원(메뉴·버튼)에는 **정책(policy)**을 건다. 로그인 시 둘을 대조해 접근을 결정한다.
- 속성 부여는 **권한 프리셋(Permission Preset)**으로 묶어 재사용하며, 프리셋 변경은 보유 사용자 전원에 즉시 반영된다(live binding).

## 2. 속성 체계 (4계층 + 커스텀)

| 계층 | 속성 키 | 타입 | 다중 | 값 | 용도 |
| --- | --- | --- | :---: | --- | --- |
| 역할 | `role` | ENUM | 단일 | `ADMIN` / `MANAGER` / `USER` | 기본 역할 |
| 모듈 | `module` | ENUM | 다중 | `ADMIN` / `FMS` | 접근 가능 상위 모듈 |
| ADMIN 범위 | `admin_scope` | ENUM | 다중 | `CODE` / `USER` / `CUSTOMER` / `CMS_NOTICE` / `ACCESS` | ADMIN 모듈 내부 관리 영역 |
| FMS 범위 | `fms_scope` | ENUM | 다중 | `SEA` / `AIR` / `TRUCK` / `NON_BL` | FMS 모듈 내부 운송 라인 |
| 조직(커스텀) | `department` / `region` / `level` | STRING/NUMBER | 가변 | 자유값 | 조직 정보 |

- 속성 키 메타는 `AttributeDefinition`(A-03), ENUM 허용값은 `AttributeValue`(A-04)가 보관한다.
- 사용자의 직접 속성은 `admin_user.attributes`(JSONB) `Map<String, List<String>>`에 저장된다. 예:
  ```json
  { "role": ["MANAGER"], "module": ["ADMIN", "FMS"], "admin_scope": ["CODE"], "fms_scope": ["SEA"] }
  ```
- **2축 범위 분리**: `module=ADMIN`은 `admin_scope`로, `module=FMS`는 `fms_scope`로 세부 영역을 가른다. 한 사용자가 두 모듈을 동시에 가질 수 있다.

## 3. 정책 (Menu / Button Policy)
- 자원(메뉴·버튼)에 `(자원, attribute_key, required_value)` 정책 행을 건다(A-09 / A-11).
- 한 자원에 정책이 여러 개면 **모두 만족(AND)**해야 접근 가능: `유효속성[key] ⊇ {required_value}` 가 모든 정책 행에 대해 참.
- 예:
  - ADMIN 코드 화면 `ADMIN_CODE_LIST`: `(module, ADMIN)` + `(admin_scope, CODE)`
  - FMS 해상수출 leaf `FMS_HOUSE_BL_SEA_EXP`: `(module, FMS)` + `(fms_scope, SEA)`
- **deny-by-default**: 정책을 만족하지 못하면 노출·실행되지 않는다.
- 정책 0개 부모 메뉴의 over-expose는 Service 후처리("자식 0개 root 제거")로 방지(`01_DOMAIN §6.5`).

## 4. 권한 프리셋 (Live Binding)
- **유효 속성(effective attributes)** = 사용자 직접 속성 ∪ 보유 active 프리셋의 속성값. attribute_key별 합집합(중복 제거).
  - `ComputeEffectiveAttributeValuesService.computeEffectiveAttributes(userId, directAttrs)`가 계산.
- 프리셋(A-05)의 `attributeValueIds` 변경은 그 프리셋을 보유한 모든 사용자에게 **즉시 반영**된다(재설정 불요). 전통적 역할 재할당 대비 운영 비용이 낮다.
- 사용자↔프리셋은 N:M(A-07), 프리셋↔속성값은 N:M(A-06). 프리셋 코드는 `PRESET_*` 강제.

## 5. 인증 (Authentication)

### 5.1 로그인 흐름 (`AuthService.login`)
1. `username`으로 사용자 조회 → 실패 시 자격 증명 오류(사용자 존재 여부 비노출)
2. `BCrypt` 비밀번호 검증(`$2a$` 해시) → active 여부 확인
3. **유효 속성 계산**(§4)
4. **접근 가능 메뉴/버튼 평가**: active 정책 전체를 조회해 `PolicyEvaluator`가 유효 속성과 대조(AND)
5. **authorities 구성**: `ROLE_{role}` + `MENU_{menuCode}` + `BTN_{buttonCode}`
6. **토큰 발급**: accessToken(JWT — subject=username, claims=authorities + attributes), refreshToken(raw 1회 반환, **해시만 저장**, TTL = `refreshTtlDays`)
7. 응답(`LoginResult`): accessToken, refreshToken, 사용자, attributes, **accessibleMenus(`MENU_` 부착)**, **accessibleButtons(`BTN_` 부착)**

### 5.2 토큰 회전·로그아웃·me
- **refresh**: 기존 refresh 해시 조회 → **revoke 후 신규 발급**(회전). 사용자 active 재확인 + 유효 속성·권한 재계산.
- **logout**: refresh 토큰 해시를 revoke.
- **getMe**: 현재 사용자 + 유효 속성 + accessible menu/button 재계산(토큰 미발급).

### 5.3 prefix 규약 (중요)
- DB에는 `menu_code`/`button_code`를 **접두 없이** 저장한다. 응답 구성 시 `AuthService`가 `MENU_`/`BTN_`를 **자동 부착**한다. 시드에 접두를 박으면 이중 접두가 되므로 금지(`01_DOMAIN §6.4`).
- `role` 값은 `ROLE_` 부착으로 Spring Security authority에 매핑.

### 5.4 JWT / 공통 표준
- 발급/검증 키·Claim 구조·Refresh 정책은 FMS·ADMIN 양 스택 공통 표준을 따른다(`docs/fms/03_TECH_STACK §2.2·§3.4`). Access/Refresh TTL 구체값은 설정값으로 운용한다.
- attributes는 JSONB 누적 머지 시 `jsonb || NULL = NULL` 함정을 피해 `COALESCE` 또는 명시적 SET을 사용한다(시드/마이그레이션 규약).

## 6. 모듈 접근 제어 연결

### 6.1 FMS 화면 접근
- FMS의 `{모드, 방향}` 화면 접근(`docs/fms/01_DOMAIN §6.7`)은 `module=FMS` + `fms_scope∈{SEA,AIR,TRUCK,NON_BL}` 속성과 FMS leaf 메뉴 정책으로 실현된다.
- 예: `fms_scope=[SEA]` 사용자는 `FMS_HOUSE_BL_SEA_EXP/IMP` 등 해상 메뉴만 접근, 항공/트럭 메뉴는 deny.
- **행(Row) 단위 권한은 도입하지 않는다**(FMS 정책 유지 — Operator/Team/Sales Man은 실적·필터용이지 권한이 아님).

### 6.2 ADMIN 화면 접근
- ADMIN 화면은 `module=ADMIN` + `admin_scope∈{CODE,USER,CUSTOMER,CMS_NOTICE,ACCESS}` 속성과 ADMIN 메뉴 정책으로 제어된다.
- 예: `admin_scope=[CODE]` 매니저는 코드 마스터 화면군만, ACCESS 관리(메뉴/버튼/프리셋) 화면은 deny.

### 6.3 PMS(실적) 화면 접근
- PMS 실적 화면(`PS-01`)은 `module=PMS` 속성 + PMS leaf 메뉴/버튼 정책으로 제어된다. `module` 카탈로그의 SSOT는 `attribute_value(attribute_key='module')`이므로(별도 `admin.module` 테이블 미사용), 신규 `PMS` 값을 시드로 추가한다.
- 사이드바는 정적 등록(FMS/BMS와 동일): `sidebar.tsx` NAV_MODULE에 PMS 그룹 + `admin-session` firstAccessibleRoute order를 권한 시드와 세트로 추가한다.
- 신규 메뉴/버튼(예: `PMS_PERFORMANCE` 메뉴, 조회/엑셀 등 ActionButton)은 `admin.menu`/`admin.button` + 정책 시드를 동반한다(§7, 02_SCREENS §2.6). DB code는 접두 없이 저장한다(§5.3).
- **행(Row) 단위 권한은 도입하지 않는다** — Operator/Team/Sales Man은 실적 집계·필터용 속성이지 권한이 아니다(FMS 정책 답습, §6.1).

## 7. 프론트엔드 권한 적용
- **라우트 가드(AdminGuard)**: 화면을 `MENU_*` 권한으로 가드. 권한 없으면 진입 차단.
- **버튼 렌더(ActionButton)**: 버튼을 `BTN_*` 권한으로 렌더하며, **권한 없으면 렌더하지 않는다(return null)**. 신규 ActionButton 도입 시 `admin.button` + 정책 시드 동반 필수(`02_SCREENS §2.6`).
- **로그인 시 캐시 초기화**: 로그인 직후 React Query 캐시를 `clear()`하여 이전 사용자의 사이드바·권한 캐시 잔존을 방지한다.
- **메서드 레벨 보안**: 백엔드는 `@EnableMethodSecurity`로 메서드 권한 매핑을 활성화한다. 멀티모듈이므로 모듈(FMS/ADMIN)별로 활성화 여부를 별도 점검한다.

## 8. 사용자 환경설정
> FMS PRD `05_NON_FUNCTIONAL §11`이 "CMS 도입 이후 서버 프로필"로 예고한 부분을 ADMIN(=CMS 실체)이 담당하므로, 본 절을 SSOT로 둔다. FMS NFR §11은 본 절을 참조한다.

### 8.1 설정 항목
| 항목 | enum | 기본값 |
| --- | --- | --- |
| Theme | `light` / `dark` | `light` |
| Tone | `cool` / `neutral` / `warm` | `cool` |
| Density | `compact` / `cozy` / `comfortable` | `cozy` |
| FontSize | 11~16 px (1px 단위) | `13` |
| Required style | `bar` / `dot` / `fill` / `subtle` | `bar` |

### 8.2 저장 방식 — 2단계
- **MVP(서버 미도입 전)**: 브라우저 `localStorage` 단일 JSON(`fms_user_preferences`, `_schemaVersion` 포함). 디바이스별 독립.
- **ADMIN(서버) 도입 후**: `admin_user`의 `preferences`를 **JSONB 단일 컬럼**으로 운영(항목 증가에 DDL 불요). 첫 로그인 시 localStorage→서버 1회 마이그레이션 후 localStorage 키 제거.

### 8.3 FOUC 방지
- MVP: `<head>` 동기 `<script>`로 localStorage 읽어 `<html data-*>` + `:root` CSS 변수 주입(본문 렌더 전 적용).
- 서버 도입 후: 로그인 시 preferences를 쿠키 미러링 → Next.js 서버 컴포넌트가 SSR 응답 `<html>`에 주입.

## 9. 인증·권한 감사
- 감사 대상 이벤트(로그인·로그아웃·권한 변경·실패한 인증 시도)는 `docs/fms/05_NON_FUNCTIONAL §5.1` 감사 정책을 공유한다.
- 시드 운영 계정(module·scope별 매니저)으로 권한 분기를 검증한다(예: code_manager=`admin_scope:CODE`, sea_manager=`fms_scope:SEA`).
