# 정산 도메인 (BMS)

> **상태: ✅ 구현·검증 완료** — 로드맵 단계 0~E(운임 입력 · 서류 발행 · 조회 리스트 · 그룹화 · 세금계산서/전표 발급) + 발급취소/재발급까지 구현·검증 완료(~2026-06-06).
>
> 본 문서는 **BMS 모듈의 도메인 언어와 엔티티**를 정의한다. BMS는 FMS(포워딩 업무)에서 발생하는 **운임을 매출/매입으로 정산하고 금융 서류·전표를 발행**하는 영역을 담당한다.
> 프로젝트 공통 사항(목표·범위·기술·비기능)은 `docs/fms/00_OVERVIEW`·`03_TECH_STACK`·`docs/fms/05_NON_FUNCTIONAL`을 공유한다. **권한·인증의 상세 메커니즘과 사용자 환경설정은 `docs/admin/03_ACCESS_CONTROL`을 SSOT로 참조**하며, 본 문서는 BMS 도메인 모델(엔티티)만 정의한다.
> 본문은 업무 용어(유비쿼터스 언어) 위주로 기술하고, **실제 컬럼명·ENUM 값·참조 키는 §7(컬럼 매핑)에 모은다.** SQL 타입은 DDL(`schema/`·Flyway)에 둔다.

## 0. 위치와 경계

- **BMS 모듈 = 운임·정산(Billing/Settlement).** `00_OVERVIEW §4.1`이 후순위(TBD)로 미뤄둔 운임 정산·금융 서류·전표 영역을 본 모듈로 구현한다.
- **운임 입력 진입점 = FMS B/L Entry의 Freight 탭.** 운임 입력 UI는 FMS B/L Entry 화면(House: S-02, Master: S-04)의 **Freight 탭**에서 제공하며, **이 탭의 저장은 BMS 소유 테이블(운임 헤더·운임 라인)에 기록**된다(쓰기 소유권 BMS). 별도 BMS 운임 입력 화면은 두지 않는다 — BMS 운영 화면은 **금융 서류 조회(Invoice·Payment·D/C Note 리스트)** 중심이다(`02_SCREENS`).
- **멀티모듈 구성**: BMS 백엔드는 `back-end/java-spring-bms` (FMS는 `java-spring`, ADMIN은 `java-spring-admin`). 세 모듈은 **같은 PostgreSQL 인스턴스**를 쓰되 BMS가 `bms` 스키마를 소유한다 (`03_TECH_STACK §5` 스키마 정책).
- **모듈 간 의존 방향**:
  - BMS → **ADMIN** 마스터를 **cross-schema 읽기**로만 참조한다(거래처·운임 항목·통화·환율·운송사·팀·사용자). 마스터의 쓰기·소유권은 ADMIN에만 있다.
  - BMS → **FMS** 원장을 **cross-schema 읽기**로 참조한다. 운임 헤더는 House B/L(E-08) 또는 Master B/L(E-01)에 **다형 참조(B/L 종류 + B/L 식별자)**로 귀속되며, Per 기준 수량 산정 시 B/L 화물 수치(중량·부피 등)를 읽는다. 정산·운임·금융 서류 데이터의 쓰기·소유는 BMS에 있다.
- **1차 범위 (본 프로젝트)**: 운임 정산(매출/매입) → 금융 서류(**Invoice · Payment · D/C Note**) → **전표(Slip) 발행**.
  - 전표(Slip) 발행은 **운영 화면을 제공하지 않고 테스트 성격의 API 호출**로 수행한다 (`02_SCREENS §1`).
- **외부 회계 연동 없음**: 외부 회계 시스템과의 연동은 두지 않는다. **회계 처리 여부는 전표 발행 API 경로에서 BMS가 판단**한다 *(상세 — 후속 라운드에서 확정)*.
- **Freight = B/L(HBL/MBL) 귀속 단일 원칙 인수**: 운임은 B/L(HBL 또는 MBL) 레벨에 귀속된다. Container 레벨 운임 귀속은 없다 (`docs/fms/01_DOMAIN §6.11`, BMS 모델링의 기본 전제. FMS·TECH_STACK의 "HBL 레벨" 문구는 본 라운드 보정 대상).

## 2. 핵심 엔티티

### 2.1 엔티티 목록
> BMS 엔티티는 `B-` 접두로 식별한다. 각 엔티티는 단일 대리키 PK를 갖고 비즈니스 키는 UNIQUE 보조로 둔다(복합 PK 회피 원칙, §6.2). 속성의 실제 컬럼명은 §7.

| # | 엔티티 | 설명 |
| --- | --- | --- |
| B-01 | 운임 헤더 (Freight Header) | B/L 1:1. 다형 B/L 참조(종류 + 식별자) · 당사자 3종(실화주·선사·정산처) · 환율 기준(통화·일자) |
| B-02 | 운임 라인 (Freight Line) | 정산 라인. 매출/매입 구분 · 운임 항목/수량/단가/Per · 통화·환율·금액·세액·USD 환산 · 세금계산서·전표·실적 · 금융 서류 연결 |
| B-03 | 금융 서류 (Financial Document) | Invoice / Payment / D/C Note. 종류 판별자 단일 테이블 · 합계 스냅샷 · 상태(라인 파생) · 실적/팀/담당자 |

### 2.2 엔티티별 핵심 속성
> 업무 의미 중심. 컬럼명·허용값은 §7. 모든 엔티티는 공통 감사 속성·활성 상태를 가진다(§6.1).

#### B-01 운임 헤더 (Freight Header)
- **B/L 1:1** — 한 B/L(House 또는 Master)당 하나. 동일 B/L에 중복 생성 불가.
- **B/L 귀속** — B/L 종류(House/Master)와 B/L 식별자로 FMS 원장(House B/L·Master B/L)을 가리킨다. 다형 참조이며 스키마 간 하드 FK는 두지 않는다(§6.4).
- **당사자**(Freight 탭 Rate Header 3종) — 실화주 · 선사 · 정산처(각각 ADMIN 거래처·운송사 마스터 참조).
- **환율 기준** — 환율 조회 기준 통화 · 기준 일자. 운임 라인 환율 자동 산정의 입력값(§6.5).
- 헤더는 FMS B/L Entry의 Freight 탭 저장 시 생성/갱신된다(별도 입력 화면 없음). B/L 단위 공유 속성(3당사자·환율 기준)을 라인에서 분리하는 용도.

#### B-02 운임 라인 (Freight Line)
- **소속** — 운임 헤더에 종속(헤더당 N행). B/L 귀속은 헤더 경유(§3.2).
- **구분** — 매출/매입(FE 입력부에서 두 영역으로 분리 표시하되 저장은 단일 엔티티) · 금융 서류 종류(연결될 서류 종류, 연결 시 서류 종류와 일치 §6.12. **고객 종류 × 매출/매입으로 라인 저장 시 자동 산정** §6.16).
- **운임 산정** — 운임 항목(ADMIN 운임 코드) · 단위 수량 · 단위 가격 · 산정 기준(Per). 수량은 Per 기준으로 산정(§6.6).
- **통화·금액** — 거래 통화 · 환율(스냅샷) · 정산 통화 금액 · 로컬(KRW) 금액 · 정산/로컬 세액 · USD 환산 환율·금액. 환율은 헤더 기준으로 자동 조회·고정(§6.5).
- **당사자** — 청구 고객(ADMIN 거래처). 서류에 묶일 때 서류 고객과 동일해야 함(§6.14). 헤더의 실화주/정산처(B/L 당사자 맥락)와는 별개 관심사.
- **세무·전표·실적** — 세금 유형(영세/면세/과세) · 세금계산서 번호·발급일 · 전표 번호·발급일 · 실적 인정 일자. 세금계산서·전표 번호는 **한 번호가 여러 라인을 묶을 수 있다**(라인별 고유 아님). 실적 일자는 서류에 묶이면 서류 값으로 갱신(§6.15).
- **금융 서류 연결** — 소속 금융 서류 참조(N:1, §3.2).

#### B-03 금융 서류 (Financial Document)
- **식별** — 금융 서류 번호(고유). 채번 규칙 §6.11.
- **종류** — Invoice / Payment / Debit / Credit(단일 테이블 + 종류 판별자). 조회 화면 3종은 종류 필터(Debit+Credit → D/C Note 리스트).
- **발급일**.
- **상태** — 5단계(§6.12). 세금/전표 단계는 구성 라인에서 파생.
- **그룹** — 동일 타입 서류를 묶는 그룹 번호(별도 엔티티 없음).
- **고객** — 서류 고객(ADMIN 거래처). 묶이는 라인은 모두 동일 고객(§6.14).
- **합계(스냅샷, §6.13)** — 정산 통화 합계 · 로컬 합계 · 정산/로컬 합계 세액 · USD 환산 합계.
- **실적 인정 일자** — 서류 기준. 라인이 묶일 때 새로 부여되며 묶인 라인에 전파(§6.15).
- **실적 귀속** — 입력 팀(ADMIN 팀) · 실제 입력 담당자(ADMIN 사용자, 대무 케이스 위해 팀과 별도).
- BMS 조회 화면 3종(Invoice·Payment·D/C Note 리스트)의 원천.

## 3. 엔티티 관계

### 3.1 관계도
```
[FMS]  House B/L (E-08) ┐
       Master B/L (E-01)┘─ 다형 참조(종류+식별자, 1:1) ─┐
                                                       │
운임 헤더 (B-01) ─┬─ 1:N ─ 운임 라인 (B-02) ─ N:1 ─ 금융 서류 (B-03)
                  ├─ N:1 ─ [ADMIN] 거래처 (A-22)   [실화주 / 정산처]
                  ├─ N:1 ─ [ADMIN] 운송사 (A-14)   [선사]
                  └─ N:1 ─ [ADMIN] 통화 (A-16)     [환율 기준 통화]

운임 라인 (B-02) ─┬─ N:1 ─ [ADMIN] 운임 항목 (A-18) [운임 코드]
                  ├─ N:1 ─ [ADMIN] 통화 (A-16)      [거래 통화]
                  ├─ N:1 ─ [ADMIN] 거래처 (A-22)    [청구 고객]
                  └─ 파생 ─ [ADMIN] 환율 (A-17)     [환율 자동 조회·스냅샷]

금융 서류 (B-03) ─┬─ N:1 ─ [ADMIN] 거래처 (A-22)    [서류 고객]
                  ├─ N:1 ─ [ADMIN] 팀 (A-02)        [입력 팀]
                  └─ N:1 ─ [ADMIN] 사용자 (A-01)    [입력 담당자]

※ B/L : 운임 라인 = 1 : N (운임 헤더 1:1 경유)
```

### 3.2 카디널리티 주석
- **B/L : 운임 라인 = 1 : N** — 한 B/L(HBL 또는 MBL)은 다수의 매출·매입 운임 라인을 가진다. 본 관계는 **운임 헤더(B/L 1:1)를 경유**해 성립한다(아래 두 항).
- **B/L : 운임 헤더 = 1 : 1** — HBL 또는 MBL 1건당 헤더 1건. B/L(종류+식별자) 단위 유일. MBL 운임 헤더와 그 하위 HBL 운임 헤더는 **각자 독립**(B/L 계층은 FMS 소유, 정산 레벨에서 부모-자식 연결 없음).
- **운임 헤더 : 운임 라인 = 1 : N** — 매출/매입은 라인의 구분값으로 나뉜다(별도 테이블 아님). 헤더 삭제 시 라인 동반 삭제.
- **운임 라인 : 금융 서류 = N : 1** — 여러 운임 라인이 하나의 금융 서류로 묶인다. 한 라인은 한 서류에만 귀속(N:M 아님). 묶이는 라인은 모두 동일 고객(§6.14).
- **금융 서류 : ADMIN = N : 1** — 서류 고객·입력 팀·입력 담당자는 cross-schema **읽기**.
- **금융 서류 그룹** — 그룹 번호는 동일 타입 서류를 묶는 키 필드일 뿐 서류 간 FK 관계는 없다(서류는 상호 독립).
- **Container 레벨 운임 없음** — Freight = B/L 귀속 단일 원칙(§6.8).

## 4. 핵심 업무 프로세스
> 고급 흐름. 전표 상세는 후속 라운드.

1. **운임 입력 (FMS Freight 탭)**: FMS B/L Entry(S-02/S-04)의 Freight 탭에서 운임 헤더와 운임 라인을 입력 → BMS 테이블에 기록. 라인은 FE에서 매출/매입 영역으로 분리 표시.
2. **금액 산정**:
   - Per 기준으로 단위 수량 산정(예: Gross W/T → 총중량 Kg, §6.6).
   - 정산 통화 금액 = 단위 수량 × 단위 가격.
   - 헤더 환율 기준(통화·일자)으로 ADMIN 환율을 조회해 로컬·USD 환산(§6.5).
   - 세금 유형에 따라 세액 산정(과세 기본 10%, §6.7).
3. **금융 서류 발행**:
   - 동일 고객 운임 라인을 묶어 금융 서류를 발행·연결, 서류 번호 채번(§6.11).
   - 서류에 **실적 일자 부여 → 묶인 라인의 실적 일자 갱신**(§6.15).
   - 합계는 구성 라인 합산으로 **스냅샷**(§6.13).
   - 동일 타입 서류는 그룹 번호로 묶음.
   - 상태는 생성→그룹 진행, 라인 세금계산서/전표 발생에 따라 세금/전표 단계 파생(§6.12).
4. **전표 발행**: 테스트 성격 API 호출로 전표 발행 → 라인 전표 번호·발급일 기록(서류 상태 전표 단계 파생), 회계 처리 여부 판단. *(상세 TBD.)*

## 6. 규칙 / 제약
> ADMIN·FMS 공통 규약을 답습한다.

### 6.1 공통 감사·상태
- 모든 엔티티는 공통 감사 속성(생성/수정 시각·주체)과 활성 상태를 가진다. 삭제 이력이 필요한 엔티티는 소프트 삭제한다.

### 6.2 단일 대리키 PK 원칙
- 각 엔티티는 단일 대리키(BIGINT IDENTITY) PK를 갖고, 비즈니스 키는 UNIQUE 보조로 둔다. 복합 PK는 회피한다 (`docs/admin/01_DOMAIN §6.2`).

### 6.3 식별 키 불변
- 비즈니스 식별 키(금융 서류 번호·전표 번호·세금계산서 번호 등)는 생성 후 변경 불가(표시 필드만 수정).

### 6.4 마스터 소유권 / B/L 다형 참조
- BMS는 ADMIN 마스터를 **읽기 전용**으로 참조한다. 정산·운임·금융 서류 데이터의 쓰기·소유는 BMS에만 있다.
- 운임 헤더의 B/L 귀속은 **다형 참조**(B/L 종류 + 식별자)이며, 스키마 간 **하드 FK를 두지 않고**(모듈 결합 회피) 앱 계층에서 존재 검증한다.

### 6.5 환율 스냅샷·자동 산정
- 운임 라인 환율은 헤더의 (기준 통화·기준 일자)와 라인 거래 통화로 **ADMIN 환율 마스터에서 조회**해 채운다. 조회 결과는 라인에 **스냅샷**으로 고정한다(이후 환율 변동에 영향받지 않음). 로컬·USD 금액은 각 환율을 적용해 산정한다.

### 6.6 Per 기준 수량 산정
- 산정 기준(Per)에 따라 단위 수량을 결정한다. 예: Per=Gross W/T → 단위 수량을 B/L 총중량(Kg) 값으로 산정. 산정에 필요한 B/L 화물 수치는 FMS 원장에서 읽는다.

### 6.7 세액 산정
- 세금 유형별로 세액을 산정한다: 영세·면세는 0, 과세는 기본 10%. 정산 통화 세액·로컬 세액을 각각 보관한다.

### 6.8 Freight = B/L(HBL/MBL) 귀속 단일 원칙
- 운임은 B/L(HBL 또는 MBL) 레벨에 귀속된다. Container 레벨 운임 귀속은 도입하지 않는다 (`docs/fms/01_DOMAIN §6.11`).

### 6.9 조회 경로 — OLTP 전용 (분석은 실적 모듈)
- BMS 운영 조회(금융 서류 리스트 등)는 OLTP(PostgreSQL)에서 처리한다. **실적·분석·집계 조회는 BMS 범위가 아니며 실적(PMS) 모듈 소관이다.** PMS는 Phase 1에서 OLTP 직접 집계로 시작하고 성능 한계 시 Phase 2에서 MongoDB(Analytics Mart)로 전환한다(상세 `docs/pms/01_PMS_DOMAIN`).

### 6.10 세금계산서·전표 발행 = API 전용
- 세금계산서 발급과 전표(Slip) 발행은 운영 화면을 제공하지 않고 **API로만** 수행한다. 전표 발행은 테스트 성격이며, 회계 처리 여부 판단도 이 경로에서 이뤄진다 *(상세 TBD)*.

### 6.11 금융 서류 채번
- 금융 서류 번호 = **종류 이니셜 1자(I=Invoice / P=Payment / D=Debit / C=Credit) + YYMM(4) + 시퀀스(5)**. 예: `I260600001`. 시퀀스는 종류·월(YYMM) 단위 5자리(월·타입별 리셋)로 자동 발번한다.

### 6.12 금융 서류 상태 파생
- 상태 우선순위: 생성 < 그룹 < 세금 < 전표 < 정산완료 (`CREATED < GROUPED < TAX < SLIP < CLEAR`). 전이는 **비순차**(단계 건너뛰기 허용)이며 서류 상태는 **도달한 것 중 최고 단계**로 유지한다.
- **세금/전표 단계는 구성 라인에서 파생**한다: 묶인 라인 중 전표 번호 보유 라인이 하나라도 있으면 서류는 전표(SLIP), 없고 세금계산서 번호 보유 라인이 있으면 세금(TAX). 세금·전표 혼재 시 더 높은 전표(SLIP).
- 생성·그룹·정산완료는 각 이벤트(서류 생성 / 그룹 번호 부여 / 정산 완료) 시 반영. (저장 컬럼 vs 실시간 파생은 조회 리스트 필터 고려해 보관 권장.)

### 6.13 금융 서류 합계 스냅샷
- 서류 합계는 구성 라인 합산을 **스냅샷**으로 보관한다. 정산 통화 합계·합계 세액은 **통화 혼재 시에도 환산 없이 단순 합산**한다(정산 통화 합계는 정보성). 로컬 합계·합계 세액·USD 환산 합계는 각 라인의 로컬/USD 환산액 합산이다.

### 6.14 서류 단일 고객 제약
- 한 금융 서류에 묶이는 운임 라인은 **모두 동일 고객**이어야 한다. 서류 고객 = 묶인 라인들의 고객.

### 6.15 실적 일자 전파
- 운임 라인이 금융 서류에 묶이면 서류에 **서류 기준 실적 일자가 새로 부여**되고, 묶인 라인의 실적 일자가 서류 값으로 **갱신**된다(서류 기준으로 일원화).

### 6.16 금융 서류 종류 자동 산정
- 운임 라인의 금융 서류 종류는 **고객 종류(ADMIN 거래처 `customer_type` — 일반 고객 / 파트너)와 매출/매입의 조합으로 라인 저장 시 자동 산정·저장**된다. 사용자가 직접 선택하지 않는다.

  | 고객 종류 | 매출(Selling) | 매입(Buying) |
  | --- | --- | --- |
  | 일반 고객 | Invoice | Payment |
  | 파트너 | Debit | Credit |

### 6.17 발행 게이팅·발행 후 불변·행 삭제 보호
- **발행 게이팅**: 운임 라인은 **B/L 저장 후에만** 금융 서류로 묶을 수 있다(미저장 라인은 발행 불가).
- **발행 후 불변**: 운임 라인이 서류로 묶여 합계가 스냅샷되면 해당 라인 값은 변경할 수 없다(§6.13 스냅샷 고수). 수정이 필요하면 **서류 삭제 후 재발행**한다(in-place 수정 없음).
- **행 삭제 보호**: 서류에 묶인 운임 라인은 행 삭제할 수 없다. 삭제하려면 먼저 **묶인 서류에서 풀어낸 뒤**(서류 삭제) 행을 삭제한다.
