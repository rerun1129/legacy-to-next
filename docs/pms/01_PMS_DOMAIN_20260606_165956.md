# 실적 도메인 (PMS)

> **상태: 📝 기획 완료 · 구현 예정** — 단일 화면 실적 조회 모듈. 본 라운드에서 도메인·화면·집계 아키텍처를 확정한다.
>
> 본 문서는 **PMS(Performance Management) 모듈의 집계 도메인**을 정의한다. PMS는 **신규 원장 엔티티를 만들지 않고**, FMS(B/L 원장)와 BMS(운임·금융 서류)의 기존 데이터를 **읽기 전용으로 집계**해 실적을 조회한다.
> 프로젝트 공통 사항(목표·범위·기술·비기능)은 `docs/fms/00_OVERVIEW`·`03_TECH_STACK`·`05_NON_FUNCTIONAL`을 공유한다. **권한·인증은 `docs/admin/03_ACCESS_CONTROL §6.3`을 SSOT로 참조**한다. 화면 명세는 `02_PMS_SCREENS`.
> 본문은 업무 용어 위주로 기술하고, 실제 컬럼명은 BMS `docs/bms/01_BMS_DOMAIN §7`(운임·서류)과 FMS `docs/fms/01_DOMAIN`(B/L)을 출처로 인용한다.

## 0. 위치와 경계

- **PMS 모듈 = 실적 집계·조회(Performance).** `docs/fms/00_OVERVIEW §4.1`이 후순위로 미뤄둔 실적 영역을 본 모듈로 구현한다. 단일 화면(`PS-01`)에서 운임·서류·세금계산서·전표 실적을 조회한다.
- **읽기 전용·무소유(無所有).** PMS는 자체 테이블을 두지 않는다. 모든 집계는 BMS(`bms.freight_line`·`bms.financial_document`·`bms.freight_header`)와 FMS(`fms.house_bl`·`fms.master_bl` 및 모드별 하위 테이블)의 **cross-schema 읽기**로 수행한다. 데이터 쓰기·소유권은 각 모듈(BMS/FMS/ADMIN)에 있다.
- **백엔드 호스팅 = `back-end/java-spring-pms` (독립 마이크로서비스).** PMS는 FMS·ADMIN·BMS와 동일하게 **독립 Spring Boot 모듈**(포트 8083)로 분리한다. 자체 스키마를 소유하지 않는 **읽기 전용** 서비스이며(Flyway 미사용), `bms`·`fms`·`admin` 스키마를 **cross-schema 읽기**(`@Immutable` ref 엔티티)로 집계한다. 집계 쿼리를 위한 인덱스 보강은 소스 스키마 소유 모듈(예: `bms` 인덱스는 `java-spring-bms` Flyway)에서 관리한다. 프론트엔드도 PMS를 **사이드바 독립 모듈**로 등록하고 전용 base URL(`PMS_API_URL`, 8083)로 호출한다.
- **집계 단위 = House B/L.** 실적 그리드의 한 행 = 하나의 B/L(원칙적으로 House B/L). 운임 라인·금융 서류를 B/L 레벨로 롤업한다(거래처·팀 등 상위 차원 집계는 본 라운드 범위 아님 — 필터로 좁혀 조회).
- **4개 집계 기준 = 집계 대상 필터.** 운임 입력 건 / 서류 생성 / 세금계산서 발행 / 전표 발행은 **무엇을 집계 대상으로 삼을지 고르는 필터**(단일 선택)이며, 선택에 따라 집계 소스 행(스코프)이 바뀐다(§2).
- **모듈 간 의존**: PMS → BMS·FMS·ADMIN을 **cross-schema 읽기**로만 참조. 하드 FK를 두지 않고 앱 계층에서 조인한다(BMS `§6.4` 답습).

## 2. 집계 기준 (집계 대상 필터)

화면 상단에서 아래 4개 중 **하나**를 선택한다. 선택값이 집계의 **모수(FROM·WHERE)**를 결정하며, 선택된 행 집합을 **House B/L별로 합산**한다.

| 기준 | 집계 모수(대상 행) | 소스 | 발급 판정 / 비고 |
| --- | --- | --- | --- |
| **운임 입력 건** | 운임 라인 **전체** | `bms.freight_line` | 조건 없음(B/L에 입력된 모든 운임 라인) |
| **서류 생성** | 금융 서류 행 **전체** | `bms.financial_document` (+ 묶인 `freight_line`) | 서류 단위 집계. 금액은 서류 합계 스냅샷 |
| **세금계산서 발행** | 세금계산서 발행된 운임 라인만 | `bms.freight_line` | `tax_no IS NOT NULL` |
| **전표 발행** | 전표 발행된 운임 라인만 | `bms.freight_line` | `slip_no IS NOT NULL` |

- **운임/세금계산서/전표 기준**은 모두 `freight_line`을 모수로 하며 WHERE 조건만 다르다. 금액 컬럼은 라인의 `local_amount`/`usd_amount`를 `financial_doc_type`(INVOICE/PAYMENT/DEBIT/CREDIT)별로 합산한다.
- **서류 생성 기준**은 `financial_document`를 모수로 하며 금액은 서류 합계 스냅샷(`local_total_amount`/`usd_total_amount`)을 `document_type`별로 합산한다. 한 서류는 1 B/L에 종속되지 않을 수 있으므로(여러 운임 라인이 여러 B/L에서 묶일 수 있음) **묶인 운임 라인의 B/L 기준으로 귀속**한다(서류↔B/L은 운임 라인 경유).
- 4개 기준 공통으로, 모수가 없는 B/L은 결과에서 제외된다(예: "세금계산서 발행" 선택 시 세금계산서 라인이 하나도 없는 B/L은 미표시).

## 3. 데이터 소스 & 조인

### 3.1 소스 테이블
| 역할 | 테이블 | 소유 | 비고 |
| --- | --- | --- | --- |
| 운임 라인(모수) | `bms.freight_line` (B-02) | BMS | 금액·세액·발급(tax/slip)·실적일자·서류연결 |
| 금융 서류(모수) | `bms.financial_document` (B-03) | BMS | 합계 스냅샷·상태·팀·담당자 |
| 운임 헤더 | `bms.freight_header` (B-01) | BMS | B/L 다형 참조(`bl_type`,`bl_id`)·실화주·선사·정산처 |
| House B/L | `fms.house_bl` (E-08) | FMS | 그룹 키. 식별·경로·일정·영업·화물 수치 |
| Master B/L | `fms.master_bl` (E-01) | FMS | `bl_type=MASTER` 또는 House의 부모 |
| 모드별 하위 | `house_bl_air(_charge)`·`house_bl_truck`·`house_bl_sea_container`·`house_bl_non_bl` | FMS | Charge W/T·Load Type·R/Ton 등(§3.3) |
| 이름 해석 | ADMIN `customer`·`carrier`·`admin_user`·`team`·`port` | ADMIN | 코드→이름(CodeNameResolver) |

### 3.2 조인 경로
```
freight_line ─ N:1 ─ freight_header(bl_type, bl_id)
                        ├─ bl_type=HOUSE → fms.house_bl (hbl_no, mbl_no, job_div, bound, etd, eta, pol/pod, sales_man, incoterms, 화물수치)
                        └─ bl_type=MASTER → fms.master_bl (mbl_no, job_div, bound, etd, eta, pol/pod)
freight_line ─ N:1 ─ financial_document (서류 기준일 때 모수)
house_bl ─ 모드별 1:N/1:1 ─ house_bl_air/truck/sea_container/non_bl (Charge W/T·Load Type·R/Ton)
코드 컬럼 ─ ADMIN 마스터 조인 ─ 이름 표시
```
- **하드 FK 없음**: `freight_header.(bl_type,bl_id)` → FMS B/L은 soft 참조(BMS `§6.4`). PMS 조회는 앱 계층/뷰에서 조인한다.
- **현 BMS B/L ref 한계**: 기존 BMS 조회용 `house_bl_ref`/`master_bl_ref`는 `job_div`/`bound`/BL번호/`etd`/`eta`만 보유한다. PMS는 **`fms.house_bl`/`master_bl` 본체(+모드별 하위)까지 조인하는 전용 read model**을 신설한다.

### 3.3 화물 수치 컬럼의 모드 분산 (구현 시 확정)
화물 수치는 모드별 하위 테이블에 분산된다. 구현 시 B/L `job_div`에 따라 소스를 선택한다.
| 컬럼 | 소스 후보 | 비고 |
| --- | --- | --- |
| Package Qty | `house_bl.pkg_qty`(헤더 요약) | 컨테이너 단위는 `house_bl_sea_container.pkg_qty` |
| CBM | `house_bl.cbm` | |
| Gross W/T | `house_bl.gross_weight_kg` | |
| Charge W/T | `house_bl_air.charge_weight_kg` / `house_bl_truck.charge_weight_kg` | AIR·TRUCK |
| Load Type | `house_bl_truck.load_type`(ENUM `LoadType`) / SEA FCL·LCL | 모드별 |
| R/Ton | SEA = 파생 `max(Gross W/T[MT], CBM)` · NON_BL = `house_bl_non_bl.rton`(저장) · **AIR·TRUCK = 공란** | 모드별 분기(확정) |

## 4. 집계 아키텍처 결정 (Phase 1 OLTP / Phase 2 Mart)

> 기존 PRD(`docs/fms/01_DOMAIN §6.11`, `03_TECH_STACK §5.1`)는 실적·분석을 MongoDB Analytics Mart 전용으로 규정했다. PMS는 아래 **단계적 예외**를 따른다(해당 절도 본 라운드에서 정합화됨).

- **Phase 1 — OLTP 직접 집계.** `bms.freight_line`/`financial_document`를 `fms.house_bl`/`master_bl`과 조인한 **평면 집계**(House B/L 단위 GROUP BY + 조건부 SUM/COUNT)로 구현한다. 레거시 실적 병목의 원인이던 `HBL × Container × Freight` 3자 카티시안 경로가 아니므로 OLTP에서 처리 가능하다. 인덱스 보강(`bms` 쿼리 인덱스)을 전제로 한다.
- **Phase 2 — Mart 전환.** 데이터량·동시성 증가로 응답·부하 한계가 확인되면 MongoDB Fact Mart로 전환한다. Fact 다큐먼트 스키마·동기화 패턴(Outbox/CDC/Batch)·파티셔닝은 **전환 라운드에서 확정**한다(`03_TECH_STACK §5.1`).
- **전환 트리거(가이드)**: 실적 조회 P95 응답이 목표(예: 수 초)를 지속 초과하거나, 집계 쿼리가 운영 OLTP 부하를 유의하게 잠식할 때. 구체 수치는 운영 데이터로 확정.

## 5. 실적 귀속 · 금액 차원 · Profit

### 5.1 실적 귀속
- **실적 일자**: `freight_line.performance_dt`(서류에 묶이면 `financial_document.performance_dt` 값으로 갱신, BMS `§6.15`). 기간 필터·표시의 기준 일자.
- **팀/담당자**: `financial_document.team_code`(입력 팀) · `financial_document.operator`(입력 담당자). **운임만 있고 서류 미생성**인 라인(운임 입력 건 기준)은 팀/담당자가 비어 있을 수 있다.
- **행(Row) 권한 없음**: Operator/Team/Sales Man은 집계·필터 속성일 뿐 권한이 아니다(`docs/admin/03_ACCESS_CONTROL §6.1·§6.3`).

### 5.2 금액 차원
- 통화 차원: **Local(KRW)** `local_amount`/`local_total_amount`, **USD** `usd_amount`/`usd_total_amount`. (정산 통화 `settle_amount`는 통화 혼재로 단순 합산이 정보성 — 본 화면 컬럼은 Local·USD 중심.)
- 서류 종류 차원: `financial_doc_type`/`document_type` = INVOICE / PAYMENT / DEBIT / CREDIT 별 합산.

### 5.3 Profit 정의 *(확정)*
- **Local Profit = (Invoice + Debit) − (Payment + Credit)** (Local 기준).
- **USD Profit = (Invoice + Debit) − (Payment + Credit)** (USD 기준).
- 근거: Invoice·Debit = 매출성(받을 금액), Payment·Credit = 매입성(줄 금액). BMS 정산 규칙(`01_BMS_DOMAIN §6.16` 고객종류×매출/매입)과 정합한다.

## 6. 규칙 / 제약

### 6.1 HOUSE vs MASTER 직속 운임 *(확정)*
- 운임 헤더는 `bl_type`이 HOUSE 또는 MASTER다. 집계 단위는 House B/L이 원칙이며, **House 없이 Master에 직속된 운임**(`bl_type=MASTER`)은 `freight_header`의 B/L로 행을 만들되 **House B/L No는 빈칸, Master B/L No만 채운다**.

### 6.2 미발급/미생성 행 처리
- 기준별 모수에 따라 결측이 발생할 수 있다(예: 운임 입력 건 기준에서 서류 미생성 → Team/서류번호·금액 일부 컬럼 공란). 결측은 공란으로 표시하고 합계에서 0으로 처리한다.

### 6.3 마감 컬럼(추후 기능)
- `B/L Closed` · `Freight Closed`는 **추후 도입 예정 기능**의 자리표시 컬럼이다. 본 라운드에서는 **컬럼 배치만** 하고 데이터 바인딩·소스는 두지 않는다(공란).

### 6.4 코드→이름 표시
- 거래처(실화주·정산처)·운송사·담당자·팀·항만 등 코드 컬럼은 코드 저장 + ADMIN 마스터 조인으로 이름을 함께 표시한다(FMS `CodeNameResolver` 패턴 답습).

### 6.5 조회 전용
- PMS 화면은 조회 전용이다. 인라인 편집·저장 액션을 두지 않는다(엑셀 내보내기 등 보조 액션은 화면 명세 `02_PMS_SCREENS` 참조).

## 7. 외부 주체 및 소비 모듈
| 구분 | 주체/시스템 | 연계 성격 |
| --- | --- | --- |

| 인증 클라이언트 | 프론트엔드(Next.js) | `module=PMS` 권한으로 라우트 가드·버튼 렌더(`docs/admin/03_ACCESS_CONTROL §6.3`) |
